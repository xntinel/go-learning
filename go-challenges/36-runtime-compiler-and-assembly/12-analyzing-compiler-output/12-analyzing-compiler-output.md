# 12. Analyzing Compiler Output

The Go toolchain exposes every stage of compilation: optimization decisions, SSA passes, and final machine instructions. Knowing which tools to reach for — and in what order — lets you answer questions that benchmarks alone cannot: why does this function allocate, why did the inliner refuse, where did the bound check survive. This lesson builds a disciplined top-down methodology and anchors it in four concrete optimization patterns.

```text
compilercheck/
  go.mod
  patterns.go
  patterns_test.go
  cmd/demo/main.go
```

The `patterns` package implements the four functions under study; the test suite verifies their correctness and doubles as the benchmark harness; `cmd/demo` prints disassembly command lines and result summaries.

## Concepts

### The Three-Layer Tool Stack

Work top-down: start with the highest-signal tool and drill only as far as the question requires.

Layer 1 — optimization decisions (`-gcflags='-m'`). The compiler prints one line per decision: whether a variable escaped, whether a call was inlined, whether a bound check was removed. This is the cheapest analysis and often answers the question outright.

```bash
go build -gcflags='-m=2' ./...
```

The `-m=2` level includes reasons for inlining rejections, not just acceptances.

Layer 2 — SSA intermediate form (`GOSSAFUNC`). For a single named function the compiler writes `ssa.html` showing every SSA pass from initial form to final scheduling. Click a value to highlight all its uses; switch between passes to watch optimizations fire.

```bash
GOSSAFUNC=ZeroSlice go build ./...
# opens ssa.html in cwd
```

Use a package-qualified suffix to disambiguate: `GOSSAFUNC=patterns.ZeroSlice`.

Layer 3 — machine instructions (`go tool objdump`, `go tool compile -S`). These show the actual bytes the CPU executes. Use `compile -S` during development (no binary needed) and `objdump` on the linked binary for the final word.

```bash
go build -gcflags='-S' . 2>&1 | grep -A 30 'compilercheck.ZeroSlice'
go build -gcflags='-l' -o /tmp/ccheck ./cmd/demo && go tool objdump -s 'ZeroSlice' /tmp/ccheck
```

### Bounds Check Elimination (BCE)

Every slice index `s[i]` compiles to a bounds check by default. The compiler's BCE pass eliminates checks that are provably safe. The critical rule: if the compiler can see that `i` is in `[0, len(s))` at the point of access — because a prior check dominates the access — it removes the check.

```go
// BCE fires: compiler proves len(s) >= 2 before indexing
func SumTwo(s []int) int {
	_ = s[1] // single panic point; proves len >= 2
	return s[0] + s[1]
}
```

The leading `_ = s[1]` is not dead code: it consolidates the two individual checks into one, letting the compiler prove that both `s[0]` and `s[1]` are safe in a single dominating check. Contrast with:

```go
func SumTwoNaive(s []int) int {
	return s[0] + s[1] // two separate bound checks
}
```

`go build -gcflags='-d=ssa/check_bce/debug=1'` lists every surviving bound check.

### The memclr Pattern

A loop that zeroes a slice:

```go
for i := range s {
    s[i] = 0
}
```

The compiler recognizes this as a `runtime.memclr` call and replaces the loop body with a single CALL or, for small sizes, a sequence of `MOVUPS XMMWORD PTR [...]` (128-bit zero stores). This is visible in `-S` output as `CALL runtime.memclrNoHeapPointers(SB)` or as a run of `PXOR`/`MOVOU` instructions for unrolled small sizes.

A hand-written loop over `s[i] = byte(0)` on a `[]byte` triggers the same pattern. Changing the element type to a non-primitive (e.g. `[]any`) does not — the compiler cannot elide the write barrier.

### Unaligned-Read Pattern Matching

The compiler recognizes sequences of byte-shift operations that reconstruct a multi-byte integer and replaces them with a single load instruction. On amd64, the pattern:

```go
uint32(b[0]) | uint32(b[1])<<8 | uint32(b[2])<<16 | uint32(b[3])<<24
```

compiles to a single `MOVL (reg), reg` — the same instruction generated by `binary.LittleEndian.Uint32(b)`. The SSA pass `rewrite386.go` / `rewriteAMD64.go` contains the pattern-matching rules. To see both paths produce identical output:

```bash
go build -gcflags='-S' . 2>&1 | grep -A 5 'compilercheck.ReadU32 '
go build -gcflags='-S' . 2>&1 | grep -A 5 'compilercheck.ReadU32Stdlib'
```

Both should show a single `MOVL` on amd64.

### Reading Assembly Output Systematically

A `go tool compile -S` line looks like:

```
0x0010 00016 (patterns.go:14)   MOVQ    (AX), CX
```

Fields: hex offset, decimal offset, source location, instruction. `FUNCDATA` and `PCDATA` lines are GC metadata emitted by the compiler; they do not execute and can be ignored when counting instructions.

To count non-metadata instructions for a function:

```bash
go build -gcflags='-S' . 2>&1 \
  | awk '/compilercheck\.FuncName /,/compilercheck\./' \
  | grep -v 'FUNCDATA\|PCDATA\|^\s*$\|compilercheck\.' \
  | wc -l
```

Note: `-S` output lines are prefixed with the full module path (e.g. `example.com/compilercheck.ReadU32 STEXT ...`). The awk range matches on the function header and stops at the next `compilercheck.` symbol header; substitute your actual function name for `FuncName`.

### Exercise 1: The Pattern Functions

Create `patterns.go`:

```go
package compilercheck

import "encoding/binary"

// SumTwo returns s[0]+s[1]. The leading bounds check on s[1]
// proves len(s)>=2 and eliminates the individual checks on s[0] and s[1].
func SumTwo(s []int) int {
	_ = s[1] // BCE sentinel: one check dominates both accesses
	return s[0] + s[1]
}

// SumTwoNaive accesses s[0] and s[1] without a dominating check.
// The compiler may emit two separate bounds checks.
func SumTwoNaive(s []int) int {
	return s[0] + s[1]
}

// ZeroSlice zeroes a []byte. The compiler replaces the range loop
// with runtime.memclrNoHeapPointers (or equivalent SIMD stores).
func ZeroSlice(b []byte) {
	for i := range b {
		b[i] = 0
	}
}

// ReadU32 reconstructs a uint32 from four bytes using explicit shifts.
// On amd64 the compiler pattern-matches this to a single MOVL.
func ReadU32(b []byte) uint32 {
	_ = b[3] // BCE sentinel
	return uint32(b[0]) | uint32(b[1])<<8 | uint32(b[2])<<16 | uint32(b[3])<<24
}

// ReadU32Stdlib uses binary.LittleEndian.Uint32, which the compiler
// also lowers to a single MOVL on amd64. Both functions should
// produce equivalent assembly.
func ReadU32Stdlib(b []byte) uint32 {
	return binary.LittleEndian.Uint32(b)
}
```

### Exercise 2: Run the Optimization Analysis

With the file above, run each analysis layer. These commands are the exercise; the output you read is the artifact.

Layer 1 — optimization decisions:

```bash
go build -gcflags='-m=2' . 2>&1 | head -40
```

Look for lines like `can inline SumTwo` or `does not escape`.

Layer 2 — surviving bounds checks:

```bash
go build -gcflags='-d=ssa/check_bce/debug=1' . 2>&1
```

You should see two bounds-check lines for `SumTwoNaive` (one per access) and exactly one line for `SumTwo` (the dominating `_ = s[1]` sentinel, which subsumes both `s[0]` and `s[1]`).

Layer 3 — assembly listing:

```bash
go build -gcflags='-S' . 2>&1 | grep -A 30 'compilercheck.ZeroSlice'
```

Look for `CALL runtime.memclrNoHeapPointers` or a run of `MOVOU` (128-bit zero stores).

```bash
go build -gcflags='-S' . 2>&1 | grep -A 10 'compilercheck.ReadU32 '
go build -gcflags='-S' . 2>&1 | grep -A 10 'compilercheck.ReadU32Stdlib'
```

On amd64 both should show a single `MOVL` load plus the bounds check sentinel.

### Exercise 3: Test the Correctness Contract

The optimizer must not change observable behavior. Create `patterns_test.go`:

```go
package compilercheck

import (
	"encoding/binary"
	"math/rand"
	"testing"

func TestSumTwoMatchesNaive(t *testing.T) {
	t.Parallel()

	cases := []struct {
		name string
		s    []int
		want int
	}{
		{"small", []int{1, 2}, 3},
		{"negative", []int{-5, 3}, -2},
		{"zeros", []int{0, 0, 9}, 0},
		{"large", []int{1000000, 2000000}, 3000000},
	}
	for _, tc := range cases {
		t.Run(tc.name, func(t *testing.T) {
			t.Parallel()
			got := SumTwo(tc.s)
			naive := SumTwoNaive(tc.s)
			if got != tc.want {
				t.Errorf("SumTwo(%v) = %d, want %d", tc.s, got, tc.want)
			}
			if got != naive {
				t.Errorf("SumTwo(%v) = %d != SumTwoNaive = %d", tc.s, got, naive)
			}
		})
	}
}

func TestSumTwoPanicsOnShortSlice(t *testing.T) {
	t.Parallel()

	defer func() {
		if r := recover(); r == nil {
			t.Error("expected panic for len(s) < 2")
		}
	}()
	SumTwo([]int{42})
}

	t.Parallel()

	b := []byte{1, 2, 3, 4, 5}
	ZeroSlice(b)

			t.Errorf("b[%d] = %d after ZeroSlice, want 0", i, v)
		}
	}

func TestZeroSliceEmpty(t *testing.T) {

	// Must not panic on empty slice.

func TestReadU32MatchesStdlib(t *testing.T) {
	t.Parallel()

	rng := rand.New(rand.NewSource(42))
	for i := 0; i < 1000; i++ {
		b := make([]byte, 4)
		rng.Read(b)
		got := ReadU32(b)
		want := binary.LittleEndian.Uint32(b)
		if got != want {
			t.Errorf("ReadU32(%x) = %d, want %d", b, got, want)
		}
	}

func TestReadU32KnownValues(t *testing.T) {

	cases := []struct {
		b    []byte
		want uint32
	}{
		{[]byte{0x01, 0x00, 0x00, 0x00}, 1},
		{[]byte{0x00, 0x01, 0x00, 0x00}, 256},
```

## Common Mistakes

Wrong: using `go build -gcflags='-m'` on the entire module and piping raw output without filtering — the noise from transitive dependencies overwhelms the signal.

What happens: you see thousands of lines for stdlib internals, not for your package.

Fix: scope the flag to your package only (`go build -gcflags='example.com/compilercheck=-m=2' ./...`) or use `go tool compile -S` directly on the file.

---

Wrong: assuming that `go build -gcflags='-S'` output and `go tool objdump` output are the same thing, or that `objdump` will show every function from a linked binary.

What happens: `-gcflags='-S'` shows the semi-abstract Go assembly before linking. `objdump` shows the actual linked machine code, but the linker inlines small leaf functions — so a function like `ZeroSlice` that the compiler inlines into callers may not appear as a standalone symbol in the binary at all. Running `go tool objdump -s 'ZeroSlice' /tmp/ccheck` on a default build returns no output because the symbol was inlined away.

Fix: build with inlining disabled (`-gcflags='-l'`) when you need a named symbol to survive in the binary for `objdump`: `go build -gcflags='-l' -o /tmp/ccheck ./cmd/demo`. Use `-gcflags='-S'` for fast per-package assembly inspection during development.

---

Wrong: concluding that BCE fired because the `_ = s[1]` sentinel compiles; not verifying with `-d=ssa/check_bce/debug=1`.

What happens: the sentinel may have been eliminated by other passes, and a bound check may still be present at the actual index.

Fix: always confirm with `-d=ssa/check_bce/debug=1`, which enumerates every surviving check. A line like `bounds check` pointing to the index expression means the check was not eliminated.

---

Wrong: counting `FUNCDATA` and `PCDATA` lines when comparing instruction counts between two versions of a function.

What happens: these are GC metadata directives, not executable instructions. Including them inflates the count and may reverse an apparent win/loss.

Fix: filter them before counting: `grep -v 'FUNCDATA\|PCDATA'`.

## Summary

- Work top-down: `-gcflags='-m'` first, SSA HTML second, `objdump` third.
- BCE consolidates multiple bound checks into a single dominating check when a sentinel access proves the slice length.
- The compiler recognizes range-loop zeroing as `memclr` and replaces it with a runtime call or SIMD stores.
- Explicit byte-shift expressions that reconstruct an integer pattern-match to a single load instruction on amd64.
- `FUNCDATA`/`PCDATA` lines in `-S` output are GC metadata, not instructions; exclude them from instruction counts.
- `compile -S` operates on a source file before linking; `objdump` operates on the linked binary. Use the layer that matches your question.
