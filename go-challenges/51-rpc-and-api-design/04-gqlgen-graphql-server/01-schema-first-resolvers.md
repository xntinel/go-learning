# Exercise 1: Schema-First Codegen — Wiring Query and Mutation Resolvers

A GraphQL service starts from its schema, not its Go. This exercise writes a
`schema.graphqls` for a small product catalog, runs `gqlgen generate` to bind
each field to a typed resolver, implements those resolvers against an in-memory
mutex-guarded store, and serves the whole thing through an explicitly wired
`handler.New` rather than the demo-only `handler.NewDefaultServer`.

This module is self-contained: its own `go mod init`, its own schema, resolvers,
demo, and tests. It is a bar-mode lesson — it depends on `github.com/99designs/
gqlgen` and its generated code, so it does not build in an offline gate; the code
is written to be correct in shape and API, and gofmt-clean.

## What you'll build

```text

    server_test.go               client tests: query, mutation, unknown-id error
```

Implement: the `ProductStore`, the `Query.products`/`Query.product`/`Mutation.createProduct` resolvers, and `NewServer` wired with `handler.New` (POST/GET/Options transports, LRU query cache, playground).

### The schema is the contract

Write the SDL first. It names the types other teams will code against; everything
Go is derived from it. `products` returns a non-null list of non-null products,
`product(id)` is nullable so a miss is `null` rather than an error on a required
field, and `createProduct` takes an input object and returns the created product.

Create `graph/schema.graphqls`:

```graphql
type Product {
  id: ID!
  name: String!
  priceCents: Int!
}

input NewProduct {
  name: String!
  priceCents: Int!
}

type Query {
  products: [Product!]!
  product(id: ID!): Product
}

type Mutation {
  createProduct(input: NewProduct!): Product!
}
```

### The codegen config

`gqlgen.yml` tells the generator where the schema is, where to write the
executable schema and models, and to use the `follow-schema` resolver layout so
new resolver stubs land next to your existing implementations on every regen.

Create `gqlgen.yml`:

```yaml
schema:
  - graph/*.graphqls

exec:
  filename: graph/generated.go
  package: graph

model:
  filename: graph/model/models_gen.go
  package: model

resolver:
  layout: follow-schema
  dir: graph
  package: graph
  filename_template: "{name}.resolvers.go"

models:
  ID:
    model:
      - github.com/99designs/gqlgen/graphql.ID
  Int:
    model:
      - github.com/99designs/gqlgen/graphql.Int
```

Run codegen. It writes `graph/generated.go` and `graph/model/models_gen.go`, and
stubs `graph/schema.resolvers.go` the first time:

```bash
go run github.com/99designs/gqlgen generate
```

Commit those generated files so the module builds and tests without re-running
the tool. Never hand-edit them.

### The store your resolvers speak to

The root `Resolver` holds your dependencies. Here that is a single in-memory
`ProductStore`. It is mutex-guarded on purpose: gqlgen resolves sibling fields
concurrently, so a `products` query that also touches the store on another
goroutine must not race. Reads take the `RLock`, the write takes the full lock,
and `All` returns a stable id-sorted slice so query output is deterministic.

Create `graph/resolver.go`:

```go
package graph

import (
	"sort"
	"strconv"
	"sync"

	"example.com/catalog/graph/model"
)

// Resolver is the root object graph.NewExecutableSchema binds to. It owns the
// dependencies resolver methods need -- here, one concurrency-safe store.
type Resolver struct {
	store *ProductStore
}

// NewResolver builds a Resolver over a fresh in-memory store.
func NewResolver() *Resolver {
	return &Resolver{store: NewProductStore()}
}

// ProductStore is an in-memory catalog. gqlgen runs sibling resolvers
// concurrently, so every access is synchronized.
type ProductStore struct {
	mu   sync.RWMutex
	seq  int
	byID map[string]*model.Product
}

// NewProductStore returns an empty store.
func NewProductStore() *ProductStore {
	return &ProductStore{byID: make(map[string]*model.Product)}
}

// All returns every product, id-sorted for deterministic output.
func (s *ProductStore) All() []*model.Product {
	s.mu.RLock()
	defer s.mu.RUnlock()
	out := make([]*model.Product, 0, len(s.byID))
	for _, p := range s.byID {
		out = append(out, p)
	}
	sort.Slice(out, func(i, j int) bool { return out[i].ID < out[j].ID })
	return out
}

// Get returns the product for id and whether it existed.
func (s *ProductStore) Get(id string) (*model.Product, bool) {
	s.mu.RLock()
	defer s.mu.RUnlock()
	p, ok := s.byID[id]
	return p, ok
}

// Add stores a new product under a fresh sequential id and returns it.
func (s *ProductStore) Add(name string, priceCents int) *model.Product {
	s.mu.Lock()
	defer s.mu.Unlock()
	s.seq++
	id := strconv.Itoa(s.seq)
	p := &model.Product{ID: id, Name: name, PriceCents: priceCents}
	s.byID[id] = p
	return p
}
```

### The resolvers gqlgen bound

`schema.resolvers.go` is where you fill in the stubbed method bodies. gqlgen
generated the interfaces (`QueryResolver`, `MutationResolver`) and the binding
methods; the signatures are fixed by the schema. `product` returns
`(*model.Product, error)`: a miss returns `(nil, ErrNotFound)`, and because the
schema made `product` nullable, that surfaces as `null` data plus an `errors`
entry rather than nulling anything else. `ErrNotFound` is a package sentinel
wrapped with `%w` so callers can `errors.Is` it — Exercise 2 turns that into a
client-safe presented error.

Create `graph/schema.resolvers.go`:

```go
package graph

import (
	"context"
	"errors"
	"fmt"

	"example.com/catalog/graph/model"
)

// ErrNotFound is returned when a product id has no match. It is wrapped with %w
// so callers can errors.Is it.
var ErrNotFound = errors.New("product not found")

// CreateProduct stores a new product and echoes it back.
func (r *mutationResolver) CreateProduct(ctx context.Context, input model.NewProduct) (*model.Product, error) {
	return r.store.Add(input.Name, input.PriceCents), nil
}

// Products returns every stored product.
func (r *queryResolver) Products(ctx context.Context) ([]*model.Product, error) {
	return r.store.All(), nil
}

// Product returns one product by id, or ErrNotFound.
func (r *queryResolver) Product(ctx context.Context, id string) (*model.Product, error) {
	p, ok := r.store.Get(id)
	if !ok {
		return nil, fmt.Errorf("product %q: %w", id, ErrNotFound)
	}
	return p, nil
}

// Mutation returns the resolver bound to Mutation fields.
func (r *Resolver) Mutation() MutationResolver { return &mutationResolver{r} }

// Query returns the resolver bound to Query fields.
func (r *Resolver) Query() QueryResolver { return &queryResolver{r} }

type mutationResolver struct{ *Resolver }
type queryResolver struct{ *Resolver }
```

### Explicit production-shaped wiring

This is the point of the lesson. `NewServer` uses `handler.New`, not
`handler.NewDefaultServer`, and opts into everything by hand. Transports are
registered explicitly (a missing transport answers HTTP 422 "transport not
supported"). `SetQueryCache(lru.New[*ast.QueryDocument](1000))` caches parsed and
validated query documents so an identical query is not re-parsed each request. A
separate playground handler is mounted for humans; note it is a distinct handler,
not part of the query endpoint.

Create `graph/server.go`:

```go
package graph

import (
	"net/http"

	"github.com/99designs/gqlgen/graphql/handler"
	"github.com/99designs/gqlgen/graphql/handler/lru"
	"github.com/99designs/gqlgen/graphql/handler/transport"
	"github.com/99designs/gqlgen/graphql/playground"
	"github.com/vektah/gqlparser/v2/ast"
)

// NewServer wires a production-shaped GraphQL handler. It deliberately avoids
// handler.NewDefaultServer, which ships no query cache and no complexity limit
// and leaves introspection on. Transports are opt-in.
func NewServer() *handler.Server {
	srv := handler.New(NewExecutableSchema(Config{Resolvers: NewResolver()}))

	srv.AddTransport(transport.Options{})
	srv.AddTransport(transport.GET{})
	srv.AddTransport(transport.POST{})

	srv.SetQueryCache(lru.New[*ast.QueryDocument](1000))

	return srv
}

// Playground returns an HTTP handler serving the GraphQL playground UI pointed
// at the query endpoint.
func Playground(endpoint string) http.Handler {
	return playground.Handler("Catalog", endpoint)
}
```

### The runnable server

`cmd/server/main.go` mounts the query endpoint and the playground and listens.
It is the real entry point; it does not print an expected output (it blocks).

Create `cmd/server/main.go`:

```go
package main

import (
	"log"
	"net/http"

	"example.com/catalog/graph"
)

func main() {
	http.Handle("/query", graph.NewServer())
	http.Handle("/", graph.Playground("/query"))

	log.Println("listening on http://localhost:8080/ (playground) and /query")
	log.Fatal(http.ListenAndServe(":8080", nil))
}
```

### The demo

`cmd/demo/main.go` proves the wiring end to end in-process. It starts the handler
behind `httptest.NewServer`, POSTs a mutation then two queries, and prints the
raw JSON bodies. The third call is the teaching moment: an unknown id returns
HTTP 200 with `null` data and an `errors` array — the status code says nothing is
wrong.

Create `cmd/demo/main.go`:

```go
package main

import (
	"bytes"
	"encoding/json"
	"fmt"
	"io"
	"net/http"
	"net/http/httptest"

	"example.com/catalog/graph"
)

func post(url, query string) string {
	body, _ := json.Marshal(map[string]string{"query": query})
	resp, err := http.Post(url, "application/json", bytes.NewReader(body))
	if err != nil {
		panic(err)
	}
	defer resp.Body.Close()
	out, _ := io.ReadAll(resp.Body)
	return fmt.Sprintf("HTTP %d %s", resp.StatusCode, bytes.TrimSpace(out))
}

func main() {
	ts := httptest.NewServer(graph.NewServer())
	defer ts.Close()

	fmt.Println(post(ts.URL, `mutation { createProduct(input:{name:"Keyboard", priceCents:4999}) { id name priceCents } }`))
	fmt.Println(post(ts.URL, `{ products { id name priceCents } }`))
	fmt.Println(post(ts.URL, `{ product(id:"999") { name } }`))
}
```

Run it:

```bash
go run ./cmd/demo
```

Expected output:

```
HTTP 200 {"data":{"createProduct":{"id":"1","name":"Keyboard","priceCents":4999}}}
HTTP 200 {"data":{"products":[{"id":"1","name":"Keyboard","priceCents":4999}]}}
HTTP 200 {"errors":[{"message":"product \"999\": product not found","path":["product"]}],"data":{"product":null}}
```

### Tests

The tests use gqlgen's own `client` package against a `handler.New` server, which
is the idiomatic way to test resolvers without a network. `newTestClient` wires
POST plus introspection (the client's typed decode relies on the response shape).
The table decodes into typed structs, `client.Var` drives the parameterized
lookup, and the unknown-id case uses `client.Post` (not `MustPost`) so the
returned error can be inspected — proving a field error is reported in the
`errors` array while `data` is null. The `Example` at the end pins one decoded
round trip with an `// Output` line.

Create `graph/server_test.go`:

```go
package graph

import (
	"fmt"
	"strings"
	"testing"

	"github.com/99designs/gqlgen/client"
	"github.com/99designs/gqlgen/graphql/handler"
	"github.com/99designs/gqlgen/graphql/handler/extension"
	"github.com/99designs/gqlgen/graphql/handler/transport"
)

func newTestClient() *client.Client {
	srv := handler.New(NewExecutableSchema(Config{Resolvers: NewResolver()}))
	srv.AddTransport(transport.POST{})
	srv.Use(extension.Introspection{})
	return client.New(srv)
}

func TestCreateAndQuery(t *testing.T) {
	t.Parallel()
	c := newTestClient()

	var created struct {
		CreateProduct struct {
			ID         string
			Name       string
			PriceCents int
		}
	}
	c.MustPost(`mutation { createProduct(input:{name:"Mouse", priceCents:2500}) { id name priceCents } }`, &created)
	if created.CreateProduct.ID != "1" || created.CreateProduct.Name != "Mouse" || created.CreateProduct.PriceCents != 2500 {
		t.Fatalf("createProduct = %+v, want id=1 name=Mouse price=2500", created.CreateProduct)
	}

	var listed struct {
		Products []struct {
			ID   string
			Name string
		}
	}
	c.MustPost(`{ products { id name } }`, &listed)
	if len(listed.Products) != 1 || listed.Products[0].Name != "Mouse" {
		t.Fatalf("products = %+v, want one Mouse", listed.Products)
	}
}

func TestProductByIDVariable(t *testing.T) {
	t.Parallel()
	c := newTestClient()
	c.MustPost(`mutation { createProduct(input:{name:"Cable", priceCents:799}) { id } }`, &struct {
		CreateProduct struct{ ID string }
	}{})

	var resp struct {
		Product *struct {
			Name string
		}
	}
	c.MustPost(`query($id: ID!) { product(id:$id) { name } }`, &resp, client.Var("id", "1"))
	if resp.Product == nil || resp.Product.Name != "Cable" {
		t.Fatalf("product(1) = %+v, want Cable", resp.Product)
	}
}

func TestUnknownIDErrors(t *testing.T) {
	t.Parallel()
	c := newTestClient()

	var resp struct {
		Product *struct{ Name string }
	}
	err := c.Post(`{ product(id:"404") { name } }`, &resp)
	if err == nil {
		t.Fatal("expected an errors array for an unknown id, got nil")
	}
	if !strings.Contains(err.Error(), "product not found") {
		t.Fatalf("error = %q, want it to mention 'product not found'", err.Error())
	}
	if resp.Product != nil {
		t.Fatalf("data.product = %+v, want null alongside the error", resp.Product)
	}
}

func Example() {
	c := newTestClient()
	var resp struct {
		CreateProduct struct {
			ID         string
			Name       string
			PriceCents int
		}
	}
	c.MustPost(`mutation { createProduct(input:{name:"Monitor", priceCents:19999}) { id name priceCents } }`, &resp)
	fmt.Printf("%s %s %d\n", resp.CreateProduct.ID, resp.CreateProduct.Name, resp.CreateProduct.PriceCents)
	// Output: 1 Monitor 19999
}
```

## Review

The service is correct when the schema and the resolvers cannot disagree: the
signatures in `schema.resolvers.go` are dictated by `schema.graphqls`, so a schema
change that your code has not caught up to is a build failure, not a runtime 500.
Confirm behavior three ways: `TestCreateAndQuery` shows the mutation mutates the
store and the query reads it back, `TestProductByIDVariable` shows `client.Var`
feeding a typed variable, and `TestUnknownIDErrors` shows a field error arriving
in the `errors` array with `data.product` null while the HTTP status stays 200.

The mistakes to avoid are the ones the concepts file names. Do not reach for
`handler.NewDefaultServer` — `NewServer` opts into the query cache and (in the
next exercise) the complexity limit and error presenter deliberately. Do not
forget `AddTransport`: without `transport.POST{}` every query is answered with a
confusing HTTP 422. Do not hand-edit `generated.go` or `models_gen.go`; change the
schema and regenerate. And keep the store mutex honest — run `go test -race` so a
future field resolver that shares the store cannot introduce a silent data race.
