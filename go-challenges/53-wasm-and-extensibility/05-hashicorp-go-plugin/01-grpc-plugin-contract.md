# Exercise 1: A gRPC Plugin Contract and Server

Every out-of-process plugin author ships against one thing: the contract. This
exercise builds that contract — a protobuf service, a `plugin.GRPCPlugin` adapter
that bridges the generated stubs to a clean Go interface, the plugin-side
implementation of that interface, a `HandshakeConfig` with a magic cookie, and
`plugin.Serve` wiring it into a runnable plugin binary.

This module is fully self-contained: its own `go mod init`, its own generated
proto package, its own demo and tests. Nothing here imports another exercise. It
is a bar-mode module — it depends on `hashicorp/go-plugin`, `grpc`, and a
protoc-generated package, so it does not build offline; the value is the correct
shape, verified APIs, and a deterministic in-process test harness.

## What you'll build

```text
processorplugin/                 independent module: example.com/processorplugin
  go.mod                         go 1.26; requires go-plugin, grpc, protobuf
  proto/
    processor.proto              service Processor { rpc Process }
    processor.pb.go              generated (protoc-gen-go)
    processor_grpc.pb.go         generated (protoc-gen-go-grpc)
  shared/
    processor.go                 Processor interface, Handshake, ProcessorGRPCPlugin, grpc bridge, Validate/ErrEmptyBatch
  plugin/
    main.go                      package main: redactor impl + plugin.Serve
  cmd/
    demo/
      main.go                    in-process reattach harness: serve, dispense, Process
  shared_test.go                 Validate table test (errors.Is), reattach round-trip, Example
```

Files: `proto/processor.proto`, `shared/processor.go`, `plugin/main.go`, `cmd/demo/main.go`, `shared_test.go`.
Implement: the `Processor` interface, the `ProcessorGRPCPlugin` adapter (`GRPCServer`/`GRPCClient`) bridging proto stubs to the interface, a `HandshakeConfig` with a magic cookie, and `plugin.Serve` wiring.
Test: a pure `Validate` table test asserting `ErrEmptyBatch` via `errors.Is`; a reattach round-trip that serves the plugin in-process, dispenses it, and asserts a batch is processed plus an error path.
Verify: `go test -race ./...` with the deps present (this is a bar-mode module; the offline gate fails to build).

### The contract, from the wire up

A go-plugin gRPC plugin has three layers, and keeping them separate is what makes
the plugin author's job clean. At the bottom is the *wire schema* — a protobuf
service. In the middle is the *bridge* — a `plugin.GRPCPlugin` that translates
between the generated protobuf stubs and a hand-written Go interface. At the top is
the *domain interface* that host and plugin code actually program against, which
never mentions protobuf. The host calls `Processor.Process(ctx, ProcessRequest)`;
it neither knows nor cares that underneath, a `*pb.ProcessRequest` was marshaled
across a socket.

Start with the schema. The service is deliberately *coarse*: one `Process` RPC
that takes a whole batch of records, not a per-record call. This is the single
most important design decision in an out-of-process plugin — every call is
serialized and round-tripped, so a chatty interface is a latency and allocation
trap.

Create `proto/processor.proto`:

```proto
syntax = "proto3";

package processor;

option go_package = "example.com/processorplugin/proto";

message Record {
  string id = 1;
  string body = 2;
}

message ProcessRequest {
  repeated Record records = 1;
}

message ProcessResponse {
  repeated Record records = 1;
  int64 processed = 2;
}

service Processor {
  rpc Process(ProcessRequest) returns (ProcessResponse);
}
```

Running `protoc` on that file produces `proto/processor.pb.go` and
`proto/processor_grpc.pb.go`. You do not hand-write those; the important part is
the shape of the symbols they export, because the bridge binds to them. From the
generated code, the bridge uses:

```go
// Generated by protoc-gen-go / protoc-gen-go-grpc (excerpt of the public shape):
type Record struct{ Id, Body string }

func (x *Record) GetId() string
func (x *Record) GetBody() string

type ProcessRequest struct{ Records []*Record }

func (x *ProcessRequest) GetRecords() []*Record

type ProcessResponse struct {
	Records   []*Record
	Processed int64
}

type ProcessorServer interface {
	Process(context.Context, *ProcessRequest) (*ProcessResponse, error)
	mustEmbedUnimplementedProcessorServer()
}
type UnimplementedProcessorServer struct{}

type ProcessorClient interface {
	Process(context.Context, *ProcessRequest, ...grpc.CallOption) (*ProcessResponse, error)
}

func RegisterProcessorServer(s grpc.ServiceRegistrar, srv ProcessorServer)
func NewProcessorClient(cc grpc.ClientConnInterface) ProcessorClient
```

### The shared package: interface, handshake, and the bridge

`shared/processor.go` is the file both the host and the plugin author import. It
defines the domain types (`Record`, `ProcessRequest`, `ProcessResponse`, and the
`Processor` interface), the `Handshake`, the `PluginMap`, and the
`ProcessorGRPCPlugin` bridge.

The handshake carries the magic cookie and the protocol version. The magic cookie
is not a secret in the cryptographic sense — it is a guard so the plugin binary
refuses to run unless launched by a host that sets the matching environment
variable, which stops the binary from being usefully executed on its own.
`ProtocolVersion` is the ABI contract: bump it whenever this interface or the
proto changes, and old plugins fail the handshake with a readable error.

`ProcessorGRPCPlugin` embeds `plugin.NetRPCUnsupportedPlugin`. That embed is not
decoration: `plugin.PluginSet` is `map[string]plugin.Plugin`, and the base
`plugin.Plugin` interface requires net/rpc `Server`/`Client` methods. A gRPC-only
plugin embeds `NetRPCUnsupportedPlugin` to inherit stubbed versions of those, so
it satisfies `plugin.Plugin` while really implementing `plugin.GRPCPlugin`. The
two methods that matter are `GRPCServer` (called on the plugin side — it registers
the concrete implementation onto the gRPC server) and `GRPCClient` (called on the
host side — it returns the client-side adapter that satisfies `Processor`).

`grpcServer` and `grpcClient` are the translation layer. `grpcServer` embeds
`pb.UnimplementedProcessorServer` (forward compatibility: new RPCs added to the
proto will not break the build) and converts the incoming `*pb.ProcessRequest`
into the domain `ProcessRequest`, calls the real `Impl`, and converts the result
back. `grpcClient` does the mirror image for the host. `Validate` and
`ErrEmptyBatch` live here as pure, host-testable logic — note that a sentinel
error returned by the plugin does *not* survive the gRPC boundary as itself, so
validation you want to assert with `errors.Is` must run on a value, not across the
wire.

Create `shared/processor.go`:

```go
package shared

import (
	"context"
	"errors"
	"fmt"

	plugin "github.com/hashicorp/go-plugin"
	"google.golang.org/grpc"

	pb "example.com/processorplugin/proto"
)

// ErrEmptyBatch is returned by Validate when a request carries no records.
var ErrEmptyBatch = errors.New("processor: empty batch")

// Record is a domain record handed to the plugin for processing.
type Record struct {
	ID   string
	Body string
}

// ProcessRequest is a coarse-grained batch. The interface is intentionally not
// per-record: every call crosses a process boundary and is serialized, so work
// is batched to amortize that cost.
type ProcessRequest struct {
	Records []Record
}

// Validate rejects an empty batch. It wraps ErrEmptyBatch with %w so callers on
// the host side of the wire can assert it with errors.Is.
func (r ProcessRequest) Validate() error {
	if len(r.Records) == 0 {
		return fmt.Errorf("validate request: %w", ErrEmptyBatch)
	}
	return nil
}

// ProcessResponse carries the transformed records and a processed count.
type ProcessResponse struct {
	Records   []Record
	Processed int64
}

// Processor is the domain interface both host and plugin program against. It
// never mentions protobuf.
type Processor interface {
	Process(ctx context.Context, req ProcessRequest) (ProcessResponse, error)
}

// Handshake gates the connection. The magic cookie stops the plugin binary from
// being runnable on its own; ProtocolVersion is the ABI contract.
var Handshake = plugin.HandshakeConfig{
	ProtocolVersion:  1,
	MagicCookieKey:   "PROCESSOR_PLUGIN",
	MagicCookieValue: "8f2a1c7e-processor-v1",
}

// PluginMap is the set of plugins this contract exposes, keyed by dispense name.
var PluginMap = plugin.PluginSet{
	"processor": &ProcessorGRPCPlugin{},
}

// ProcessorGRPCPlugin bridges the Processor interface to gRPC. Embedding
// NetRPCUnsupportedPlugin satisfies the base plugin.Plugin interface for a
// gRPC-only plugin. Impl is set only on the plugin (server) side.
type ProcessorGRPCPlugin struct {
	plugin.NetRPCUnsupportedPlugin
	Impl Processor
}

// GRPCServer runs on the plugin side: it registers the concrete implementation.
func (p *ProcessorGRPCPlugin) GRPCServer(_ *plugin.GRPCBroker, s *grpc.Server) error {
	pb.RegisterProcessorServer(s, &grpcServer{impl: p.Impl})
	return nil
}

// GRPCClient runs on the host side: it returns the adapter that satisfies Processor.
func (p *ProcessorGRPCPlugin) GRPCClient(_ context.Context, _ *plugin.GRPCBroker, c *grpc.ClientConn) (interface{}, error) {
	return &grpcClient{client: pb.NewProcessorClient(c)}, nil
}

// grpcServer adapts an incoming gRPC call to the domain Processor.
type grpcServer struct {
	pb.UnimplementedProcessorServer
	impl Processor
}

func (s *grpcServer) Process(ctx context.Context, req *pb.ProcessRequest) (*pb.ProcessResponse, error) {
	out, err := s.impl.Process(ctx, ProcessRequest{Records: fromProto(req.GetRecords())})
	if err != nil {
		return nil, err
	}
	return &pb.ProcessResponse{Records: toProto(out.Records), Processed: out.Processed}, nil
}

// grpcClient adapts the domain Processor to an outgoing gRPC call.
type grpcClient struct {
	client pb.ProcessorClient
}

func (c *grpcClient) Process(ctx context.Context, req ProcessRequest) (ProcessResponse, error) {
	resp, err := c.client.Process(ctx, &pb.ProcessRequest{Records: toProto(req.Records)})
	if err != nil {
		return ProcessResponse{}, err
	}
	return ProcessResponse{Records: fromProto(resp.GetRecords()), Processed: resp.GetProcessed()}, nil
}

func toProto(rs []Record) []*pb.Record {
	out := make([]*pb.Record, len(rs))
	for i, r := range rs {
		out[i] = &pb.Record{Id: r.ID, Body: r.Body}
	}
	return out
}

func fromProto(rs []*pb.Record) []Record {
	out := make([]Record, len(rs))
	for i, r := range rs {
		out[i] = Record{ID: r.GetId(), Body: r.GetBody()}
	}
	return out
}
```

### The plugin binary

The plugin is a `package main` that implements `Processor` and hands it to
`plugin.Serve`. This one is a PII redactor: it masks card-number-shaped digit runs
in each record's body. `Serve` blocks, speaking the go-plugin protocol on stdio
until the host closes the connection. `GRPCServer: plugin.DefaultGRPCServer` tells
go-plugin to serve over gRPC.

Create `plugin/main.go`:

```go
package main

import (
	"context"
	"regexp"

	plugin "github.com/hashicorp/go-plugin"

	"example.com/processorplugin/shared"
)

// cardRE matches a run of 13-19 digits, the shape of a payment card number.
var cardRE = regexp.MustCompile(`\b\d{13,19}\b`)

// redactor is the plugin's implementation of shared.Processor.
type redactor struct{}

func (redactor) Process(_ context.Context, req shared.ProcessRequest) (shared.ProcessResponse, error) {
	if err := req.Validate(); err != nil {
		return shared.ProcessResponse{}, err
	}
	out := make([]shared.Record, len(req.Records))
	for i, r := range req.Records {
		out[i] = shared.Record{ID: r.ID, Body: cardRE.ReplaceAllString(r.Body, "[REDACTED]")}
	}
	return shared.ProcessResponse{Records: out, Processed: int64(len(out))}, nil
}

func main() {
	plugin.Serve(&plugin.ServeConfig{
		HandshakeConfig: shared.Handshake,
		Plugins: plugin.PluginSet{
			"processor": &shared.ProcessorGRPCPlugin{Impl: redactor{}},
		},
		GRPCServer: plugin.DefaultGRPCServer,
	})
}
```

### The runnable demo

The demo does not spawn a subprocess. It uses go-plugin's in-process test harness
so the whole contract runs in one process and produces deterministic output:
`plugin.Serve` runs in a goroutine with `ServeConfig.Test`, which hands back a
`*plugin.ReattachConfig`; the client connects with `ClientConfig.Reattach` and no
`exec`. This is the same pattern the tests use, and it is the honest way to show a
plugin round trip without a compiled binary.

Create `cmd/demo/main.go`:

```go
package main

import (
	"context"
	"fmt"
	"log"
	"regexp"

	plugin "github.com/hashicorp/go-plugin"

	"example.com/processorplugin/shared"
)

var cardRE = regexp.MustCompile(`\b\d{13,19}\b`)

// demoImpl is an in-process stand-in for the plugin binary's redactor: it applies
// the same card-masking rule.
type demoImpl struct{}

func (demoImpl) Process(_ context.Context, req shared.ProcessRequest) (shared.ProcessResponse, error) {
	if err := req.Validate(); err != nil {
		return shared.ProcessResponse{}, err
	}
	out := make([]shared.Record, len(req.Records))
	for i, r := range req.Records {
		out[i] = shared.Record{ID: r.ID, Body: cardRE.ReplaceAllString(r.Body, "[REDACTED]")}
	}
	return shared.ProcessResponse{Records: out, Processed: int64(len(out))}, nil
}

func main() {
	ch := make(chan *plugin.ReattachConfig, 1)
	closeCh := make(chan struct{})
	go plugin.Serve(&plugin.ServeConfig{
		HandshakeConfig: shared.Handshake,
		Plugins: plugin.PluginSet{
			"processor": &shared.ProcessorGRPCPlugin{Impl: demoImpl{}},
		},
		GRPCServer: plugin.DefaultGRPCServer,
		Test:       &plugin.ServeTestConfig{ReattachConfigCh: ch, CloseCh: closeCh},
	})

	client := plugin.NewClient(&plugin.ClientConfig{
		HandshakeConfig:  shared.Handshake,
		Plugins:          shared.PluginMap,
		Reattach:         <-ch,
		AllowedProtocols: []plugin.Protocol{plugin.ProtocolGRPC},
	})
	defer client.Kill()

	rpc, err := client.Client()
	if err != nil {
		log.Fatalf("connect: %v", err)
	}
	raw, err := rpc.Dispense("processor")
	if err != nil {
		log.Fatalf("dispense: %v", err)
	}
	proc := raw.(shared.Processor)

	resp, err := proc.Process(context.Background(), shared.ProcessRequest{
		Records: []shared.Record{
			{ID: "r1", Body: "charge 4111111111111111 approved"},
			{ID: "r2", Body: "hello world"},
		},
	})
	if err != nil {
		log.Fatalf("process: %v", err)
	}

	fmt.Printf("processed %d records\n", resp.Processed)
	for _, r := range resp.Records {
		fmt.Printf("%s: %s\n", r.ID, r.Body)
	}
}
```

Run it:

```bash
go run ./cmd/demo
```

Expected output:

```
processed 2 records
r1: charge [REDACTED] approved
r2: hello world
```

### Tests

Two kinds of test. `TestValidate` is pure and deterministic: it exercises the
one piece of logic whose error identity you can assert, checking `ErrEmptyBatch`
travels through `%w` and is caught by `errors.Is`. `TestRoundTrip` uses the
reattach harness to prove the whole bridge works end to end in-process: it serves
a stub implementation, dispenses the `Processor`, and asserts both the happy path
(records processed) and that a plugin-returned error surfaces as a non-nil error
on the host (it is a gRPC status now, not the sentinel, which is exactly why the
assertion checks for non-nil rather than `errors.Is`).

Create `shared_test.go`:

```go
package shared

import (
	"context"
	"errors"
	"fmt"
	"testing"

	plugin "github.com/hashicorp/go-plugin"
)

// stub is an in-test Processor: it uppercases nothing, just echoes and counts,
// and rejects an empty batch so the error path can be exercised.
type stub struct{}

func (stub) Process(_ context.Context, req ProcessRequest) (ProcessResponse, error) {
	if err := req.Validate(); err != nil {
		return ProcessResponse{}, err
	}
	return ProcessResponse{Records: req.Records, Processed: int64(len(req.Records))}, nil
}

func TestValidate(t *testing.T) {
	t.Parallel()
	tests := []struct {
		name    string
		req     ProcessRequest
		wantErr error
	}{
		{"empty", ProcessRequest{}, ErrEmptyBatch},
		{"one record", ProcessRequest{Records: []Record{{ID: "a"}}}, nil},
	}
	for _, tc := range tests {
		t.Run(tc.name, func(t *testing.T) {
			t.Parallel()
			err := tc.req.Validate()
			if tc.wantErr == nil {
				if err != nil {
					t.Fatalf("Validate() = %v, want nil", err)
				}
				return
			}
			if !errors.Is(err, tc.wantErr) {
				t.Fatalf("Validate() = %v, want errors.Is %v", err, tc.wantErr)
			}
		})
	}
}

// dispenseTest serves a stub in-process and returns a connected Processor.
func dispenseTest(t *testing.T) Processor {
	t.Helper()
	ch := make(chan *plugin.ReattachConfig, 1)
	closeCh := make(chan struct{})
	go plugin.Serve(&plugin.ServeConfig{
		HandshakeConfig: Handshake,
		Plugins:         plugin.PluginSet{"processor": &ProcessorGRPCPlugin{Impl: stub{}}},
		GRPCServer:      plugin.DefaultGRPCServer,
		Test:            &plugin.ServeTestConfig{ReattachConfigCh: ch, CloseCh: closeCh},
	})
	client := plugin.NewClient(&plugin.ClientConfig{
		HandshakeConfig:  Handshake,
		Plugins:          PluginMap,
		Reattach:         <-ch,
		AllowedProtocols: []plugin.Protocol{plugin.ProtocolGRPC},
	})
	t.Cleanup(client.Kill)
	rpc, err := client.Client()
	if err != nil {
		t.Fatalf("connect: %v", err)
	}
	raw, err := rpc.Dispense("processor")
	if err != nil {
		t.Fatalf("dispense: %v", err)
	}
	return raw.(Processor)
}

func TestRoundTrip(t *testing.T) {
	proc := dispenseTest(t)
	ctx := t.Context()

	resp, err := proc.Process(ctx, ProcessRequest{Records: []Record{{ID: "a"}, {ID: "b"}}})
	if err != nil {
		t.Fatalf("Process: %v", err)
	}
	if resp.Processed != 2 {
		t.Fatalf("Processed = %d, want 2", resp.Processed)
	}

	// A plugin-returned error arrives as a gRPC status, not the sentinel, so
	// assert non-nil rather than errors.Is across the boundary.
	if _, err := proc.Process(ctx, ProcessRequest{}); err == nil {
		t.Fatal("Process(empty) = nil error, want a transport error")
	}
}

func ExampleProcessRequest_Validate() {
	err := ProcessRequest{}.Validate()
	fmt.Println(errors.Is(err, ErrEmptyBatch))
	// Output: true
}
```

## Review

The contract is correct when the three layers stay separate: the proto is the
wire schema, the bridge is the only place that mentions `*pb.*` types, and the
host and plugin program purely against `Processor`. Confirm it by reading the
bridge — if any domain code outside `shared/processor.go` imports the `proto`
package, the separation has leaked. Confirm the round trip with `TestRoundTrip`,
which serves and dispenses in-process; if it hangs, the plugin type is probably
missing the `NetRPCUnsupportedPlugin` embed and does not satisfy `plugin.Plugin`,
or `AllowedProtocols` is unset so the client will not negotiate gRPC.

The mistakes to avoid are the structural ones. Do not make `Process` per-record
"for convenience" — every call is serialized across a process boundary, and the
coarse batch interface is the whole point. Do not expect `errors.Is` to catch a
plugin's sentinel on the host side: the error becomes a gRPC status on the wire,
which is why `TestRoundTrip` asserts non-nil and `TestValidate` asserts the
sentinel only on a local value. And keep the magic cookie and `ProtocolVersion`
honest — the cookie makes the binary refuse to run standalone, and bumping the
version is how you invalidate an incompatible plugin at the handshake instead of
at runtime.
