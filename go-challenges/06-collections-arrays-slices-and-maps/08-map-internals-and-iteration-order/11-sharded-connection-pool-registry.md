# Exercise 11: Sharded Map Registry for a High-Fanout Connection Pool

**Nivel: Avanzado** — validacion normal (tabla de casos, incluye borde o concurrencia).

A sidecar proxy in front of a fleet of backends keeps one persistent
connection per upstream address, looked up and refreshed by thousands of
request-handling goroutines a second -- the exact shape of Envoy's
per-upstream connection pool. The obvious implementation, a single
`sync.RWMutex` wrapped around one `map[string]*Conn`, is correct and it is
also the whole system's bottleneck the moment traffic outgrows one core: every
writer across every backend serializes behind the same lock, no matter how
many CPUs are idle. The fix that shows up across proxies, caches, and
connection pools alike is a sharded map -- an array of `N` independently
locked shards, each holding a slice of the keyspace -- so unrelated backends
can be written concurrently.

Sharding only works if the function choosing a key's shard actually spreads
keys across all `N` of them. It is easy to write a selector that *looks*
data-dependent and is not: keying off the address string's length, or its
first byte, both compile, both "use" the key, and both collapse an entire
fleet of addresses that happen to share a length or a leading character onto
one or two shards. The sharded map then behaves exactly like the single
global lock it was built to replace, and nothing about the code makes that
obvious -- there is no error, no panic, just one hot shard absorbing all the
contention while N-1 sit idle. This module proves the difference by measuring
where keys land, not by timing anything.

The registry built here is generic over the value type, so the same package
serves a connection pool, a health-check registry, or any other per-address
state a proxy needs to shard.

This module is fully self-contained: its own `go mod init`, a reusable
package, and its tests. Nothing here imports another exercise.

### A sharded map is only as good as its shard selector

The concept sheet's fourth concurrency option, after a plain map, an
`RWMutex`+map, and `sync.Map`, is a sharded map: `N` independent
`RWMutex`+map pairs, and a function mapping each key to exactly one of them.
Reads and writes to different shards never contend, because they hold
different mutexes -- the same idea as striping a lock, applied to a map. The
entire benefit depends on one property: the selector must distribute keys
close to uniformly across `[0, N)`. A selector that ignores most of the key's
entropy still compiles and still "picks a shard", it just tends to pick the
same one:

```go
// looks data-dependent; is not, for a fleet whose addresses share a length.
func shardIndexNaive(key string, n int) int {
	return len(key) % n
}
```

A real fleet's addresses are rarely random-length strings -- they are
`backend-000.svc:8443` through `backend-999.svc:8443`, generated by the same
template, and `len(key)` is identical for every one of them. `len(key) % n`
is then the same integer for every key, every write lands on one shard, and
the other `n-1` shards sit empty while that one shard's mutex becomes exactly
the single choke point sharding exists to remove. The fix is to hash the
*entire* key -- every byte contributes to the result -- and reduce that hash
modulo `n`. FNV-1a, the algorithm `hash/fnv` implements, folds each input
byte into the running hash with a multiply and an xor, so two keys that
differ anywhere, including only in a suffix a length-based or first-byte
selector never looks at, land in unrelated buckets:

```go
func shardIndex(key string, n int) int {
	h := fnv.New32a()
	_, _ = h.Write([]byte(key))
	return int(h.Sum32() % uint32(n))
}
```

Beyond the selector, the rest of the type is the same `RWMutex`+map pattern
the concept sheet already established, replicated `N` times: `Get` takes the
owning shard's `RLock`, `Set` and `Delete` take its `Lock`, and `Len` sums
across every shard's own lock in turn rather than holding one lock for the
whole registry (which would reintroduce the very serialization sharding
avoids). `ShardSizes` exists for the same reason a real deployment would want
it: an operator watching a sharded registry needs a way to see a collapsed
shard before it becomes the bottleneck that pages someone.

Create `registry.go`:

```go
// Package shardedregistry implements a sharded map: an array of N
// independently locked map shards, keyed by a hash of the lookup key modulo
// N, so that concurrent access spreads across N locks instead of one.
//
// It exists for the connection-pool shape a sidecar proxy needs: one entry
// per backend address, looked up and refreshed by thousands of goroutines a
// second. A single sync.RWMutex around one map serializes every writer
// behind the same lock regardless of how many CPUs are available; splitting
// the map into shards lets unrelated keys be written concurrently, as long
// as the shard-selection function actually spreads keys across shards. See
// the package tests for a demonstration of what happens when it does not.
package shardedregistry

import (
	"errors"
	"fmt"
	"hash/fnv"
	"sync"
)

// ErrInvalidShardCount is returned by New when the requested shard count is
// not positive.
var ErrInvalidShardCount = errors.New("shardedregistry: shard count must be positive")

// shard is one lock-protected partition of the registry.
type shard[V any] struct {
	mu sync.RWMutex
	m  map[string]V
}

// ShardedRegistry maps string keys (backend addresses) to values of type V
// across a fixed number of independently locked shards.
//
// ShardedRegistry is safe for concurrent use by multiple goroutines: each
// shard carries its own sync.RWMutex, so callers touching different shards
// never block each other. The shard count is fixed at construction and
// never changes, so a key always maps to the same shard for the lifetime of
// the registry.
//
// Get and Set do not copy V beyond the assignment itself: if V is a pointer
// or contains one, the registry and the caller share the pointee, and the
// caller is responsible for synchronizing any further mutation of it. This
// mirrors the aliasing contract of a plain Go map.
type ShardedRegistry[V any] struct {
	shards []*shard[V]
}

// New returns a ShardedRegistry partitioned into the given number of
// shards. It returns ErrInvalidShardCount if shards is not positive.
func New[V any](shards int) (*ShardedRegistry[V], error) {
	if shards <= 0 {
		return nil, fmt.Errorf("%w: got %d", ErrInvalidShardCount, shards)
	}
	r := &ShardedRegistry[V]{shards: make([]*shard[V], shards)}
	for i := range r.shards {
		r.shards[i] = &shard[V]{m: make(map[string]V)}
	}
	return r, nil
}

// shardIndex hashes key with FNV-1a and reduces it modulo n. FNV-1a mixes
// every input byte into the final hash, so keys that differ in only one
// position (or share a length, or a common prefix) still land across the
// full range of shards rather than colliding on a subset of them.
func shardIndex(key string, n int) int {
	h := fnv.New32a()
	_, _ = h.Write([]byte(key))
	return int(h.Sum32() % uint32(n))
}

// shardFor returns the shard that owns addr.
func (r *ShardedRegistry[V]) shardFor(addr string) *shard[V] {
	return r.shards[shardIndex(addr, len(r.shards))]
}

// Get returns the value stored for addr and whether it was present.
func (r *ShardedRegistry[V]) Get(addr string) (V, bool) {
	s := r.shardFor(addr)
	s.mu.RLock()
	defer s.mu.RUnlock()
	v, ok := s.m[addr]
	return v, ok
}

// Set stores v for addr, replacing any existing value.
func (r *ShardedRegistry[V]) Set(addr string, v V) {
	s := r.shardFor(addr)
	s.mu.Lock()
	defer s.mu.Unlock()
	s.m[addr] = v
}

// Delete removes addr from the registry. Deleting an absent key is a no-op.
func (r *ShardedRegistry[V]) Delete(addr string) {
	s := r.shardFor(addr)
	s.mu.Lock()
	defer s.mu.Unlock()
	delete(s.m, addr)
}

// Len returns the total number of entries across all shards.
func (r *ShardedRegistry[V]) Len() int {
	total := 0
	for _, s := range r.shards {
		s.mu.RLock()
		total += len(s.m)
		s.mu.RUnlock()
	}
	return total
}

// ShardCount returns the number of shards the registry was constructed
// with.
func (r *ShardedRegistry[V]) ShardCount() int {
	return len(r.shards)
}

// ShardSizes returns the number of entries in each shard, indexed by shard
// number. It is an operational diagnostic: a production registry exposes
// this (or a metric derived from it) so an operator can see whether the
// shard-selection function is actually spreading load, rather than
// discovering a collapsed shard only when one lock becomes a bottleneck.
//
// The returned slice is freshly allocated on every call and does not alias
// any internal state.
func (r *ShardedRegistry[V]) ShardSizes() []int {
	sizes := make([]int, len(r.shards))
	for i, s := range r.shards {
		s.mu.RLock()
		sizes[i] = len(s.m)
		s.mu.RUnlock()
	}
	return sizes
}
```

### Using it

Construct one `ShardedRegistry[V]` at startup with the shard count your
deployment has tuned (a power of two close to `GOMAXPROCS` is a common
starting point), and share the single value across every goroutine that
touches the connection pool -- that is what the concurrency contract on the
type promises. `Get`, `Set`, and `Delete` take an address and route to the
owning shard internally; callers never see or choose a shard directly. `Len`
and `ShardSizes` are safe to call while other goroutines are writing, since
each reads one shard's lock at a time rather than requiring a registry-wide
pause.

The one contract worth internalizing before dropping this into a service: if
`V` is a pointer type (a `*Conn`, say), the registry only ever stores and
returns the pointer, never a copy of what it points to. Two goroutines
holding the same `*Conn` from two `Get` calls are sharing state exactly as
they would with a plain map, and must synchronize any mutation of the
pointee themselves -- the registry's own locking only protects the map
structure, not the values inside it.

`ExampleShardedRegistry` in the `_test.go` is the runnable demonstration of
the whole API; `go test` executes it and compares its output against the
`// Output:` comment, so the usage below cannot drift from the code:

```go
r, err := New[string](4)
if err != nil {
	panic(err)
}

r.Set("10.0.0.1:8443", "conn-a")
r.Set("10.0.0.2:8443", "conn-b")
r.Set("10.0.0.3:8443", "conn-c")

conn, ok := r.Get("10.0.0.2:8443")
fmt.Printf("10.0.0.2:8443 -> %q, found=%v\n", conn, ok)

r.Delete("10.0.0.1:8443")
fmt.Println("len:", r.Len())
```

### Tests

`TestSetGetDelete` runs a table over an ordinary address, an empty-string
key, and a long hostname, checking `Get` reports absence before `Set` and
presence after, with `Delete` returning it to absence.
`TestDeleteAbsentKeyIsNoOp` and `TestLenSumsAcrossShards` cover the
remaining edges: deleting a key that was never set must not panic, and
`Len` must equal the sum `ShardSizes` reports after a batch of inserts.
`TestSingleShardCollapsesEveryKey` pins the degenerate case `shards=1`,
where every key legitimately lands on the one shard that exists -- a
reminder that "one hot shard" is only a bug when `N > 1`.

`TestNaiveSelectorCollapsesFixedLengthKeys` is the heart of the module.
`shardIndexNaive` is unexported and unreachable from the package API; the
test builds 1000 addresses of the fixed-width form `backend-000.svc:8443`
through `backend-999.svc:8443`, feeds them through `shardIndexNaive`, and
asserts the naive selector's busiest shard holds *all* of them -- `len(key)`
is constant across the batch, so `len(key)%n` is one constant shard index for
every key. The same 1000 addresses through the real `ShardedRegistry` are
asserted to land with a busiest-shard share strictly below the naive one, and
below 0.5 outright: a documented property, not a hand-picked count, since the
exact split FNV-1a produces is not part of the contract.
`TestConcurrentAccessAcrossShards` drives many goroutines each owning a
disjoint slice of the keyspace through `Set`, `Get`, and `Delete`
concurrently, catching any lock ordering mistake under `-race`.

Create `registry_test.go`:

```go
package shardedregistry

import (
	"errors"
	"fmt"
	"sync"
	"testing"
)

// shardIndexNaive is the shard-selection function a first draft of this type
// tends to ship with: it "looks" like a hash because it depends on the key,
// but it only depends on the key's length. It is never exported and never
// reachable from ShardedRegistry; it exists so the tests can pin exactly
// what goes wrong when a sharded map's selector fails to mix the key.
func shardIndexNaive(key string, n int) int {
	return len(key) % n
}

func TestNewRejectsNonPositiveShardCount(t *testing.T) {
	t.Parallel()

	for _, n := range []int{0, -1, -5} {
		if _, err := New[string](n); !errors.Is(err, ErrInvalidShardCount) {
			t.Errorf("New(%d) error = %v, want ErrInvalidShardCount", n, err)
		}
	}
}

func TestSetGetDelete(t *testing.T) {
	t.Parallel()

	tests := []struct {
		name string
		addr string
	}{
		{name: "simple address", addr: "10.0.0.1:8443"},
		{name: "empty key", addr: ""},
		{name: "long hostname", addr: "backend-047.svc.cluster.local:8443"},
	}

	for _, tc := range tests {
		t.Run(tc.name, func(t *testing.T) {
			t.Parallel()

			r, err := New[string](8)
			if err != nil {
				t.Fatalf("New: %v", err)
			}
			if _, ok := r.Get(tc.addr); ok {
				t.Fatalf("Get(%q) before Set: ok = true, want false", tc.addr)
			}
			r.Set(tc.addr, "conn-1")
			got, ok := r.Get(tc.addr)
			if !ok || got != "conn-1" {
				t.Fatalf("Get(%q) = (%q, %v), want (conn-1, true)", tc.addr, got, ok)
			}
			r.Delete(tc.addr)
			if _, ok := r.Get(tc.addr); ok {
				t.Fatalf("Get(%q) after Delete: ok = true, want false", tc.addr)
			}
		})
	}
}

func TestDeleteAbsentKeyIsNoOp(t *testing.T) {
	t.Parallel()

	r, err := New[int](4)
	if err != nil {
		t.Fatalf("New: %v", err)
	}
	r.Delete("never-set") // must not panic
	if got := r.Len(); got != 0 {
		t.Fatalf("Len() = %d, want 0", got)
	}
}

func TestLenSumsAcrossShards(t *testing.T) {
	t.Parallel()

	r, err := New[int](4)
	if err != nil {
		t.Fatalf("New: %v", err)
	}
	if got := r.Len(); got != 0 {
		t.Fatalf("Len() on empty registry = %d, want 0", got)
	}

	for i := range 50 {
		r.Set(fmt.Sprintf("backend-%03d:8443", i), i)
	}
	if got := r.Len(); got != 50 {
		t.Fatalf("Len() = %d, want 50", got)
	}
	sum := 0
	for _, n := range r.ShardSizes() {
		sum += n
	}
	if sum != 50 {
		t.Fatalf("sum of ShardSizes() = %d, want 50", sum)
	}
}

func TestSingleShardCollapsesEveryKey(t *testing.T) {
	t.Parallel()

	r, err := New[int](1)
	if err != nil {
		t.Fatalf("New: %v", err)
	}
	for i := range 20 {
		r.Set(fmt.Sprintf("backend-%d", i), i)
	}
	sizes := r.ShardSizes()
	if len(sizes) != 1 || sizes[0] != 20 {
		t.Fatalf("ShardSizes() = %v, want a single shard holding all 20 entries", sizes)
	}
}

// maxShare returns the largest shard's fraction of the total entries.
func maxShare(sizes []int) float64 {
	total, max := 0, 0
	for _, n := range sizes {
		total += n
		if n > max {
			max = n
		}
	}
	if total == 0 {
		return 0
	}
	return float64(max) / float64(total)
}

// TestNaiveSelectorCollapsesFixedLengthKeys is the heart of the module. It
// builds 1000 backend addresses that are all the same length, exactly the
// shape "backend-000.svc:8443" through "backend-999.svc:8443" produces, and
// shows that shardIndexNaive sends every single one of them to the same
// shard: len(key) is constant, so len(key)%n is the same integer for every
// key. shardIndex, hashing the full key with FNV-1a, spreads the same 1000
// keys across every shard instead.
func TestNaiveSelectorCollapsesFixedLengthKeys(t *testing.T) {
	t.Parallel()

	const shards = 8
	addrs := make([]string, 1000)
	for i := range addrs {
		addrs[i] = fmt.Sprintf("backend-%03d.svc:8443", i)
	}

	naiveSizes := make([]int, shards)
	for _, a := range addrs {
		naiveSizes[shardIndexNaive(a, shards)]++
	}
	if got := maxShare(naiveSizes); got != 1.0 {
		t.Fatalf("naive max shard share = %v, want 1.0 (every key on one shard): %v", got, naiveSizes)
	}

	r, err := New[struct{}](shards)
	if err != nil {
		t.Fatalf("New: %v", err)
	}
	for _, a := range addrs {
		r.Set(a, struct{}{})
	}
	realSizes := r.ShardSizes()
	realShare := maxShare(realSizes)
	if !(realShare < maxShare(naiveSizes)) {
		t.Fatalf("real max shard share = %v, want it below the naive share %v: %v", realShare, maxShare(naiveSizes), realSizes)
	}
	if realShare > 0.5 {
		t.Fatalf("real max shard share = %v, want under 0.5 across %d shards: %v", realShare, shards, realSizes)
	}
}

func TestConcurrentAccessAcrossShards(t *testing.T) {
	t.Parallel()

	r, err := New[int](16)
	if err != nil {
		t.Fatalf("New: %v", err)
	}

	const goroutines, perGoroutine = 32, 200
	var wg sync.WaitGroup
	for g := range goroutines {
		wg.Add(1)
		go func(g int) {
			defer wg.Done()
			for i := range perGoroutine {
				addr := fmt.Sprintf("g%d-backend-%d", g, i)
				r.Set(addr, g*perGoroutine+i)
				if v, ok := r.Get(addr); !ok || v != g*perGoroutine+i {
					t.Errorf("Get(%q) = (%d, %v), want (%d, true)", addr, v, ok, g*perGoroutine+i)
				}
				r.Delete(addr)
			}
		}(g)
	}
	wg.Wait()

	if got := r.Len(); got != 0 {
		t.Fatalf("Len() after all deletes = %d, want 0", got)
	}
}

// ExampleShardedRegistry demonstrates the whole API: construct with a shard
// count, store and look up connection IDs by backend address, and inspect
// the shard distribution.
func ExampleShardedRegistry() {
	r, err := New[string](4)
	if err != nil {
		panic(err)
	}

	r.Set("10.0.0.1:8443", "conn-a")
	r.Set("10.0.0.2:8443", "conn-b")
	r.Set("10.0.0.3:8443", "conn-c")

	conn, ok := r.Get("10.0.0.2:8443")
	fmt.Printf("10.0.0.2:8443 -> %q, found=%v\n", conn, ok)

	r.Delete("10.0.0.1:8443")
	fmt.Println("len:", r.Len())

	_, ok = r.Get("10.0.0.1:8443")
	fmt.Println("10.0.0.1:8443 found:", ok)

	// Output:
	// 10.0.0.2:8443 -> "conn-b", found=true
	// len: 2
	// 10.0.0.1:8443 found: false
}
```

## Review

The registry is correct when concurrent writers to different backends never
block each other, and that depends entirely on `shardIndex` mixing the whole
key rather than a fragment of it. `New` rejects a non-positive shard count
with `ErrInvalidShardCount`, checkable with `errors.Is`. `Get`, `Set`, and
`Delete` each touch exactly one shard's `RWMutex`; `Len` and `ShardSizes` sum
across every shard's own lock rather than a registry-wide one. The trap this
module isolates is the shard selector that compiles, "uses" the key, and
still collapses a whole fleet onto one shard because it only reads the key's
length or its first byte -- `shardIndexNaive` pins that failure numerically
against 1000 fixed-length addresses, and the real `shardIndex`, hashing every
byte with FNV-1a, is asserted to spread the same addresses with a strictly
lower busiest-shard share. `ShardedRegistry` is safe for concurrent use, and `ExampleShardedRegistry` is
the executable documentation `go test` verifies. Run
`go test -count=1 -race ./...`.
