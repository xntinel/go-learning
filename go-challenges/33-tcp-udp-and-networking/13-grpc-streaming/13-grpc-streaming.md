# 13. gRPC Streaming

gRPC supports four communication patterns over a single HTTP/2 connection: unary (one request, one response), server streaming (one request, many responses), client streaming (many requests, one response), and bidirectional streaming (many requests, many responses). The hard parts are managing stream lifetimes correctly — when to close, how to honour context cancellation mid-stream, and how to test without opening a network port.

This lesson builds a `MetricsService` that exercises all four patterns and tests them in-process with `bufconn`.

```text
grpcstreaming/
  go.mod
  proto/metrics.proto
  pb/                    # generated by protoc (Exercise 1)
  service.go             # package grpcstreaming
  service_test.go        # package grpcstreaming
  cmd/demo/main.go       # package main
```

## Concepts

### The Four gRPC Patterns

A gRPC service method is either unary or streaming, and streaming can apply to the request side, the response side, or both:

| Pattern | Proto syntax | Typical use |
| --- | --- | --- |
| Unary | `rpc F(Req) returns (Resp)` | single round trip |
| Server streaming | `rpc F(Req) returns (stream Resp)` | feed, subscription |
| Client streaming | `rpc F(stream Req) returns (Resp)` | bulk upload |
| Bidirectional | `rpc F(stream Req) returns (stream Resp)` | live aggregation, chat |

All four share the same gRPC server registration and dial paths. The `stream` keyword in the proto is the only syntax difference; the generated Go interfaces differ per pattern.

### Stream Lifetimes and EOF

Each streaming RPC has an explicit close signal:

- **Server streaming**: the server handler returns `nil`; the client sees `io.EOF` on the next `Recv`.
- **Client streaming**: the client calls `CloseAndRecv` (which calls `CloseSend` internally); the server sees `io.EOF` on the next `Recv` and must then call `SendAndClose` to deliver the final response.
- **Bidirectional**: each side closes independently. The client calls `CloseSend` when it has no more messages; the server drains remaining messages (until `io.EOF`), returns `nil`, and the client's next `Recv` returns `io.EOF`.

Forgetting to call `CloseSend` or `CloseAndRecv` in client-streaming code is the most common deadlock: the server's receive loop never sees `io.EOF` and the RPC hangs indefinitely.

### Context Cancellation in Long-Running Streams

Every stream carries a context accessible via `stream.Context()`. When the client cancels or times out, that context is cancelled server-side. A server-streaming loop that ignores this will keep computing and sending long after the client has gone away:

```go
select {
case <-stream.Context().Done():
	return stream.Context().Err()
case t := <-ticker.C:
	_ = stream.Send(msg)
}
```

The `select` exits immediately on cancellation rather than blocking until the next tick.

### Embedding UnimplementedMetricsServiceServer

`protoc-gen-go-grpc` generates an `UnimplementedMetricsServiceServer` that returns `codes.Unimplemented` for every method. Embedding it in the concrete server means:

- The struct satisfies the full interface even before all methods are implemented.
- Adding a new RPC to the proto later does not immediately break existing server code; it keeps returning `Unimplemented` for the new method until an implementation is written.

### In-Process Testing With bufconn

`google.golang.org/grpc/test/bufconn` provides an in-memory `net.Listener` backed by a byte buffer. The test routes the client's dial through it via `grpc.WithContextDialer`:

```go
lis := bufconn.Listen(1 << 20)
go srv.Serve(lis)

conn, err := grpc.NewClient(
	"passthrough:///bufnet",
	grpc.WithContextDialer(func(ctx context.Context, _ string) (net.Conn, error) {
		return lis.DialContext(ctx)
	}),
	grpc.WithTransportCredentials(insecure.NewCredentials()),
)
```

No real network port is opened. The test is hermetic, deterministic, and does not require a listening address to be free.

### Exercise 1: Proto Definition and Code Generation

Create `proto/metrics.proto`:

```protobuf
syntax = "proto3";

package metrics;

option go_package = "example.com/grpcstreaming/pb;pb";

service MetricsService {
  // Unary: return the stored summary for a named metric.
  rpc GetSummary(MetricRequest) returns (Summary);

  // Server streaming: push MetricUpdate messages on each tick for Count iterations.
  rpc Subscribe(SubscribeRequest) returns (stream MetricUpdate);

  // Client streaming: accept DataPoints and return an aggregate Summary.
  rpc Upload(stream DataPoint) returns (Summary);

  // Bidirectional: receive DataPoints and reply with a running Summary after each.
  rpc Monitor(stream DataPoint) returns (stream Summary);
}

message MetricRequest {
  string name = 1;
}

message SubscribeRequest {
  string name  = 1;
  int32  count = 2; // number of updates to deliver
}

message DataPoint {
  string metric = 1;
  double value  = 2;
}

message MetricUpdate {
  string name  = 1;
  double value = 2;
  int64  ts    = 3; // unix nanoseconds
}

message Summary {
  string metric = 1;
  int64  count  = 2;
  double sum    = 3;
  double mean   = 4;
  double min    = 5;
  double max    = 6;
}
```

### Exercise 2: Server Implementation

Create `service.go`:

```go
package grpcstreaming

import (
	"context"
	"io"
	"math"
	"sync"
	"time"

	"google.golang.org/grpc/codes"
	"google.golang.org/grpc/status"

	pb "example.com/grpcstreaming/pb"
)

// accumulator holds running statistics for one metric.
type accumulator struct {
	count int64
	sum   float64
	min   float64
	max   float64
}

func newAccumulator() *accumulator {
	return &accumulator{min: math.MaxFloat64, max: -math.MaxFloat64}
}

func (a *accumulator) add(v float64) {
	a.count++
	a.sum += v
	if v < a.min {
		a.min = v
	}
	if v > a.max {
		a.max = v
	}
}

func (a *accumulator) mean() float64 {
	if a.count == 0 {
		return 0
	}
	return a.sum / float64(a.count)
}

func (a *accumulator) toProto(name string) *pb.Summary {
	return &pb.Summary{
		Metric: name,
		Count:  a.count,
		Sum:    a.sum,
		Mean:   a.mean(),
		Min:    a.min,
		Max:    a.max,
	}
}

// metricsServer implements pb.MetricsServiceServer.
type metricsServer struct {
	pb.UnimplementedMetricsServiceServer

	mu           sync.RWMutex
	stored       map[string]*accumulator // metrics persisted by Upload
	tickInterval time.Duration           // Subscribe tick rate; 0 is treated as 500 ms
}

// NewMetricsServer returns a MetricsServiceServer with a 500 ms Subscribe interval.
func NewMetricsServer() pb.MetricsServiceServer {
	return newMetricsServer(500 * time.Millisecond)
}

// newMetricsServer lets tests inject a faster tick interval.
func newMetricsServer(tickInterval time.Duration) *metricsServer {
	if tickInterval <= 0 {
		tickInterval = 500 * time.Millisecond
	}
	return &metricsServer{
		stored:       make(map[string]*accumulator),
		tickInterval: tickInterval,
	}
}

// GetSummary is the unary RPC. It returns the stored Summary for a named
// metric or codes.NotFound if the metric has not been uploaded yet.
func (s *metricsServer) GetSummary(_ context.Context, req *pb.MetricRequest) (*pb.Summary, error) {
	if req.GetName() == "" {
		return nil, status.Error(codes.InvalidArgument, "metric name is required")
	}
	s.mu.RLock()
	acc, ok := s.stored[req.GetName()]
	s.mu.RUnlock()
	if !ok {
		return nil, status.Errorf(codes.NotFound, "metric %q not found", req.GetName())
	}
	return acc.toProto(req.GetName()), nil
}

// Subscribe is the server-streaming RPC. It sends req.Count MetricUpdate
// messages, one per tick, honouring context cancellation between sends.
func (s *metricsServer) Subscribe(req *pb.SubscribeRequest, stream pb.MetricsService_SubscribeServer) error {
	if req.GetName() == "" {
		return status.Error(codes.InvalidArgument, "metric name is required")
	}
	n := int(req.GetCount())
	if n <= 0 {
		n = 1
	}

	ticker := time.NewTicker(s.tickInterval)
	defer ticker.Stop()

	for i := 0; i < n; i++ {
		select {
		case <-stream.Context().Done():
			return stream.Context().Err()
		case t := <-ticker.C:
			if err := stream.Send(&pb.MetricUpdate{
				Name:  req.GetName(),
				Value: float64(i + 1), // deterministic sequence; replace with real data in production
				Ts:    t.UnixNano(),
			}); err != nil {
				return err
			}
		}
	}
	return nil
}

// Upload is the client-streaming RPC. The client sends DataPoints; the server
// accumulates them and calls SendAndClose with the aggregate Summary when the
// client calls CloseAndRecv.
func (s *metricsServer) Upload(stream pb.MetricsService_UploadServer) error {
	acc := newAccumulator()
	var name string

	for {
		pt, err := stream.Recv()
		if err == io.EOF {
			if acc.count == 0 {
				return status.Error(codes.InvalidArgument, "no data points received")
			}
			// Persist so GetSummary can find this metric later.
			s.mu.Lock()
			s.stored[name] = acc
			s.mu.Unlock()
			return stream.SendAndClose(acc.toProto(name))
		}
		if err != nil {
			return err
		}
		if pt.GetMetric() == "" {
			return status.Error(codes.InvalidArgument, "data point metric name is required")
		}
		if name == "" {
			name = pt.GetMetric()
		}
		acc.add(pt.GetValue())
	}
}

// Monitor is the bidirectional-streaming RPC. After each received DataPoint
// the server replies with the updated running Summary for that metric.
// The server returns nil when the client sends EOF (via CloseSend), which
// causes the client's next Recv to return io.EOF.
func (s *metricsServer) Monitor(stream pb.MetricsService_MonitorServer) error {
	accs := make(map[string]*accumulator) // stream-local state; not shared across RPCs

	for {
		pt, err := stream.Recv()
		if err == io.EOF {
			return nil
		}
		if err != nil {
			return err
		}
		name := pt.GetMetric()
		if name == "" {
			return status.Error(codes.InvalidArgument, "data point metric name is required")
		}
		if accs[name] == nil {
			accs[name] = newAccumulator()
		}
		accs[name].add(pt.GetValue())
		if err := stream.Send(accs[name].toProto(name)); err != nil {
			return err
		}
	}
}
```

Key points:
- `pb.UnimplementedMetricsServiceServer` is embedded so the struct satisfies the full interface even if a new RPC is added to the proto later.
- `status.Error` and `status.Errorf` with `codes` constants are the correct way to return gRPC errors. Callers inspect the code with `status.Code(err)`, not by matching the error string.
- `Subscribe` selects between `stream.Context().Done()` and `ticker.C`; the server stops immediately on client cancellation.
- `Upload` validates the empty-stream case explicitly rather than returning a zero-value Summary with all fields at their zero values.
- `Monitor` keeps per-stream state in `accs`; the map is not shared across concurrent RPCs.

```go
// newTestClient starts an in-process gRPC server with a fast tick interval
// and returns a client connected to it over bufconn. The server and
// connection are registered with t.Cleanup and torn down automatically.
func newTestClient(t *testing.T) pb.MetricsServiceClient {
	t.Helper()
	lis := bufconn.Listen(1 << 20)
	srv := grpc.NewServer()
	pb.RegisterMetricsServiceServer(srv, newMetricsServer(5*time.Millisecond))
	go func() { _ = srv.Serve(lis) }()
	t.Cleanup(srv.GracefulStop)

	conn, err := grpc.NewClient(
		"passthrough:///bufnet",

	if err != nil {
		t.Fatalf("grpc.NewClient: %v", err)

func TestSubscribe_CancelStopsStream(t *testing.T) {
	t.Parallel()
	client := newTestClient(t)

	ctx, cancel := context.WithCancel(context.Background())
	stream, err := client.Subscribe(ctx, &pb.SubscribeRequest{Name: "cpu", Count: 100})
	if err != nil {
		t.Fatalf("Subscribe: %v", err)
	}

	// Receive at least one update then cancel the context.
	if _, err := stream.Recv(); err != nil {
		t.Fatalf("first Recv: %v", err)
	}
	cancel()

	// Drain until the first non-nil error; it must be Canceled.
	for {
		_, err := stream.Recv()
		if err == nil {
			continue
		}
		if got := status.Code(err); got != codes.Canceled {
			t.Fatalf("code = %v, want Canceled", got)
		}
		break
	}
}

func TestMonitor_RunningAggregate(t *testing.T) {
	t.Parallel()
	client := newTestClient(t)

	stream, err := client.Monitor(context.Background())
	if err != nil {
		t.Fatalf("Monitor: %v", err)
	}

	steps := []struct {
		send      float64
		wantCount int64
		wantSum   float64
		wantMean  float64
	}{
		{10, 1, 10, 10},
		{20, 2, 30, 15},
		{30, 3, 60, 20},
	}

	for _, tc := range steps {
		if err := stream.Send(&pb.DataPoint{Metric: "mem", Value: tc.send}); err != nil {
			t.Fatalf("Send(%v): %v", tc.send, err)
		}
		s, err := stream.Recv()
		if err != nil {
			t.Fatalf("Recv after %v: %v", tc.send, err)
		}
		if s.GetCount() != tc.wantCount || s.GetSum() != tc.wantSum || s.GetMean() != tc.wantMean {
			t.Errorf("after %.0f: got {count=%d sum=%.0f mean=%.0f}, want {count=%d sum=%.0f mean=%.0f}",
				tc.send,
				s.GetCount(), s.GetSum(), s.GetMean(),
				tc.wantCount, tc.wantSum, tc.wantMean)
		}
	}

	if err := stream.CloseSend(); err != nil {
		t.Fatalf("CloseSend: %v", err)
	}
	// After CloseSend, the server's Recv returns io.EOF, the server returns nil,
	// and the client's next Recv must return io.EOF.
	if _, err := stream.Recv(); err != io.EOF {
		t.Fatalf("after CloseSend, Recv = %v, want io.EOF", err)
	}
}
```

### Exercise 4: Demo Program

Create `cmd/demo/main.go`:

```go
package main

import (
	"context"
	"fmt"
	"io"
	"log"
	"net"
	"time"

	"google.golang.org/grpc"
	"google.golang.org/grpc/credentials/insecure"

	grpcstreaming "example.com/grpcstreaming"
	pb "example.com/grpcstreaming/pb"
)

func main() {
	// Start an in-process server on a random loopback port.
	lis, err := net.Listen("tcp", "127.0.0.1:0")
	if err != nil {
		log.Fatal(err)
	}
	srv := grpc.NewServer()
	pb.RegisterMetricsServiceServer(srv, grpcstreaming.NewMetricsServer())
	go func() { _ = srv.Serve(lis) }()
	defer srv.GracefulStop()

	conn, err := grpc.NewClient(
		lis.Addr().String(),
		grpc.WithTransportCredentials(insecure.NewCredentials()),
	)
	if err != nil {
		log.Fatal(err)
	}
	defer conn.Close()

	client := pb.NewMetricsServiceClient(conn)
	ctx := context.Background()

	// 1. Client streaming: upload five data points.
	fmt.Println("=== Upload (client streaming) ===")
	uStream, err := client.Upload(ctx)
	if err != nil {
		log.Fatal(err)
	}
	for _, v := range []float64{10, 20, 30, 40, 50} {
		if err := uStream.Send(&pb.DataPoint{Metric: "cpu", Value: v}); err != nil {
			log.Fatal(err)
		}
	}
	summary, err := uStream.CloseAndRecv()
	if err != nil {
		log.Fatal(err)
	}
	fmt.Printf("  count=%d sum=%.0f mean=%.0f min=%.0f max=%.0f\n",
		summary.GetCount(), summary.GetSum(), summary.GetMean(),
		summary.GetMin(), summary.GetMax())

	// 2. Unary: retrieve the stored summary.
	fmt.Println("=== GetSummary (unary) ===")
	s, err := client.GetSummary(ctx, &pb.MetricRequest{Name: "cpu"})
	if err != nil {
		log.Fatal(err)
	}
	fmt.Printf("  metric=%s count=%d mean=%.0f\n", s.GetMetric(), s.GetCount(), s.GetMean())

	// 3. Server streaming: subscribe to 4 updates (500 ms each; ~2 s total).
	fmt.Println("=== Subscribe (server streaming, 4 updates) ===")
	subCtx, subCancel := context.WithTimeout(ctx, 10*time.Second)
	defer subCancel()
	subStream, err := client.Subscribe(subCtx, &pb.SubscribeRequest{Name: "cpu", Count: 4})
	if err != nil {
		log.Fatal(err)
	}
	for {
		u, err := subStream.Recv()
		if err == io.EOF {
			break
		}
		if err != nil {
			log.Fatal(err)
		}
		fmt.Printf("  update name=%s value=%.0f\n", u.GetName(), u.GetValue())
	}

	// 4. Bidirectional streaming: send three points, print running summary after each.
	fmt.Println("=== Monitor (bidirectional streaming) ===")
	monStream, err := client.Monitor(ctx)
	if err != nil {
		log.Fatal(err)
	}
	for _, v := range []float64{5, 15, 25} {
		if err := monStream.Send(&pb.DataPoint{Metric: "mem", Value: v}); err != nil {
			log.Fatal(err)
		}
		s, err := monStream.Recv()
		if err != nil {
			log.Fatal(err)
		}
		fmt.Printf("  after %.0f: count=%d mean=%.0f\n", v, s.GetCount(), s.GetMean())
	}
	if err := monStream.CloseSend(); err != nil {
		log.Fatal(err)
	}
	for {
		_, err := monStream.Recv()
		if err == io.EOF {
			break
		}
		if err != nil {
			log.Fatal(err)
		}
	}
}
```


## Common Mistakes

### Not Calling CloseSend or CloseAndRecv

Wrong: the client sends all its messages and then returns without calling `CloseSend` or `CloseAndRecv`.

What happens: the server's `Recv` loop never receives `io.EOF`. The server blocks forever and the RPC never completes. This also leaks the gRPC stream.

Fix: always call `CloseAndRecv()` (client-streaming) or `CloseSend()` followed by draining `Recv` (bidirectional) to signal that no more messages will be sent. The test `TestUpload_AggregatesPoints` shows the correct pattern.

### Returning io.EOF from a Server Handler

Wrong:

```go
func (s *metricsServer) Upload(stream pb.MetricsService_UploadServer) error {
	acc := newAccumulator()
	var name string
	for {
		pt, realErr := stream.Recv()
		if realErr == io.EOF {
			return stream.SendAndClose(acc.toProto(name))
		}
		if realErr != nil {
			return io.EOF // caller intended "done", but this swallows the real error
		}
		name = pt.GetMetric()
		acc.add(pt.GetValue())
	}
}
```

What happens: the gRPC runtime treats `io.EOF` returned from a server handler as a clean stream close, not as an error. The status code and message carried by the real error are discarded before they reach the wire. The client's `CloseAndRecv` (or the final `Recv` in a bidirectional stream) returns `io.EOF` — with no indication that anything went wrong — instead of the gRPC status that describes the actual failure.

Fix: return `nil` from the server handler to signal clean close; return a real error (via `status.Error`) for failures. Never return `io.EOF` from a server handler directly. The pattern in `service.go` follows this rule throughout.

### Ignoring Context Cancellation in Server Streaming

Wrong: a server-streaming handler loops over a ticker without a `select`:

```go
for i := 0; i < n; i++ {
	<-ticker.C
	_ = stream.Send(msg) // blocks for one tick even after the client cancelled
}
```

What happens: after the client cancels its context, `stream.Send` eventually returns an error (because the transport is torn down), but the server waits a full tick before discovering this. For a 500 ms ticker, the server wastes half a second per iteration.

Fix: use a `select` with `stream.Context().Done()`, as shown in `service.go`'s `Subscribe`. The server exits within microseconds of the client cancellation.

### Sharing Accumulator State Across Concurrent RPCs

Wrong: storing per-RPC state in a field on the server struct without a mutex:

```go
func (s *metricsServer) Monitor(stream ...) error {
	// s.runningAcc is shared — all concurrent Monitor RPCs corrupt it
	s.runningAcc.add(pt.GetValue())
	...
}
```

What happens: concurrent Monitor calls interleave their data points into the same accumulator. The summaries returned to each client are wrong.

Fix: allocate accumulator state on the stack inside the handler, as `service.go` does with `accs := make(map[string]*accumulator)`. Each RPC invocation gets its own map; no lock is needed.

## Summary

- gRPC defines four patterns in the proto file using the `stream` keyword; the Go interfaces differ per pattern but the server registration and dial paths are the same.
- Client-streaming RPCs must call `CloseAndRecv` or `CloseSend`; without it the server blocks on `Recv` indefinitely.
- Server-streaming handlers must select on `stream.Context().Done()` between sends to honour client cancellation without a full-tick delay.
- The bidirectional pattern closes independently on each side: client calls `CloseSend`, server sees `io.EOF` and returns `nil`, client sees `io.EOF` on its next `Recv`.
- `status.Error` and `status.Errorf` with typed `codes` constants are the correct way to return gRPC errors; callers use `status.Code(err)` to inspect them, not string matching.
- `bufconn` provides a hermetic in-process test environment: no network port, no flakiness, fast.
