# Exercise 7: Reproducible Dev Tooling with the go.mod tool Directive

Every engineer and CI runner on a team must run byte-identical versions of code
generators and linters, or generated code churns and reviews fight noise. Go 1.24
made that a first-class feature: the `tool` directive in `go.mod`. This exercise
replaces the legacy `tools.go` hack with `go get -tool` and generates a `String`
method with the pinned `stringer`.

This module is fully self-contained: its own `go mod init`, all code inline, its
own tests. Nothing here imports another exercise.

## What you'll build

```text
jobstate/                   independent module: example.com/jobstate
  go.mod                    (a tool directive + pinned require, after go get -tool)
  state.go                  type State + consts + //go:generate directive
  state_string.go           generated by stringer (checked in)
  cmd/
    demo/
      main.go               prints each State
  state_test.go             asserts String() for known and unknown values
```

- Files: `state.go`, `state_string.go`, `cmd/demo/main.go`, `state_test.go`.
- Implement: a `State int` enum with a `//go:generate go tool stringer` directive; commit the generated `state_string.go`.
- Test: assert `String()` for each known state and the out-of-range fallback.
- Verify: `go get -tool`, `go generate ./...`, `go tool` lists the tool; `go build`, `go test -race`.

### The tool directive versus tools.go

Before Go 1.24, the way to pin a code-generation tool to a module was a `tools.go`
file: a build-tagged file with blank imports of the tool packages, kept only so
`go mod tidy` would record them in `go.mod`. It was a workaround, and `go build`
had to be told to ignore it. Go 1.24 replaced it with a real directive. Running:

```bash
go get -tool golang.org/x/tools/cmd/stringer
```

adds a `tool` line to `go.mod` and a pinned `require` for the tool's module:

```text
tool golang.org/x/tools/cmd/stringer

require (
	golang.org/x/mod v0.37.0 // indirect
	golang.org/x/sync v0.21.0 // indirect
	golang.org/x/tools v0.47.0 // indirect
)
```

Now the tool is version-pinned in `go.mod` exactly like any dependency, so every
checkout and every CI runner builds and runs the *same* `stringer`. List the
module's tools and run one:

```bash
go tool
go tool stringer -help
```

`go tool` prints the tools declared in the current module (plus the built-in
toolchain commands); the module-local line is:

```text
golang.org/x/tools/cmd/stringer
```

`go tool` only sees tools in the *current* module's `go.mod`, which is precisely
the per-module reproducibility you want. Upgrade a tool later with `go get tool`
or `go get -tool pkg@version`.

### The enum and the generate directive

`stringer` turns an integer enum into a `String() string` method so the constants
print as their names. You write the type and a `//go:generate` directive; `go
generate` runs it.

Create `state.go`:

```go
package jobstate

//go:generate go tool stringer -type=State

// State is the lifecycle state of a background job.
type State int

const (
	StatePending State = iota
	StateRunning
	StateSucceeded
	StateFailed
)
```

Generate the `String` method with the pinned tool:

```bash
go generate ./...
```

`go generate` scans for `//go:generate` directives and runs each command; here it
invokes the module-pinned `stringer`, which writes `state_string.go`. That file is
checked in (generated code is committed so consumers do not need the tool to
build). The real output looks exactly like this â€” do not hand-edit it:

Create `state_string.go`:

```go
// Code generated by "stringer -type=State"; DO NOT EDIT.

package jobstate

import "strconv"

func _() {
	// An "invalid array index" compiler error signifies that the constant values have changed.
	// Re-run the stringer command to generate them again.
	var x [1]struct{}
	_ = x[StatePending-0]
	_ = x[StateRunning-1]
	_ = x[StateSucceeded-2]
	_ = x[StateFailed-3]
}

const _State_name = "StatePendingStateRunningStateSucceededStateFailed"

var _State_index = [...]uint8{0, 12, 24, 38, 49}

func (i State) String() string {
	idx := int(i) - 0
	if i < 0 || idx >= len(_State_index)-1 {
		return "State(" + strconv.FormatInt(int64(i), 10) + ")"
	}
	return _State_name[_State_index[idx]:_State_index[idx+1]]
}
```

The `func _()` block is a compile-time guard: if someone reorders the constants so
the values drift, the array indexing fails to compile and forces a regenerate. The
`String` method indexes a packed name string, and an out-of-range `State` falls
back to `State(<n>)`.

### The demo

Create `cmd/demo/main.go`:

```go
package main

import (
	"fmt"

	"example.com/jobstate"
)

func main() {
	for s := jobstate.StatePending; s <= jobstate.StateFailed; s++ {
		fmt.Printf("%d = %s\n", int(s), s)
	}
	fmt.Println("unknown:", jobstate.State(9))
}
```

Run it:

```bash
go run ./cmd/demo
```

Expected output:

```
0 = StatePending
1 = StateRunning
2 = StateSucceeded
3 = StateFailed
unknown: State(9)
```

### The test

The test asserts the generated `String()` for each known state and for an
out-of-range value, so a stale `state_string.go` (constants changed but not
regenerated) is caught by CI, not by a surprised reader.

Create `state_test.go`:

```go
package jobstate

import "testing"

func TestStateString(t *testing.T) {
	t.Parallel()

	tests := []struct {
		state State
		want  string
	}{
		{state: StatePending, want: "StatePending"},
		{state: StateRunning, want: "StateRunning"},
		{state: StateSucceeded, want: "StateSucceeded"},
		{state: StateFailed, want: "StateFailed"},
		{state: State(9), want: "State(9)"},
	}

	for _, tc := range tests {
		t.Run(tc.want, func(t *testing.T) {
			t.Parallel()

			if got := tc.state.String(); got != tc.want {
				t.Fatalf("State(%d).String() = %q, want %q", int(tc.state), got, tc.want)
			}
		})
	}
}
```

Run the gate:

```bash
gofmt -l .
go vet ./...
go build ./...
go test -count=1 -race ./...
```

## Review

The tooling is reproducible when `go.mod` carries a `tool` directive plus a pinned
`require`, `go tool` lists the tool, and the checked-in `state_string.go` matches
what a fresh `go generate` would produce. The mistakes to avoid: recreating the
removed `tools.go` blank-import pattern on Go 1.24+ (use `go get -tool`); expecting
`go tool` to see a tool that is not declared in the current module's `go.mod` (it
only sees this module's tools); and hand-editing generated code â€” change the enum
and regenerate instead, which the `func _()` guard actively enforces by failing to
compile when the constants drift. Commit the generated file so consumers can build
without the tool, but keep the tool pinned so anyone who *does* regenerate produces
the identical bytes.
