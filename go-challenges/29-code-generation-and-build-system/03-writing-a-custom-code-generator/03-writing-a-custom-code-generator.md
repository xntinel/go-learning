# 3. Writing A Custom Code Generator

While `stringer` handles a specific pattern, production Go projects often need generators tailored to their domain: marshaling enums to JSON, building type-safe API clients, generating validators, or emitting mock implementations. Writing your own generator means you control the output completely and can encode invariants that no general-purpose tool knows about. The hard part is not the generation itself but keeping the generator, the pre-generated files, and the consuming package in sync across the codebase.

```text
enumjson/
  go.mod
  color.go
  color_json_gen.go
  enumjson_test.go
  cmd/enumgen/main.go
```

The `enumjson` package exposes a `Color` enum. The generator (`cmd/enumgen/main.go`) writes `color_json_gen.go` with `MarshalJSON` and `UnmarshalJSON` methods. The pre-generated file is committed so the package compiles and tests pass without running the generator.

## Concepts

### Generator Programs Are Regular Go Programs

A generator is a normal Go program invoked by `go generate`. There is nothing special about it. The `//go:build ignore` constraint excludes it from `go build` and `go test` so it does not pollute the package it generates into. You run it with `go run cmd/enumgen/main.go -type=Color -values=Red,Green,Blue,Yellow` or let `go generate` do it. The generator reads environment variables, writes files, and exits. That is the entire contract.

### text/template For Code Emission

The standard approach is:

1. Define a `text/template` string that looks like Go source with `{{ .Field }}` placeholders.
2. Execute the template against a data struct.
3. Pass the result through `go/format.Source`.

`go/format` is not optional. It does two things: it verifies that the output is syntactically valid Go (catching template mistakes early), and it normalizes indentation so the generated file is `gofmt`-clean. If `format.Source` returns an error, the generator must print the unformatted output and exit non-zero so the mistake is visible.

### The DO NOT EDIT Marker

The convention is `// Code generated by <tool>; DO NOT EDIT.` as the first line of the file. Tools like `gopls`, `goimports`, and most CI linters use this marker to skip the file. The Go specification does not mandate this text, but the go toolchain does recognize it: `go generate -v` prints it, and `go fix` skips files with it. The marker must appear within the first line of the file — not after a package declaration.

### GOPACKAGE Environment Variable

`go generate` sets `GOPACKAGE` to the package name of the file that contains the `//go:generate` directive. A generator reads `os.Getenv("GOPACKAGE")` to know which package name to emit. This avoids hardcoding the package name in the generator or the directive.

### Pre-Generated Files In Version Control

Generated files are typically committed. This allows consumers to `go get` the package without installing or running any generator. The generator is an implementation detail of the maintainer workflow; the generated file is part of the public API surface. CI may enforce that generated files are up to date by running `go generate ./...` and checking for a clean diff.

### Exercise 1: Write The Color Enum And Generator

Create `color.go`:

```go
// color.go
package enumjson

import "errors"

//go:generate go run cmd/enumgen/main.go -type=Color -values=Red,Green,Blue,Yellow

// Color represents a named color constant.
type Color int

const (
	ColorRed Color = iota
	ColorGreen
	ColorBlue
	ColorYellow
)

// ErrInvalidColor is returned when a Color value has no string mapping.
var ErrInvalidColor = errors.New("invalid color value")

// ErrUnknownColor is returned when a string cannot be mapped to a Color.
var ErrUnknownColor = errors.New("unknown color name")
```

Create `cmd/enumgen/main.go`:

```go
// cmd/enumgen/main.go
//go:build ignore

package main

import (
	"bytes"
	"flag"
	"fmt"
	"go/format"
	"os"
	"strings"
	"text/template"
)

var tmpl = template.Must(template.New("enumjson").Parse(`// Code generated by cmd/enumgen/main.go; DO NOT EDIT.

package {{ .Package }}

import (
	"encoding/json"
	"fmt"
)

var _{{ .Type }}NameMap = map[{{ .Type }}]string{
{{- range .Values }}
	{{ $.Type }}{{ . }}: "{{ . }}",
{{- end }}
}

var _{{ .Type }}ValueMap = map[string]{{ .Type }}{
{{- range .Values }}
	"{{ . }}": {{ $.Type }}{{ . }},
{{- end }}
}

// MarshalJSON implements json.Marshaler for {{ .Type }}.
func (t {{ .Type }}) MarshalJSON() ([]byte, error) {
	name, ok := _{{ .Type }}NameMap[t]
	if !ok {
		return nil, fmt.Errorf("MarshalJSON: invalid {{ .Type }} value %d: %w", int(t), ErrInvalid{{ .Type }})
	}
	return json.Marshal(name)
}

// UnmarshalJSON implements json.Unmarshaler for {{ .Type }}.
func (t *{{ .Type }}) UnmarshalJSON(data []byte) error {
	var s string
	if err := json.Unmarshal(data, &s); err != nil {
		return err
	}
	val, ok := _{{ .Type }}ValueMap[s]
	if !ok {
		return fmt.Errorf("UnmarshalJSON: unknown {{ .Type }} %q: %w", s, ErrUnknown{{ .Type }})
	}
	*t = val
	return nil
}
`))

type templateData struct {
	Package string
	Type    string
	Values  []string
}

func main() {
	typeName := flag.String("type", "", "type name (required)")
	values := flag.String("values", "", "comma-separated value names without type prefix (required)")
	output := flag.String("output", "", "output file (default: lowercase(type)_json_gen.go)")
	flag.Parse()

	if *typeName == "" || *values == "" {
		fmt.Fprintln(os.Stderr, "usage: enumgen -type=T -values=A,B,C [-output=file.go]")
		os.Exit(1)
	}

	pkg := os.Getenv("GOPACKAGE")
	if pkg == "" {
		pkg = "enumjson"
	}

	outFile := *output
	if outFile == "" {
		outFile = strings.ToLower(*typeName) + "_json_gen.go"
	}

	vals := strings.Split(*values, ",")
	for i := range vals {
		vals[i] = strings.TrimSpace(vals[i])
	}

	d := templateData{
		Package: pkg,
		Type:    *typeName,
		Values:  vals,
	}

	var buf bytes.Buffer
	if err := tmpl.Execute(&buf, d); err != nil {
		fmt.Fprintf(os.Stderr, "template error: %v\n", err)
		os.Exit(1)
	}

	formatted, err := format.Source(buf.Bytes())
	if err != nil {
		fmt.Fprintf(os.Stderr, "format error: %v\n%s\n", err, buf.String())
		os.Exit(1)
	}

	if err := os.WriteFile(outFile, formatted, 0o644); err != nil {
		fmt.Fprintf(os.Stderr, "write error: %v\n", err)
		os.Exit(1)
	}

	fmt.Fprintf(os.Stdout, "generated %s for type %s\n", outFile, *typeName)
}
```

The generator reads `-type` and `-values` flags, reads `GOPACKAGE` from the environment, renders the template, formats the output with `go/format`, and writes the file. The `//go:build ignore` constraint on line one keeps it out of normal builds.

Run the generator manually to see what it produces:

```bash
GOPACKAGE=enumjson go run cmd/enumgen/main.go -type=Color -values=Red,Green,Blue,Yellow
cat color_json_gen.go
```

### Exercise 2: Add The Pre-Generated File

The generator writes `color_json_gen.go`. Commit this file so the package compiles and tests pass without running `go generate`. Here is the file the generator produces:

Create `color_json_gen.go`:

```go
// Code generated by cmd/enumgen/main.go; DO NOT EDIT.

package enumjson

import (
	"encoding/json"
	"fmt"
)

var _ColorNameMap = map[Color]string{
	ColorRed:    "Red",
	ColorGreen:  "Green",
	ColorBlue:   "Blue",
	ColorYellow: "Yellow",
}

var _ColorValueMap = map[string]Color{
	"Red":    ColorRed,
	"Green":  ColorGreen,
	"Blue":   ColorBlue,
	"Yellow": ColorYellow,
}

// MarshalJSON implements json.Marshaler for Color.
func (t Color) MarshalJSON() ([]byte, error) {
	name, ok := _ColorNameMap[t]
	if !ok {
		return nil, fmt.Errorf("MarshalJSON: invalid Color value %d: %w", int(t), ErrInvalidColor)
	}
	return json.Marshal(name)
}

// UnmarshalJSON implements json.Unmarshaler for Color.
func (t *Color) UnmarshalJSON(data []byte) error {
	var s string
	if err := json.Unmarshal(data, &s); err != nil {
		return err
	}
	val, ok := _ColorValueMap[s]
	if !ok {
		return fmt.Errorf("UnmarshalJSON: unknown Color %q: %w", s, ErrUnknownColor)
	}
	*t = val
	return nil
}
```

The generated file is in `package enumjson` and references `ErrInvalidColor` and `ErrUnknownColor` from `color.go`. Both files compile together as a single package. The `MarshalJSON` and `UnmarshalJSON` methods wrap errors with `%w` so callers can use `errors.Is`.

### Exercise 3: Test The Generated Methods

Create `enumjson_test.go`:

```go
package enumjson

import (
	"encoding/json"
	"errors"
	"testing"
)

func TestMarshalJSONKnownValues(t *testing.T) {
	t.Parallel()

	cases := []struct {
		color Color
		want  string
	}{
		{ColorRed, `"Red"`},
		{ColorGreen, `"Green"`},
		{ColorBlue, `"Blue"`},
		{ColorYellow, `"Yellow"`},
	}

	for _, tc := range cases {
		t.Run(tc.want, func(t *testing.T) {
			t.Parallel()
			got, err := json.Marshal(tc.color)
			if err != nil {
				t.Fatalf("Marshal(%d): unexpected error: %v", tc.color, err)
			}
			if string(got) != tc.want {
				t.Fatalf("Marshal(%d) = %s, want %s", tc.color, got, tc.want)
			}
		})
	}
}

func TestMarshalJSONInvalidValue(t *testing.T) {
	t.Parallel()

	_, err := json.Marshal(Color(99))
	if err == nil {
		t.Fatal("Marshal(Color(99)) should fail")
	}
	if !errors.Is(err, ErrInvalidColor) {
		t.Fatalf("Marshal(Color(99)) error = %v, want wrapping ErrInvalidColor", err)
	}
}

func TestUnmarshalJSONKnownValues(t *testing.T) {
	t.Parallel()

	cases := []struct {
		input string
		want  Color
	}{
		{`"Red"`, ColorRed},
		{`"Green"`, ColorGreen},
		{`"Blue"`, ColorBlue},
		{`"Yellow"`, ColorYellow},
	}

	for _, tc := range cases {
		t.Run(tc.input, func(t *testing.T) {
			t.Parallel()
			var c Color
			if err := json.Unmarshal([]byte(tc.input), &c); err != nil {
				t.Fatalf("Unmarshal(%s): unexpected error: %v", tc.input, err)
			}
			if c != tc.want {
				t.Fatalf("Unmarshal(%s) = %d, want %d", tc.input, c, tc.want)
			}
		})
	}
}

func TestUnmarshalJSONUnknownValue(t *testing.T) {
	t.Parallel()

	var c Color
	err := json.Unmarshal([]byte(`"Purple"`), &c)
	if err == nil {
		t.Fatal(`Unmarshal("Purple") should fail`)
	}
	if !errors.Is(err, ErrUnknownColor) {
		t.Fatalf("Unmarshal error = %v, want wrapping ErrUnknownColor", err)
	}
}

func TestRoundTrip(t *testing.T) {
	t.Parallel()

	type Palette struct {
		Primary   Color `json:"primary"`
		Secondary Color `json:"secondary"`
	}

	p := Palette{Primary: ColorBlue, Secondary: ColorYellow}
	data, err := json.Marshal(p)
	if err != nil {
		t.Fatalf("Marshal: %v", err)
	}

	var p2 Palette
	if err := json.Unmarshal(data, &p2); err != nil {
		t.Fatalf("Unmarshal: %v", err)
	}

	if p.Primary != p2.Primary || p.Secondary != p2.Secondary {
		t.Fatalf("round-trip failed: got %+v, want %+v", p2, p)
	}
}
```

The tests use `errors.Is` to verify that error wrapping chains work correctly. `TestMarshalJSONInvalidValue` proves the `%w` wrap in `MarshalJSON` is reachable. `TestRoundTrip` proves that a struct with `Color` fields survives a full marshal/unmarshal cycle.

## Common Mistakes

### Omitting `//go:build ignore` On The Generator

Wrong: the generator file at `cmd/enumgen/main.go` has no build constraint.

What happens: `go build ./...` and `go test ./...` try to compile the generator as part of the package. Since the generator is `package main`, it either conflicts with other `package main` files or introduces imports (like `text/template`) that are not needed at runtime and make the package heavier.

Fix: add `//go:build ignore` as the very first line of the generator file. This tells the Go toolchain to skip the file during normal builds. The line must be first, before the package declaration, and must be followed by a blank line before the package clause.

### Skipping `go/format` And Shipping Malformed Output

Wrong: the generator writes template output directly to the file without calling `format.Source`.

What happens: the generated file compiles if the template is correct, but it is not `gofmt`-clean. CI may reject it. Worse, if the template has a subtle syntax error (a missing brace, a stray comma), the file is written and the error is only discovered when the package is compiled — far from the generator that caused it.

Fix: always pass the rendered template output through `go/format.Source`. If `format.Source` returns an error, print the raw output to stderr and exit non-zero. This surfaces template bugs immediately when the generator runs, not later during compilation.

### Naming The Error Variable Inconsistently With The Template

Wrong: `color.go` defines `ErrInvalidColor` but the template emits `ErrColor` (the generator uses `Err{{ .Type }}` which expands to `ErrColor`, not `ErrInvalidColor`).

What happens: the generated file references `ErrColor`, which does not exist in `color.go`. The build fails with `undefined: ErrColor`.

Fix: align the template placeholders with the actual variable names in the source package. Either change the template to emit `ErrInvalid{{ .Type }}` (which expands to `ErrInvalidColor`), or rename the variable in `color.go` to match what the template emits. The convention in this lesson is `ErrInvalid<Type>` and `ErrUnknown<Type>`, and the template uses `ErrInvalid{{ .Type }}` and `ErrUnknown{{ .Type }}` accordingly.

## Summary

- Generator programs are ordinary Go programs invoked by `//go:generate`; `//go:build ignore` keeps them out of the normal build.
- `text/template` defines the code shape; `go/format.Source` validates syntax and enforces `gofmt` style.
- `GOPACKAGE` provides the target package name without hardcoding it in the directive or the generator.
- The `// Code generated by ...; DO NOT EDIT.` marker must be the first line so tools skip the file.
- Commit the pre-generated file so consumers do not need to run the generator.
- Wrap sentinel errors with `%w` in generated methods so callers can use `errors.Is`.
