# 1. Go Generate Basics

`go generate` is not a build step: the Go toolchain never runs it automatically. It is a convention for embedding shell commands in source comments and running them on demand. The hard part is understanding what `go generate` actually does (it is a dumb command expander), what it does not do (it does not parse Go, it does not know about types), and how to ship generated files so that consumers never have to run generators themselves.

```text
buildinfo/
  go.mod
  doc.go
  gen/genbuildinfo.go
  buildinfo.go
  buildinfo_gen.go
  buildinfo_test.go
```

The package exposes two constants produced by the generator and two functions that use them. The pre-generated `buildinfo_gen.go` ships with the lesson so tests pass without running `go generate`.

## Concepts

### The Directive Is A Comment, Not A Build Rule

`//go:generate COMMAND` is a specially formatted comment. `go generate` scans source files for lines matching that pattern and executes `COMMAND` as a shell command. It does not track dependencies, does not cache results, and does not know whether the output is up to date. It runs every directive in every matching file, in order, stopping if any command exits non-zero.

The canonical form is:

```
//go:generate go run gen/mygenerator.go -type=MyType
```

No space between `//` and `go:generate`. The rest of the line is passed verbatim to the shell.

### Environment Variables Set By go generate

When `go generate` runs a directive, it sets several variables the generator can read:

| Variable    | Value                                      |
|-------------|---------------------------------------------|
| GOPACKAGE   | the package name of the file containing the directive |
| GOFILE      | the base name of the file containing the directive   |
| GOLINE      | the line number of the directive                     |
| GOROOT      | the Go installation root                             |
| GOARCH      | the target architecture                              |
| GOOS        | the target OS                                        |

A generator that reads `GOPACKAGE` can emit a file with the correct `package` declaration without receiving it as a flag. This is the reason `doc.go` is the natural home for generate directives: its package comment and the `GOPACKAGE` variable are always consistent.

### The //go:build ignore Convention

A generator is a Go program that is not part of the package it generates code for. Putting it under `gen/genbuildinfo.go` with `//go:build ignore` as the first line tells the Go toolchain to exclude it from every build and test. Without that constraint, `go build ./...` would try to compile it as part of the package and fail.

### Generated File Conventions

Generated files follow two conventions enforced by the Go ecosystem:

1. The file name ends in `_gen.go` (or begins with `zz_generated_`). This makes it easy to exclude them from grep searches and code reviews.
2. The first line of the file is a comment matching the pattern `// Code generated ... DO NOT EDIT.` Formatters, linters, and editors use this marker to skip the file.

Generated files are committed to version control. Consumers of a package run `go get`, not `go generate`. Only authors run generators, and only when they change the input.

### Trade-offs And Failure Modes

`go generate` is intentionally simple. That simplicity means:

- It has no dependency tracking. If you rename a type and forget to regenerate, the build breaks at compile time, not at generate time.
- It relies on the generator being installed. `go install` is the right way to distribute a generator; absence of the tool produces a non-zero exit from the directive.
- It has no parallel execution. Directives in the same file run sequentially. Across packages, `go generate ./...` processes packages in unspecified order.

### Exercise 1: Write The Generator

The generator reads `GOPACKAGE` and writes `buildinfo_gen.go` with two constants.

Create `gen/genbuildinfo.go`:

```go
//go:build ignore

package main

import (
	"fmt"
	"os"
	"time"
)

func main() {
	pkg := os.Getenv("GOPACKAGE")
	if pkg == "" {
		pkg = "buildinfo"
	}

	now := time.Now().UTC().Format(time.RFC3339)

	f, err := os.Create("buildinfo_gen.go")
	if err != nil {
		fmt.Fprintf(os.Stderr, "create: %v\n", err)
		os.Exit(1)
	}
	defer f.Close()

	fmt.Fprintf(f, "// Code generated by gen/genbuildinfo.go; DO NOT EDIT.\n\n")
	fmt.Fprintf(f, "package %s\n\n", pkg)
	fmt.Fprintf(f, "const (\n")
	fmt.Fprintf(f, "\tGeneratedAt = %q\n", now)
	fmt.Fprintf(f, "\tGeneratedBy = %q\n", "genbuildinfo")
	fmt.Fprintf(f, ")\n")
}
```

The `//go:build ignore` constraint prevents this file from being compiled as part of the `buildinfo` package. The generator uses `GOPACKAGE` so the package declaration in the output is always correct regardless of how the package is renamed.

### Exercise 2: Write The Package And Its go:generate Directive

The directive lives in `doc.go` so it stays close to the package declaration and the `GOPACKAGE` variable is correctly populated.

Create `doc.go`:

```go
// Package buildinfo exposes constants produced by go generate.
//
//go:generate go run gen/genbuildinfo.go
package buildinfo
```

Create `buildinfo.go`:

```go
package buildinfo

import "fmt"

// Info returns a human-readable build info string using the generated constants.
func Info() string {
	return fmt.Sprintf("generated at %s by %s", GeneratedAt, GeneratedBy)
}

// HasGeneratedAt reports whether GeneratedAt is non-empty.
func HasGeneratedAt() bool {
	return GeneratedAt != ""
}
```

The functions reference `GeneratedAt` and `GeneratedBy` declared in `buildinfo_gen.go`. The package compiles only when that file exists.

### Exercise 3: Ship The Pre-Generated File And Write Tests

Because `go generate` is never run automatically, the generated file must be committed so that anyone cloning the repository can run `go test` immediately. The pre-generated file below uses a fixed timestamp so tests are deterministic.

Create `buildinfo_gen.go`:

```go
// Code generated by gen/genbuildinfo.go; DO NOT EDIT.

package buildinfo

const (
	GeneratedAt = "2024-01-01T00:00:00Z"
	GeneratedBy = "genbuildinfo"
)
```

Create `buildinfo_test.go`:

```go
package buildinfo

import (
	"strings"
	"testing"
)

func TestInfoContainsGeneratedAt(t *testing.T) {
	t.Parallel()
	got := Info()
	if !strings.Contains(got, GeneratedAt) {
		t.Fatalf("Info() = %q, want it to contain GeneratedAt %q", got, GeneratedAt)
	}
}

func TestInfoContainsGeneratedBy(t *testing.T) {
	t.Parallel()
	got := Info()
	if !strings.Contains(got, GeneratedBy) {
		t.Fatalf("Info() = %q, want it to contain GeneratedBy %q", got, GeneratedBy)
	}
}

func TestHasGeneratedAt(t *testing.T) {
	t.Parallel()
	if !HasGeneratedAt() {
		t.Fatal("HasGeneratedAt() should return true when GeneratedAt is set")
	}
}

func TestGeneratedAtIsNonEmpty(t *testing.T) {
	t.Parallel()
	if GeneratedAt == "" {
		t.Fatal("GeneratedAt should be set by go generate")
	}
}

func TestGeneratedByIsNonEmpty(t *testing.T) {
	t.Parallel()
	if GeneratedBy == "" {
		t.Fatal("GeneratedBy should be set by go generate")
	}
}
```

Run the generator to see it produce a real timestamp, then confirm the package still builds:

```bash
cd ~/go-exercises/buildinfo
go generate ./...
cat buildinfo_gen.go
go build ./...
```

The first line of `buildinfo_gen.go` is the `DO NOT EDIT` marker. The `GeneratedAt` constant now holds the real current time rather than the fixed value from Exercise 3.

## Common Mistakes

### Missing //go:build ignore On The Generator

Wrong: `gen/genbuildinfo.go` has no build constraint.

What happens: `go build ./...` tries to compile `gen/genbuildinfo.go` as part of `package buildinfo`. It fails because the file declares `package main`, producing a "found packages main and buildinfo" error.

Fix: the first line of every generator program must be `//go:build ignore`. That line excludes the file from all normal builds.

### Generating Into The Wrong Directory

Wrong: the generator calls `os.Create("buildinfo_gen.go")` but is invoked from a directory other than the package root. The file lands in the wrong place.

What happens: `go build ./...` cannot find `GeneratedAt` and `GeneratedBy`, producing "undefined" errors.

Fix: `go generate` always changes the working directory to the directory of the file containing the directive before running the command. If the generator is invoked with `go run gen/genbuildinfo.go`, the working directory is the package root (where `doc.go` lives), so `os.Create("buildinfo_gen.go")` creates the file in the right place. Never use absolute paths or `..` in `os.Create` inside a generator.

### Editing The Generated File By Hand

Wrong: a developer edits `buildinfo_gen.go` directly to fix a bug in the generated output.

What happens: the fix works until someone runs `go generate ./...` again, at which point the hand-edit is overwritten and the bug returns.

Fix: fix the generator (`gen/genbuildinfo.go`), then regenerate. The `DO NOT EDIT` marker exists specifically to warn against hand-edits. If the generator does not produce the right output, the generator is wrong.

## Summary

- `//go:generate COMMAND` is a comment; `go generate` runs it on demand, never automatically.
- `go generate` sets `GOPACKAGE`, `GOFILE`, `GOLINE`, `GOARCH`, `GOOS`, and `GOROOT` for each directive.
- Generator programs belong under `gen/` (or a similar directory) with `//go:build ignore` to exclude them from the package build.
- Generated files must include `// Code generated ... DO NOT EDIT.` as their first line.
- Generated files are committed to version control so consumers can build without running generators.
