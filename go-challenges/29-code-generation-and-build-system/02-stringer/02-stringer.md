# 2. Stringer

The `stringer` tool generates `String()` methods for integer-based enum types. The hard part is not the tool itself but understanding why the generated code looks the way it does: an index into a concatenated string literal is O(1) and allocation-free, while a `map[Status]string` is not. This lesson also introduces the validity-guard trick that turns a forgotten regeneration into a compile error rather than a silent wrong value.

This lesson is OFFLINE-BAR: `stringer` is an external tool (`golang.org/x/tools/cmd/stringer`) that must be installed separately. The pre-generated `status_string.go` shipped with this lesson lets `go test` pass without running `go generate`.

```text
taskstatus/
  go.mod
  taskstatus.go
  status_string.go   (pre-generated)
  taskstatus_test.go
```

## Concepts

### Stringer's Index-Based String Lookup

The generated `String()` method does not use a `map[Status]string`. Instead it builds two compile-time constants:

```go
const _Status_name = "PendingRunningDoneFailedCancelled"
var _Status_index = [...]uint8{0, 7, 14, 18, 24, 33}
```

A single string literal holds all names concatenated. The index array records where each name starts and ends. Looking up `StatusRunning` (value 1) returns `_Status_name[7:14]` ŌĆö a string slice with no heap allocation and O(1) cost regardless of how many constants exist. A map would require a hash computation and bucket lookup on every call, and its entries would live on the heap.

### The Validity Function

The generated file contains:

```go
func _() {
	var x [1]struct{}
	_ = x[StatusPending-0]
	_ = x[StatusRunning-1]
	_ = x[StatusDone-2]
	_ = x[StatusFailed-3]
	_ = x[StatusCancelled-4]
}
```

This function is never called. Its purpose is compile-time verification. Indexing a `[1]struct{}` with any value other than `0` is a compile error ("invalid array index"). If you insert a new constant in the middle of the block ŌĆö for example, inserting `StatusQueued` before `StatusRunning`, which shifts `StatusRunning` from value 1 to value 2 ŌĆö and forget to re-run `stringer`, the existing guard line `_ = x[StatusRunning-1]` now evaluates to `x[2-1]` which is `x[1]`, an out-of-bounds index on a `[1]struct{}`, and the compiler rejects the package. You get a build failure instead of silently wrong `String()` output. Note that appending a new constant at the end of the block is NOT caught by this guard: no guard line references the new constant until you regenerate, so the package compiles and the new constant silently falls back to `"Status(N)"` ŌĆö see Common Mistakes.

### When Trimprefix Matters

Without `-trimprefix=Status`, `StatusPending.String()` returns `"StatusPending"`. In log messages like `status=StatusPending` the type prefix is redundant because the field name already conveys "this is a status". With `-trimprefix=Status` the output is `"Pending"`, which reads naturally in structured logs and error messages.

The trade-off: trimmed names are less self-documenting when printed in isolation without context. Use trimprefix when you control how the value is displayed; omit it when the enum value may appear outside its domain (e.g., in a generic audit log).

### Out-of-Range Values

For unknown values the generated code falls back to:

```go
"Status(" + strconv.FormatInt(int64(i), 10) + ")"
```

`Status(99)` is more useful than an empty string or a panic. The numeric form lets you identify the raw value in logs, which is important during schema migrations when new constants may be written to a database before the consumer binary is updated.

	StatusRunning
	StatusDone
	StatusFailed
	StatusCancelled
)

The `//go:generate` directive records the exact command to regenerate the `String()` method. Anyone who adds a new constant runs `go generate ./...` to update the generated file.

### Exercise 2: Add The Pre-Generated String Method

In practice you run `go generate` once and commit the result. The file below is exactly what `stringer -type=Status -trimprefix=Status` produces for the constants above.

Create `status_string.go`:

```go
// Code generated by "stringer -type=Status -trimprefix=Status"; DO NOT EDIT.

package taskstatus

import "strconv"

func _() {
	// An "invalid array index" compiler error signifies that the constant values have changed.
	// Re-run the stringer command to generate them again.
	var x [1]struct{}
	_ = x[StatusPending-0]
	_ = x[StatusRunning-1]
	_ = x[StatusDone-2]
	_ = x[StatusFailed-3]
	_ = x[StatusCancelled-4]
}

const _Status_name = "PendingRunningDoneFailedCancelled"

var _Status_index = [...]uint8{0, 7, 14, 18, 24, 33}

func (i Status) String() string {
	if i < 0 || i >= Status(len(_Status_index)-1) {
		return "Status(" + strconv.FormatInt(int64(i), 10) + ")"
	}
	return _Status_name[_Status_index[i]:_Status_index[i+1]]
}
```

### Exercise 3: Test The String Method

Create `taskstatus_test.go`:

```go
package taskstatus

import (
	"testing"
)

func TestStatusStringKnownValues(t *testing.T) {
	t.Parallel()

	cases := []struct {
		status Status
		want   string
	}{
		{StatusPending, "Pending"},
		{StatusRunning, "Running"},
		{StatusDone, "Done"},
		{StatusFailed, "Failed"},
		{StatusCancelled, "Cancelled"},
	}

	for _, tc := range cases {
		tc := tc
		t.Run(tc.want, func(t *testing.T) {
			t.Parallel()
			if got := tc.status.String(); got != tc.want {
				t.Fatalf("Status(%d).String() = %q, want %q", int(tc.status), got, tc.want)
			}
		})
	}
}

func TestStatusStringUnknownValue(t *testing.T) {
	t.Parallel()

	unknown := Status(99)
	got := unknown.String()
	want := "Status(99)"
	if got != want {
		t.Fatalf("Status(99).String() = %q, want %q", got, want)
	}
}

func TestStatusStringNegativeValue(t *testing.T) {
	t.Parallel()

	neg := Status(-1)
	got := neg.String()
	want := "Status(-1)"
	if got != want {
		t.Fatalf("Status(-1).String() = %q, want %q", got, want)
	}
}
```

The table-driven test in `TestStatusStringKnownValues` pins the contract: each constant maps to exactly the trimmed string. `TestStatusStringUnknownValue` and `TestStatusStringNegativeValue` verify the fallback format so a reader knows what to expect in logs when an unrecognised value appears.

## Common Mistakes

### Adding A Constant Without Regenerating

Wrong: you add `StatusArchived Status = iota` after `StatusCancelled` and run `go build`. The package compiles because the generated `String()` bounds check only covers indices 0-4; `StatusArchived` (value 5) silently returns `"Status(5)"`.

What happens: logs contain `"Status(5)"` instead of `"Archived"`, obscuring the value in dashboards and alerts.

Fix: run `go generate ./...` after every change to the constant block. The validity function `func _()` catches index mismatches at compile time only when the new constant value is outside the range already covered; adding values inside the existing range does not trigger the guard, so regenerating is always the safe action.

### Editing The Generated File By Hand

Wrong: you open `status_string.go` and change `"Pending"` to `"pending"` to match a database column.

What happens: the next `go generate` run silently overwrites your edit, losing the change.

Fix: never edit `*_string.go` files. If the display form differs from the constant name, wrap `Status.String()` in a helper that post-processes the output, or use a separate lookup table in a non-generated file.

### Mismatched Index Array After Manual Constant Reordering

Wrong: you swap `StatusRunning` and `StatusDone` in `taskstatus.go` without regenerating. The `_Status_index` array still encodes the old order. `StatusRunning` (now value 2) returns `_Status_name[14:18]` which is `"Done"`.

What happens: `StatusRunning.String()` returns `"Done"` ŌĆö a wrong value with no error.

Fix: treat the generated file as a pure derivative of `taskstatus.go`. Any edit to the constant block, including reordering, requires an immediate `go generate ./...`.

## Summary

- `stringer` generates `String()` methods for integer enum types; install it with `go install golang.org/x/tools/cmd/stringer@latest`.
- The generated code uses a concatenated string literal and an index array for O(1), allocation-free lookups.
- The `func _()` validity guard turns a forgotten regeneration into a compile error when existing constant values shift (reordering or inserting in the middle); appending at the end is not caught until you regenerate.
- `-trimprefix=T` removes the type-name prefix from string output, producing cleaner log messages.
- Out-of-range values fall back to `"Status(N)"` ŌĆö never panic, always informative.
- Always regenerate after any change to the constant block; commit the generated file.
