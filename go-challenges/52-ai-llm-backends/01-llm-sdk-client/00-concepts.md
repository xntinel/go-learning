# Calling LLMs with the Anthropic and OpenAI Go SDKs — Concepts

A service that calls a large language model is, structurally, an HTTP client with
unusually leaky semantics: a moving JSON API, credentials pulled from the
environment, retries that must back off on the right status codes, polymorphic
response bodies, and per-vendor quirks that no amount of wishful abstraction
makes go away. This lesson is the ports-and-adapters pattern applied to that
problem — the same hexagonal split this infrastructure already runs (a
dependency-free domain with narrow ports, and cloud-specific adapters behind
them). You do not sprinkle `client.Messages.New` across HTTP handlers. You define
one narrow `Completer` port in the domain, implement it once per vendor in an
adapter package, drive it from config, and test it against `httptest` with zero
network. Read this file once and you have the model you need for all three
exercises that follow.

## Concepts

### Why an official SDK instead of hand-rolled HTTP

Both the Anthropic SDK (`github.com/anthropics/anthropic-sdk-go`) and the OpenAI
SDK (`github.com/openai/openai-go/v3`) are generated by Stainless from an OpenAPI
spec. That is not a trivia fact; it is why the two libraries are shaped almost
identically and why building on them beats a hand-rolled `net/http` client. You
get typed request and response structs, credential resolution from the
environment (`ANTHROPIC_API_KEY`, `OPENAI_API_KEY`), automatic retries with
exponential backoff on transient failures, streaming helpers, and union types for
polymorphic content. Reimplementing any one of those against an API that ships
breaking-ish changes every few months is wasted, error-prone work. The SDK is the
adapter's implementation detail — but it is a good one, and you should not fight
it.

### The port belongs to the domain; the SDK stays in the adapter

The load-bearing design decision is that the domain never imports a vendor SDK.
It declares an interface:

```go
type Completer interface {
	Complete(ctx context.Context, req Request) (Response, error)
}
```

`Request` and `Response` are neutral types the domain owns: a model ID, a system
prompt, a slice of role-tagged messages, a max-token cap on the way in; extracted
text, token usage, and a stop reason on the way out. Each vendor gets an adapter
package that wraps its SDK client and translates the neutral shapes to and from
that vendor's typed params. A handler depends on `Completer`, not on Anthropic or
OpenAI, so switching vendors is a config change and testing is a matter of
substituting a fake. This is the whole payoff: vendor independence and
testability. The rest of this file is about where that abstraction is real and
where it leaks.

### The two APIs have genuinely different shapes

The abstraction is real because the two request/response models rhyme, but it
leaks because they differ in ways the port cannot paper over — the adapters must
own the differences.

Anthropic's Messages API takes the system prompt as a top-level `System` field,
separate from the message array. It returns a list of content *blocks*, not a
single string: a response may contain a `thinking` block followed by a `text`
block, and you narrow the union to pull the text out. And it *requires*
`MaxTokens` — omit it and the call errors.

OpenAI's Chat Completions API folds the system (or `developer`) prompt into the
`messages` array as a message with `role: "system"`, returns
`Choices[0].Message.Content` as a plain string, and treats the output cap
(`MaxCompletionTokens`) as optional. OpenAI also exposes a newer Responses API
(`client.Responses.New`) alongside Chat Completions; this lesson uses Chat
Completions because it is the common denominator.

So the neutral `Request` keeps `System` as its own field (Anthropic wants it
top-level; the OpenAI adapter prepends a system message), and the neutral
`Response` carries a single flattened `Text` string (the Anthropic adapter
concatenates its text blocks; the OpenAI adapter reads `Choices[0]`). Design the
port around what is common; let each adapter absorb the shape mismatch.

### The optional-field model: `param.Opt[T]` and the constructor helpers

Both SDKs face the same JSON problem: how do you distinguish "temperature is
unset, use the server default" from "temperature is deliberately 0"? A bare Go
`float64` field cannot — its zero value *is* 0.0, and it would serialize as an
explicit `"temperature": 0` on the wire. The Stainless answer is a wrapper type,
`param.Opt[T]`, plus constructor helpers: `openai.Int`, `openai.Float`,
`openai.String`, `openai.Bool` (and the identical `anthropic.Int`,
`anthropic.Float`, `anthropic.String`). A field left as its zero `param.Opt[T]`
is omitted from the request entirely; a field set with `openai.Float(0)` sends an
explicit 0.

```go
params.Temperature = openai.Float(0.2) // sends "temperature": 0.2
// params.Temperature left zero        // field omitted; server default applies
```

This matters precisely when the caller wants a deterministic-ish `temperature: 0`
and a naive adapter that copies a plain `float64` would send 0 unconditionally,
overriding the server default for every request. Use the constructors; never set
a bare zero and expect it to mean "unset".

Reading the response is the mirror skill. Anthropic content is a union — you call
`block.AsAny()` and type-switch to `anthropic.TextBlock` before touching `.Text`.
OpenAI gives you `Choices`, a slice you must length-check before indexing `[0]`.
Both narrowings are where careless code panics.

### Client options versus request params

There are two configuration surfaces, and conflating them is a common source of
confusion. **Request params** (the `MessageNewParams` / `ChatCompletionNewParams`
struct) carry the semantic request: model, messages, sampling. **Client options**
(`option.With*`) configure the client and its transport:
`option.WithAPIKey`, `option.WithBaseURL`, `option.WithRequestTimeout`,
`option.WithMaxRetries`, `option.WithHTTPClient`. Options can be applied at client
construction or passed per-call as trailing variadic arguments.

`WithBaseURL` is the single most valuable production knob. The same adapter code
points at `api.anthropic.com` in production, at an `httptest.NewServer` in CI, or
at an LLM gateway or corporate egress proxy (LiteLLM, Cloudflare AI Gateway) when
traffic must be routed, budgeted, or observed centrally. Because the base URL is a
client option and not baked into the SDK, none of that requires touching the
adapter's request-building logic.

### Model IDs are configuration, not constants

A model ID is an opaque, volatile string. New models ship, old ones are retired,
and aliases exist alongside dated snapshots. The SDKs expose typed constants where
they can — `anthropic.ModelClaudeOpus4_8`, `openai.ChatModelGPT5_2` — and those
are convenient defaults, but you should treat the actual ID as env/config-driven
and let the adapter accept a string. Never guess or construct an ID, and never
append a date suffix to an alias: a wrong ID is a 404 at runtime, not a compile
error. Config-driven model selection also means a single deploy can move traffic
between a cheap model and an expensive one without a code change.

### Timeout semantics are a trap

`option.WithRequestTimeout` sets a *per-attempt* bound. With auto-retries enabled
(default 2 retries), the wall-clock ceiling is roughly
`timeout × (max_retries + 1)` — three attempts of 20 seconds each is a full
minute, not twenty seconds. For a hard end-to-end deadline that bounds the whole
operation including every retry, pass a `context.WithTimeout` deadline into the
call. The context deadline is the true ceiling; the request timeout only shapes
each individual attempt. When the deadline fires, the SDK surfaces an error that
wraps `context.DeadlineExceeded`, which `errors.Is` detects.

### Retry and error taxonomy

Both SDKs auto-retry connection errors and HTTP 408, 409, 429, and 5xx with
exponential backoff; the default is 2 retries, tunable with
`option.WithMaxRetries` (and `WithMaxRetries(0)` disables it, which is what your
tests want so a stubbed 500 surfaces deterministically instead of being retried
into a timeout). Non-retryable 4xx — 401 auth, 400 bad request — should be
classified and surfaced immediately; retrying them just wastes the deadline.

Classify by unwrapping the vendor error type with `errors.As`, not by
string-matching messages. `errors.As(err, &apiErr)` where `apiErr` is
`*anthropic.Error` or `*openai.Error` gives you a `StatusCode` to switch on; the
Anthropic error also carries a `RequestID` you want in logs when you open a
support ticket. Map those into a shared taxonomy — auth, rate-limited,
bad-request, transient, timeout — so the domain reacts to *kinds* of failure, not
to a vendor's error struct. Leaking `*openai.Error` through the `Completer`
interface would defeat the port.

### Read usage and check the stop reason on every response

Token usage is not an afterthought; it is the basis for cost governance,
chargeback, and budget enforcement. Anthropic reports `Usage.InputTokens` and
`Usage.OutputTokens` (plus cache fields); OpenAI reports `Usage.PromptTokens`,
`Usage.CompletionTokens`, and `Usage.TotalTokens`. The field names differ — the
adapter normalizes them — but you read them on *every* response, because you
cannot reconstruct per-call cost after the fact.

Truncation is silent unless you check. Anthropic sets `stop_reason == "max_tokens"`
and OpenAI sets `finish_reason == "length"` when the model was cut off mid-output.
A truncated response is not a complete answer; treat it as an error condition or a
signal to raise the token cap, never as a finished result you hand downstream.

### The lowest-common-denominator trade-off, and when not to abstract

A `Completer` port buys vendor independence and offline testability, but the
abstraction is a lossy projection. System-prompt placement, tool and function
schemas, sampling-parameter acceptance, and token-counting fields all differ per
vendor. One specific sharp edge: sampling-parameter acceptance is
vendor- *and* model-specific — recent Anthropic models reject `temperature`,
`top_p`, and `top_k` with a 400, so a "generic" adapter that always sends
temperature will break on those models. The adapter must gate which params it
sends per target model. Design the port around the common core; let each adapter
own the quirks. And know when *not* to abstract: if you need one vendor's unique
feature, forcing it through the port makes the port a liability. The port is worth
it exactly when you want the common denominator across vendors; it is a cost when
you want a vendor's differentiator.

## Common Mistakes

### Omitting MaxTokens on the Anthropic request

`MaxTokens` is required on `MessageNewParams`. Developers coming from OpenAI, where
the output cap is optional, hit this immediately: the call errors with a
bad-request. The neutral `Request` should carry a max-tokens value the Anthropic
adapter always sets (defaulting it if the caller left it zero), even though the
OpenAI adapter treats it as optional.

### Reading resp.Content[0].Text directly on the Anthropic response

Anthropic content is a union of blocks, and the first block may be a `thinking`
block, not text. Indexing `[0]` and reading `.Text` returns the wrong content or
panics. Iterate the blocks, narrow each with `AsAny()`, and collect the text
blocks. Symmetrically, do not index `Choices[0]` on the OpenAI response without a
length check — an empty `Choices` slice panics.

### Treating a bare zero as "unset"

Setting `Temperature: 0.0` as a plain field (or expecting the zero
`param.Opt[T]` to carry a deliberate 0) is wrong in both directions. A bare zero
either serializes as an explicit 0 or is indistinguishable from unset. Use
`openai.Float` / `anthropic.Float` to send a value; leave the field at its zero
`param.Opt` to omit it.

### Treating the request timeout as a wall-clock cap

`WithRequestTimeout` is per-attempt. With retries, the real ceiling is
`timeout × (retries + 1)`. Code that wants a hard 20-second end-to-end bound and
sets only `WithRequestTimeout(20*time.Second)` can run for a minute. Use a
`context.WithTimeout` deadline for the operation-wide ceiling.

### Hardcoding or guessing a model ID

A model ID baked in as a string constant (or worse, guessed, or date-suffixed
onto an alias) becomes a runtime 404 the moment that ID is retired or was never
valid. Model IDs belong in config; use the SDK's typed constants only as
defaults.

### Catching one broad error type

Wrapping every failure as a single opaque error throws away the retryable
(429/5xx) versus non-retryable (401/400) distinction and the RequestID you need
for support. Classify with `errors.As` into the vendor error and switch on
`StatusCode`.

### Putting the API key in code

Both SDKs resolve `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` from the environment; the
zero-argument client works. A literal key in source is a leaked credential and an
avoidable one — pass `option.WithAPIKey` only when injecting a specific key (for
example, a dummy in tests).

### Hitting the live API in unit tests

A unit test that calls the real endpoint is slow, flaky, non-deterministic,
rate-limited, and expensive. Point the client at an `httptest.NewServer` via
`option.WithBaseURL` and return canned vendor JSON. Reserve real-network smoke
tests for a separate file behind a `//go:build online` tag so the default test
path never leaves the machine.

### Leaking vendor concepts through the port

Returning raw Anthropic content blocks, exposing provider tool schemas, or
surfacing `*openai.Error` through the `Completer` interface all defeat the point
of the port. The neutral types are the contract; the vendor types stop at the
adapter boundary.
