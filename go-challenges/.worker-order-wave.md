## EXERCISE QUALITY (highest priority)

- A library lesson is verified by a real `*_test.go`, NEVER by `main()`+printf+
  "Expected output". The verification must fail on its own when code regresses.
- Every library lesson includes, IN ADDITION to the `*_test.go`:
  1. an `Example...` function with an `// Output:` comment (auto-verified), and
  2. a real `cmd/demo/main.go` runnable with `go run ./cmd/demo` (touches only
     EXPORTED API; add small exported accessors if the demo needs to read state).
- The lesson is a real package (e.g. `package scheduler`), not `package main`,
  so same-package tests reach unexported fields.
- Tests: table-driven, `t.Parallel()`, sentinel errors wrapped with `%w`,
  assert with `errors.Is`. End at least one exercise with a "your turn" test.

## HARD RULES

- ALL Go code in fenced blocks MUST use TAB indentation, not spaces. Space-
  indented Go fails `gofmt -l` and is an automatic failure. Verify with the gate.
- NO emojis or decorative symbols (no checkmarks, stars, triangles, warnings,
  any unicode emoji/dingbat). Use plain text: "OK"/"FAIL", "Wrong"/"Fix". Only
  functional math operators are allowed.
- English everywhere.
- Do NOT invent Go APIs. Every stdlib signature must be real and compile with
  modern Go (corpus pins `go 1.26`). Prefer the stdlib over hand-rolling.
- No dead code; every branch reachable and, where it matters, tested.
- Research authoritative sources (go.dev/ref/spec, go.dev/blog, pkg.go.dev, the
  real source of any library named) before writing. Never write APIs from memory
  alone. Cite the 3-5 best sources you actually consulted in `## Resources`.
