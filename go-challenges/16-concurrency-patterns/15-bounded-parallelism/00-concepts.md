# 15. Bounded Parallelism — Concepts

Unbounded goroutine creation is one of the most common ways a Go service falls over in production. Launching one goroutine per item looks harmless on a developer laptop with a dozen items, but the same loop fed a queue of a hundred thousand items will open a hundred thousand sockets, allocate a hundred thousand stacks, and aim every one of them at a downstream that can serve perhaps fifty concurrent requests. The fix is bounded parallelism: a hard ceiling on how many tasks run at once. This file is the conceptual foundation for the four exercises that follow, each of which builds one bounded-parallelism pattern as an independent, self-contained Go module: a counting semaphore, a weighted semaphore, a semaphore-bounded HTTP fan-out, and a bounded object processor that aggregates every error.

## Concepts

### A Buffered Channel Is A Counting Semaphore

The idiom at the center of every pattern here is that a buffered channel of capacity N is a counting semaphore, and Go ships nothing more specialized in the standard library because nothing more is needed. A token is an empty struct, the cheapest value Go has. Acquiring a slot is a send: it succeeds while the buffer has room and blocks once the buffer is full. Releasing a slot is a receive: it frees exactly one unit of capacity. The capacity of the channel is the maximum number of tokens that can be held at once, which is the maximum number of tasks that may run concurrently. There is no counter to maintain and no mutex to forget: the channel's own buffer accounting is the semaphore's state.

The discipline that makes this correct is "acquire before you launch, release in a deferred close." The acquire must happen in the producer loop, before the goroutine is started, because the whole point is to make the loop itself block when the system is saturated. If you start the goroutine first and acquire inside it, the goroutine already exists and already holds a stack and a scheduler slot before the semaphore has any say; you have bounded nothing. The release must be deferred so that it runs even if the task panics, because a release that is skipped on the error path leaks a token permanently, and once enough tokens leak the channel can never be filled again and the next acquire blocks forever.

### Weighted Semaphores Let A Task Declare Its Cost

A counting semaphore treats every task as one indistinguishable unit, which is the right model when tasks are roughly equal. When they are not — a job that uploads one kilobyte and a job that uploads one hundred megabytes should not each consume "one slot" of a memory budget — the weighted semaphore lets each task declare a cost and consume that many units of a shared capacity. The standard-library construction is a buffered channel of `int` (or any token type) whose capacity is the total budget; a task of weight w sends w tokens to acquire and receives w tokens to release. The running sum of in-flight weights is held implicitly in the channel's buffer occupancy and is, by construction, never allowed to exceed the capacity.

The weighted version has a sharp edge that the counting version does not. Acquiring w tokens is not a single atomic operation on a channel; it is w separate sends, and a producer that has sent some but not all of them is holding a partial acquisition that blocks every later task behind it. This is acceptable and even desirable — it preserves the invariant that in-flight weight never exceeds capacity — but it means a single weight that exceeds the total capacity can never be satisfied and will block forever. A robust implementation clamps any individual weight to the capacity so that the largest possible task still fits. `golang.org/x/sync/semaphore.Weighted` is the canonical packaged form of this idea; its `Acquire(ctx, n)` does with a single call and a wait queue what the channel construction does with a loop, and it is the right reach when you want context-aware acquisition without writing the loop yourself. The channel version is the hermetic, dependency-free equivalent and is worth building once to understand what the package is doing.

### Measuring The Peak Is The Only Honest Proof

A bound that is claimed but not measured is a bound that is wrong. The way to prove a semaphore actually holds is to count tasks as they enter and leave the critical region and record the high-water mark, then assert that the high-water mark never exceeded the limit. The counter must be an atomic — `atomic.Int64` — because it is incremented and read from many goroutines at once, and recording the maximum requires a compare-and-swap loop rather than a plain store, because between reading the old maximum and writing the new one another goroutine may have moved it. The pattern is: add one on entry, then loop reading the current peak and attempting to swap in the new value only while the new value is larger; subtract one on exit. Run the assertion under the race detector. A test that observes a peak with an ordinary `int` will pass on a quiet laptop and the data race it hides will surface as a corrupted count, or worse a corrupted bound, on a loaded production host. The race detector turns that latent bug into a failed test, which is exactly where you want it.

### Bounding Is A Resource Trade-Off, Not A Universal Constant

There is no single correct concurrency limit; the right ceiling is whichever resource is the bottleneck. For CPU-bound work the answer is usually around `GOMAXPROCS`, because past the number of cores the extra goroutines only add scheduling overhead and cache pressure. For I/O-bound work the useful ceiling is far higher, often a few hundred, because each task spends most of its life blocked on the network and a single core can shepherd many such tasks. For work that fans out to a downstream service, the correct ceiling is the downstream's own concurrency budget: exceeding it converts your throughput problem into the downstream's outage, and a well-behaved client never sends more concurrent requests than the service has told it it can handle. The semaphore is the mechanism; choosing its size is a capacity-planning decision about which resource you are protecting.

### Aggregating Errors Versus Failing Fast

When a bounded runner processes many items, two error policies are both legitimate and the choice is a design decision, not an accident. Fail-fast returns the first error and abandons the rest, which suits a pipeline where any failure invalidates the whole batch. Aggregate-all runs every item to completion and returns a combined error that names every failure, which suits a batch job — reconciling a thousand objects, validating a directory of files — where the operator needs the complete list of what went wrong, not just the first thing the scheduler happened to surface. Go 1.20 added `errors.Join`, which wraps any number of errors into a single value whose `Error()` lists them all and against which `errors.Is` matches every joined member, making aggregate-all a few lines rather than a custom multierror type. The bounded runner is the natural place to implement both policies, because it already sits at the join point where every task's outcome is collected.

## Common Mistakes

Acquiring the semaphore inside the launched goroutine instead of in the producer loop bounds nothing: the goroutine and its stack already exist before the token is requested, so the loop never blocks and the system saturates exactly as if there were no semaphore at all. The acquire belongs in the loop, before `go`.

Releasing the token without a deferred call leaks it on the panic path. A task that panics after acquiring but before an undeferred release never returns its token; enough such leaks and the channel stays full and every future acquire blocks forever, which presents as a hang with no error. The release must be in a `defer` so it survives the panic.

Letting a single weighted task exceed the total capacity is a guaranteed deadlock: the producer can never gather more tokens than the buffer holds, so it waits forever for a slot that will never open. Clamp each weight to the capacity.

Observing the in-flight peak with a non-atomic counter is a data race that the race detector will catch and a quiet machine will not. The high-water mark must be an `atomic.Int64` updated with a compare-and-swap loop, and the test that asserts on it must run under `-race`.

Sizing the bound by guesswork rather than by the bottleneck resource either wastes capacity (too low) or overruns the thing you were trying to protect (too high). The limit should be derived from cores for CPU work, from connection budgets for I/O, and from the downstream's stated concurrency for fan-out.
