# 3. AST Representation and Manipulation — Concepts

The abstract syntax tree is the pivot point of an interpreter. Every phase after parsing either reads the tree (the evaluator, a type checker, a pretty printer) or rewrites it (an optimizer, a desugarer). A poorly shaped AST makes each of those phases fight the data structure; a well-shaped one makes them nearly trivial. This file is the conceptual foundation for the Monkey language's tree: how the node types form a closed hierarchy through marker methods, how every node carries a source position and a canonical string form, how traversal works through Go's idiomatic visitor pattern (the same one `go/ast.Walk` uses), and how the three manipulation primitives — `Clone`, `Equal`, and `Transform` — let you copy, compare, and optimize a tree without surprises. Read it once and you have everything you need to reason through the exercises, which build the tree piece by piece as independent, self-contained Go modules.

## Concepts

### Nodes as a Closed Interface Hierarchy

Go has no sum types — no `enum` of node variants the way Rust or OCaml would express an AST. The idiomatic substitute is a small set of interfaces tied together by *marker methods*. The base interface is `Node`: every node in the tree implements it, and it carries the three operations that are meaningful for any node at all — `TokenLiteral() string` (the literal text of the token that began the node, useful for debugging), `String() string` (a canonical textual form, discussed below), and `Pos() Position` (the source span). On top of `Node` sit two narrower interfaces. `Statement` embeds `Node` and adds an unexported method `statementNode()`; `Expression` embeds `Node` and adds an unexported method `expressionNode()`. Those two extra methods do nothing — their bodies are empty — and they exist purely to partition the node types into two roles.

The marker method is what makes the hierarchy *closed*. Because `statementNode()` and `expressionNode()` are unexported, only types declared in the same package can implement them, so no caller outside the package can fabricate a rogue type that satisfies `Statement` or `Expression`. Inside the package, a concrete type opts into exactly the role it should play: `IntegerLiteral` implements `expressionNode()` and is therefore an `Expression`; `LetStatement` implements `statementNode()` and is therefore a `Statement`. This is precisely the technique the standard library uses in `go/ast`, where `ast.Node`, `ast.Stmt`, and `ast.Expr` are knit together by the same unexported markers. The payoff is a compile-time guarantee: a function that takes a `Statement` cannot be handed an expression, and the set of things that can ever be a statement is fixed and auditable.

One node deliberately straddles the line. `BlockStatement` — the `{ ... }` body of an `if` or a function — implements `statementNode()`, because a block appears in statement position. It is *not* an expression. The fields that hold blocks (`IfExpression.Consequence`, `FunctionLiteral.Body`) are typed as `*BlockStatement` directly rather than as the `Expression` interface, so there is never any ambiguity about where a block may appear. The root of every tree, `Program`, is a special case: it is a `Node` (it has `String`, `TokenLiteral`, and `Pos`) but implements neither marker, because a whole program is neither a single statement nor a single expression — it is a slice of statements.

### The Canonical String Form

Every node implements `String()`, which renders the subtree rooted at that node back into source-like text. This is not a faithful pretty printer and it is explicitly not guaranteed to re-parse to the same tree; its job is to give a deterministic, readable representation for debugging, for golden-file tests, and for asserting that the parser built the structure you expected. The convention that makes it useful is *explicit parenthesization of every compound expression*. An `InfixExpression` renders as `(left op right)` — always parenthesized, always with single spaces around the operator — so `2 + 3 * 4` prints as `(2 + (3 * 4))` and the precedence the parser chose is visible on the page. A `PrefixExpression` renders as `(op right)`, e.g. `(-5)` or `(!flag)`. Literals render as their value (`42`, `true`, `"hi"`), an `Identifier` as its name, and a `LetStatement` as `let x = <value>;`. Because the form is fully parenthesized and whitespace is fixed, a single string comparison in a test pins down the entire shape of a subtree, which is why parser tests across the Monkey ecosystem assert on `node.String()` rather than walking fields by hand.

### Position Tracking on Every Node

Good error messages need to point at source. The `Position` type records a complete span — a start line and column, an end line and column, and an optional file name — and every concrete node stores one in an unexported `pos` field, returned through `Pos()`. Keeping the field unexported is a deliberate choice: it forces construction to go through a constructor (`NewIntegerLiteral`, `NewIdentifier`, and so on) that takes the position as an argument, which prevents the easy mistake of building a node with a struct literal and forgetting to set its position. `Position` also implements `String()` so a diagnostic can interpolate it directly, printing `file:line:column` when a file name is present and `line:column` when it is not.

The cost of this design is one small struct per node — a handful of integers, on the order of forty bytes on a 64-bit machine. The payoff is that any downstream phase can emit a precise "line 7, column 12" diagnostic with no extra bookkeeping, because the location traveled with the node from the moment the parser created it. That trade — a few bytes per node for free diagnostics everywhere — is one that every production compiler makes.

### The Visitor Pattern in Go

A real tree needs traversal: count nodes, find every identifier, check an invariant on every subtree. The classic object-oriented visitor uses method overloading — a `visit(IntegerLiteral)`, a `visit(InfixExpression)`, one overload per concrete type — and dynamic dispatch to pick the right one. Go has neither overloading nor virtual dispatch, so it uses a different shape, the same one `go/ast` settled on. A `Visitor` is any type with a single method, `Visit(node Node) Visitor`. A free function `Walk(v Visitor, node Node)` drives the traversal: it calls `v.Visit(node)` on the current node *before* descending, then type-switches on the node's concrete type to recurse into each child.

The return value of `Visit` is the control knob. If `Visit` returns `nil`, `Walk` prunes the subtree — the node's children are not visited. If it returns a non-nil `Visitor` (almost always the receiver itself), traversal continues into the children using that returned visitor. Returning a *different* visitor for a subtree is what lets a traversal carry per-subtree state without closures: a visitor entering a function body can hand the children a child visitor that knows it is "inside a function." This pre-order, prune-on-nil contract is exactly `go/ast.Walk`'s, and adopting it means anyone who has used the standard library's AST tools already knows how this one behaves.

Two conveniences sit on top of `Walk`. `Inspect(node, func(Node) bool)` wraps a plain function as a visitor — the function returns `true` to descend and `false` to prune — matching the signature of `go/ast.Inspect`, which is the ninety-percent case where you do not need stateful visitor objects. And a generic `Collect[T Node](root) []T` uses `Inspect` to gather every node whose concrete type is exactly `T`: `Collect[*Identifier](prog)` returns all the identifiers in pre-order. Generics make this possible because the type parameter `T` constrained to `Node` lets the type assertion `n.(T)` be checked against the caller's chosen type at each node.

### The Visit(nil) Post-Visit Nuance

There is one deliberate simplification worth calling out because it differs from the standard library. The real `go/ast.Walk` does more than a pre-order visit: after it finishes walking a node's children, it calls `walk(v, nil)` — that is, it invokes the visitor one more time with a `nil` node to signal "this subtree is now complete." That trailing `Visit(nil)` is what enables genuine *post-order* hooks: a visitor that wants to act after a subtree is done (closing a scope it opened on the way in, decrementing a depth counter, printing a closing bracket) keys off the `nil` call. The `Walk` built in these exercises omits that post-visit call to keep the traversal easy to follow — it is pure pre-order. If you later need post-order behavior or depth tracking, the extension is small and mechanical: after the type-switch recurses into the children, call `v.Visit(nil)` once before returning, and have visitors treat a `nil` argument as the "leaving" signal. Knowing the omission is there means you will not be surprised when you read the standard library's source and find a `Visit(nil)` your own `Walk` never makes.

### Clone, Equal, and Transform

Three operations turn a static tree into something you can manipulate safely. `Clone(node)` produces a fully independent deep copy: every node and every slice is freshly allocated, so a mutation anywhere in the copy is invisible to the original. Cloning is the prerequisite for any pass that wants to rewrite a tree while keeping the input intact — you clone first, then mutate the clone. The implementation is a type-switch that reconstructs each node from its fields and recurses into children, and it ends in a `default` that panics on an unknown node type, so adding a new node without teaching `Clone` about it fails loudly in tests rather than silently dropping a subtree.

`Equal(a, b)` reports structural equivalence, and its whole reason to exist is that `reflect.DeepEqual` is *wrong* for this job. `reflect.DeepEqual` compares every field, including the `pos` field — so two trees parsed from identical source at different offsets, or an original tree and a clone whose synthesized nodes carry different positions, would compare unequal even though they mean the same thing. `Equal` is a hand-written recursive comparator that ignores `pos` entirely and compares only the semantically meaningful fields: node type, operator strings, literal values, names, and (recursively) children. Two integer literals with value 5 are equal regardless of where in the file each was written. This is the comparison you want in tests ("did the optimizer produce the tree I expected?") and the one `reflect.DeepEqual` cannot give you.

`Transform(node, fn)` rewrites a tree *bottom-up*. It recurses into a node's children first, replacing each child slot with the result of transforming it, and only then calls `fn` on the (now already-transformed) parent. If `fn` returns the same node, nothing changes; if it returns a new node, the parent's slot is updated to point at the replacement. Bottom-up order is not an implementation detail — it is what makes multi-level optimization work in a single pass. Consider constant folding `(2 + 3) * 4`: the transform descends to the inner `2 + 3`, folds it to the literal `5`, returns up to the `* 4` node whose left child is *now* the literal `5`, and folds `5 * 4` to `20`. Because children are always folded before their parent is examined, the parent sees literals where it used to see expressions, and one traversal collapses the whole chain. `Transform` mutates in place, so a caller that must preserve the original writes the canonical pairing `Transform(Clone(prog), ConstantFold())` — clone to get an independent copy, transform the copy.

`ConstantFold` is the example pass these primitives are built to support. It is a function suitable for `Transform` that folds integer arithmetic (`+`, `-`, `*`, `/`) and boolean operations (`&&`, `||`) whose operands are already literals. It is careful about one thing: division by zero is left *un-folded*, returned intact so the runtime can raise the error it must — folding it at compile time would either crash the compiler or invent a wrong answer. That restraint is the general rule for any constant-folding pass: fold only operations that cannot fail, and leave anything that can fault for the evaluator to handle with full runtime context.

## Common Mistakes

### Using reflect.DeepEqual for Structural Equality

It is tempting to write `reflect.DeepEqual(a, b)` and be done. It returns the wrong answer whenever the two nodes carry different `pos` fields, even when they represent the identical expression. Parse the same source twice and the positions match, so the bug hides; clone a tree and run an optimization pass, and the synthesized nodes get different positions, so the comparison that worked yesterday fails today. The fix is a manual recursive `Equal` that skips `pos` and compares only the semantic fields. Writing it by hand also documents, explicitly and in one place, exactly which fields constitute "sameness" for your tree — which is information `reflect.DeepEqual` can never express.

### Forgetting to Clone Before Transform

Writing `folded := Transform(prog, ConstantFold())` mutates `prog` in place. After the call `prog.String()` returns the folded tree, and any pointer to a sub-node saved before the transform may now point at a node that was replaced or rewritten. The original is gone. The fix is to clone first: `folded := Transform(Clone(prog), ConstantFold()).(*Program)`. `Clone` produces an independent copy and `Transform` then mutates only that copy, leaving every earlier reference to the original valid.

### Incomplete Walk Switch Causes Silent Skips

Add a new node type — say an `ArrayLiteral` — and forget to add a case for it in `Walk`'s type-switch, and nothing complains. The node is visited (the pre-order `Visit` still fires) but its children are silently never traversed, so a "find all identifiers" pass quietly misses the ones inside the array. The fix is a `default: panic(...)` arm in `Walk` that fires on any unhandled node type. The panic surfaces immediately the first time a test walks a tree containing the new node, turning a silent correctness bug into a loud, located failure.

### Marker Method Confusion

Implementing the wrong marker — for example giving a `BreakStatement` an `expressionNode()` method — compiles cleanly, because Go is happy to let a type satisfy any interface whose methods it has. But the node now satisfies `Expression` and can be slotted anywhere an expression is expected, including places the evaluator has no case for, producing a confusing runtime failure far from the declaration. The fix is discipline at the point of declaration: implement only the marker that matches the node's role. Statements implement `statementNode()`; expressions implement `expressionNode()`. `BlockStatement` implements `statementNode()` precisely because blocks live in statement position, and the fields that hold them are typed `*BlockStatement` rather than `Expression` so the type system enforces the distinction for you.
