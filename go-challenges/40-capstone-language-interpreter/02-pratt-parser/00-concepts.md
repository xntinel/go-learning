# 2. Pratt Parser — Concepts

A Pratt parser — top-down operator precedence parsing, the technique Vaughan Pratt published in 1973 — is the most economical way to turn a stream of tokens into a correctly nested expression tree. The thing it has to get right is precedence and associativity: `1 + 2 * 3` must become `1 + (2 * 3)` and never `(1 + 2) * 3`, while `a = b = c` must associate to the right as `a = (b = c)`. A traditional recursive-descent parser bakes those rules into the shape of the call graph, one function per precedence level. Pratt parsing instead encodes them as small integers — binding powers — attached to operator tokens, and drives the whole grammar from a single loop. This file is the conceptual foundation: read it once and the two exercises that follow, which build the algorithm and then its error-recovery layer as independent self-contained Go modules, will read as variations on the ideas here.

## Concepts

### Binding Power Replaces Grammar Rules

A traditional recursive-descent expression parser encodes precedence in the call stack by nesting functions: `parseAdditive` calls `parseMultiplicative`, which calls `parseUnary`, which finally reaches a primary. Each layer exists only to say "this operator binds more tightly than the one above it." Adding a new precedence level means inserting a new function into the chain and rewiring its neighbours, and the structure of the code stops resembling the structure of the grammar.

Pratt parsing throws that ladder away. Precedence becomes a number — a binding power — associated with each operator token. Higher numbers bind tighter: multiplication's binding power is greater than addition's, exponentiation's is greater still. The core of the parser is a single function, conventionally `parseExpression(minBP)`, that parses a left-hand operand and then loops, consuming infix operators for as long as the next operator's binding power is strictly greater than `minBP`. When it meets an operator weaker than or equal to `minBP`, it stops and returns, leaving that operator for an enclosing call to handle.

The parameter `minBP` is the entire control mechanism. Calling `parseExpression(0)` — the lowest binding power, never assigned to any operator — means "consume the whole expression; nothing is too weak to stop here." Passing a higher value means "stop as soon as you reach an operator weaker than this." Every recursive call inside the parser is just a `parseExpression` invoked with a `minBP` chosen to capture exactly the right-hand operand and no more. Precedence is no longer a property of where a function sits in a call chain; it is a number compared against a threshold, and the comparison happens in one place.

### Prefix and Infix Parse Functions

The second idea is that each token type is associated with up to two small functions, distinguished by where the token appears relative to the expression it participates in.

A prefix parse function knows how to start an expression with a given token. An integer token's prefix function builds an integer literal; an identifier's builds a name reference; a `-` or `!` token's prefix function parses a unary operator applied to the operand that follows; a `(` token's prefix function parses a parenthesized group; an `if` or `fn` keyword's prefix function parses an entire conditional or function literal. The prefix function is what `parseExpression` calls first, before its loop, to obtain the initial left-hand side. If the current token has no prefix function, the input is malformed — there is no way to begin an expression with, say, a `+` — and that is exactly where the parser reports "unexpected token."

An infix parse function knows how to extend an already-parsed left-hand expression when its token appears in the middle. It receives that left side as an argument and returns a larger expression with the left side embedded in it. The `+` token's infix function takes the left operand, parses a right operand, and returns a binary node. The crucial and easily missed point is that the same token can carry both kinds of function. A `(` is a prefix function (a parenthesized group, `(a + b)`) and also an infix function (a call, `f(a, b)` — the `(` appears just after an already-parsed callee). A `[` is a prefix function (an array literal, `[1, 2]`) and an infix function (an index, `a[0]`). The parser decides which role applies purely by position: if `(` shows up where an expression must begin, it groups; if it shows up after a complete expression, it calls. The binding-power loop and the two function tables are all the machinery needed to make that distinction fall out automatically.

### Associativity Is rbp = lbp ± 1

Associativity is the subtlest part, and it reduces to a single arithmetic choice. Every infix operator has a left binding power — the value the outer loop compares against `minBP` to decide whether to consume the operator at all. When an infix function then recurses to parse its right-hand operand, it passes a right binding power, and the relationship between that right binding power and the operator's own left binding power decides the direction of association.

For a left-associative operator the right binding power is set equal to the left binding power, which is the same as saying `rbp = lbp` (equivalently `lbp + 0`). Consider `a + b + c`. The first `+` is consumed, and its infix function recurses with `minBP` equal to `+`'s own binding power. Inside that recursive call the loop sees the second `+`, whose binding power is not strictly greater than the threshold (it is equal), so the recursion stops and returns just `b`. The second `+` is left for the outer loop, which attaches it to the already-built `a + b`. The result nests to the left: `(a + b) + c`. Making the right-hand recursion use a threshold equal to the operator's own power is precisely what refuses the second same-level operator and forces left association.

For a right-associative operator the right binding power is set one below the left binding power: `rbp = lbp - 1`. Consider `a ** b ** c` with `**` right-associative. The first `**` is consumed and its infix function recurses with `minBP = lbp - 1`. Now inside the recursion the loop sees the second `**`, whose binding power is strictly greater than `lbp - 1`, so it is consumed by the recursion rather than left for the outer loop. The right operand of the first `**` therefore becomes the whole `b ** c`, and the tree nests to the right: `a ** (b ** c)`. Assignment behaves the same way, which is why `a = b = c` parses as `a = (b = c)`. The mechanism is not "give the right side more power"; it is the opposite — lowering the threshold by one lets a peer operator slip into the recursive call instead of being held back. That single `± 1` is the whole of associativity. (The `lbp ± 1` convention also leaves room for a third state: a `lbp - 2` style gap, or distinct left and right powers per operator, is how some parsers encode non-associative operators that must reject `a < b < c` outright.)

### Two-Token Lookahead

The parser needs to see a little of the future to make its decisions: the loop in `parseExpression` must inspect the operator that comes after the current operand before deciding whether to consume it, and statement parsing must peek at the token after a keyword to tell `let x = ...` from a malformed `let = ...`. The standard arrangement is two fields, `cur` (the token under the cursor) and `peek` (the one after it), advanced in lockstep by a single `advance` method that copies `peek` into `cur` and pulls a fresh token from the lexer into `peek`. One token of lookahead — `peek` — is enough for this grammar; the parser never needs to look two tokens ahead.

The detail that bites is initialization. Both `cur` and `peek` must be primed before parsing begins, which means `advance` has to be called twice in the constructor: the first call loads the real first token into `peek` (leaving `cur` at its zero value), and the second shifts it into `cur` and loads the second token into `peek`. Prime only once and `cur` is left holding a zero-value token whose type is the enum's zero — typically `EOF` — so the parser believes the input is empty and the loop never runs. Priming the pair is a one-line ritual that is invisible when correct and total when wrong.

## Common Mistakes

### Confusing the Direction of the Associativity Trick

A developer who wants `**` to be right-associative reasons that the right operand should bind "more tightly," and tries to express that by giving the recursive call a higher threshold than the operator's own binding power. This does not work, and it is the single most common Pratt bug. A higher threshold makes the recursion stop sooner, which is left association, the opposite of what was intended. The mechanism is to lower the threshold: recurse with `lbp - 1` so that a second same-level operator is strictly greater than the threshold and is therefore swallowed by the recursion. For a left-associative operator, recurse with the operator's own `lbp`; for a right-associative one, recurse with `lbp - 1`. Nothing else about the loop changes.

### Forgetting to Advance Before Recursing in an Infix Function

An infix parse function is called while the cursor is still sitting on the operator token. If it recurses into `parseExpression` without first advancing past that operator, the recursion starts with the operator as its current token and looks for a prefix function for, say, `+`. There is none, so the parser reports "unexpected token" on a perfectly valid input. The fix is the discipline that the infix function always advances off the operator before parsing the right-hand operand: capture whatever you need from the operator token first (its literal, its binding power), then call `advance`, then recurse. Skipping the advance turns every binary expression into a spurious syntax error.

### Treating a Bare `{` as a Block Instead of a Hash Literal

In a language where `{` introduces both a statement block (the body of `if` or `fn`) and a hash literal (`{"a": 1}`), the tempting shortcut is a `parseStatement` case for `{` that calls the block parser directly. That case misfires whenever a hash literal appears at the start of a statement: the `:` between key and value has no meaning inside a block, and the parser emits a cascade of "unexpected token" errors. The resolution is to let context decide. The block parser is called only from constructs that explicitly introduce a block — the bodies of `if` and `fn`. At the statement level a leading `{` falls through to ordinary expression parsing, whose prefix function for `{` builds a hash literal. Block and hash are syntactically identical only up to the first element; the surrounding construct, not the brace itself, determines which one is correct.

### Not Priming the Two-Token Lookahead

The parser maintains `cur` and `peek`, and both must be loaded before the first parse step. Call `advance` only once in the constructor and `cur` keeps its zero value — usually the `EOF` token, since `EOF` is the first constant in the token enum — while only `peek` holds real data. The first loop condition then sees `cur.Type == EOF`, concludes the input is empty, and returns an empty program without an error. The bug is silent: well-formed input simply produces nothing. Call `advance` twice in the constructor so that `cur` holds the first real token and `peek` holds the second before parsing starts.
