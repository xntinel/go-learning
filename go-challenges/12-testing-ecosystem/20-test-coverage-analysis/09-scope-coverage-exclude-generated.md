# Exercise 9: Scope coverage to business logic and exclude generated/wiring code

The reported coverage percentage is a fraction, and both parts are yours to
control. Generated stubs, `main()` wiring, and vendored code land in the
denominator under a naive `./...` and distort the signal — pushing you to write
low-value tests that hit generated setters just to move the number. This module
builds a module with a business package, a generated package, and a wiring
`main`, then scopes `-coverpkg` to the logic so the number reflects code that
*should* be unit-tested.

This module is fully self-contained: its own `go mod init`, three packages, and
tests on the business logic only.

## What you'll build

```text
billing/                   independent module: example.com/billing
  go.mod
  invoice/
    invoice.go             business logic: Compute; the code under test
    invoice_test.go        table tests over Compute
  pb/
    invoice.pb.go          "generated" code (DO NOT EDIT); no tests
  cmd/
    app/
      main.go              wiring: build a request, call Compute, print
```

- Files: `invoice/invoice.go`, `invoice/invoice_test.go`, `pb/invoice.pb.go`, `cmd/app/main.go`.
- Implement: `invoice.Compute` (the logic), a generated `pb` package (marked `// Code generated ... DO NOT EDIT.`, untested by design), and a `main` that wires them.
- Test: table tests on `invoice.Compute` only; the generated and wiring code carries no unit tests.
- Verify: `go test -coverpkg=./invoice ./...` versus a naive `go test -cover ./...`

### Why the naive number is the wrong denominator

Run `go test -cover ./...` over this module and three packages report:
`invoice` at its real logic coverage, `pb` at 0% (generated code, no tests), and
`cmd/app` at 0% (a `main` that is exercised by running the program, not by unit
tests). The headline is dragged down by two packages that *should not* be
unit-tested — and the wrong reaction is to write tests that call the generated
getters and the `main` wiring to inflate it. That produces low-value tests and a
number that measures test-writing effort, not logic coverage.

The right move is to scope the denominator. Coverage should measure the packages
that carry business logic, so you point `-coverpkg` at exactly those:
`-coverpkg=./invoice`. Now the percentage answers the question that matters — how
much of the *logic* is exercised — without penalizing generated stubs or wiring
that has no business being unit-tested. In a larger repo you compute the target
set programmatically: `go list ./... | grep -v /pb | grep -v /cmd/`, then pass that
list to `-coverpkg`. Generated code is conventionally recognizable by its header
comment `// Code generated ... DO NOT EDIT.`, which is also what tools like
`golangci-lint` use to skip it.

Create `invoice/invoice.go` — the code that should be covered:

```go
package invoice

import (
	"errors"
	"fmt"
)

// ErrNoLines is returned by Compute for an invoice with no line items.
var ErrNoLines = errors.New("invoice has no lines")

// Line is a single billed item: unit price in cents and quantity.
type Line struct {
	UnitCents int64
	Qty       int64
}

// Compute returns the invoice total in cents after an optional whole-invoice
// discount in basis points (100 bps = 1%). It returns ErrNoLines for an empty
// invoice and rejects a discount over 100%.
func Compute(lines []Line, discountBps int64) (int64, error) {
	if len(lines) == 0 {
		return 0, ErrNoLines
	}
	if discountBps < 0 || discountBps > 10000 {
		return 0, fmt.Errorf("discount %d bps out of range", discountBps)
	}
	var subtotal int64
	for _, l := range lines {
		subtotal += l.UnitCents * l.Qty
	}
	discount := subtotal * discountBps / 10000
	return subtotal - discount, nil
}
```

The `// Code generated` header is the real convention; this stub stands in for
protobuf output you would never hand-write or unit-test.

Create `pb/invoice.pb.go`:

```go
// Code generated by protoc-gen-go. DO NOT EDIT.

package pb

// InvoiceRequest is a generated message type.
type InvoiceRequest struct {
	Lines       []*LineItem
	DiscountBps int64
}

// LineItem is a generated message type.
type LineItem struct {
	UnitCents int64
	Qty       int64
}

// GetDiscountBps returns the discount, or 0 if the request is nil (generated
// accessor style).
func (r *InvoiceRequest) GetDiscountBps() int64 {
	if r == nil {
		return 0
	}
	return r.DiscountBps
}

// GetLines returns the line items, or nil if the request is nil.
func (r *InvoiceRequest) GetLines() []*LineItem {
	if r == nil {
		return nil
	}
	return r.Lines
}
```

Create `cmd/app/main.go` — the wiring that adapts the generated request into the
business call. This is exercised by running the app, not by unit tests:

```go
package main

import (
	"fmt"
	"log"

	"example.com/billing/invoice"
	"example.com/billing/pb"
)

func main() {
	req := &pb.InvoiceRequest{
		Lines: []*pb.LineItem{
			{UnitCents: 1500, Qty: 2},
			{UnitCents: 500, Qty: 3},
		},
		DiscountBps: 1000, // 10%
	}

	lines := make([]invoice.Line, 0, len(req.GetLines()))
	for _, li := range req.GetLines() {
		lines = append(lines, invoice.Line{UnitCents: li.UnitCents, Qty: li.Qty})
	}

	total, err := invoice.Compute(lines, req.GetDiscountBps())
	if err != nil {
		log.Fatal(err)
	}
	fmt.Printf("invoice total: %d cents\n", total)
}
```

Run it:

```bash
go run ./cmd/app
```

Expected output:

```
invoice total: 4050 cents
```

(Subtotal: 1500x2 + 500x3 = 4500; 10% discount = 450; total 4050.)

### The tests cover only the logic

Create `invoice/invoice_test.go`:

```go
package invoice

import (
	"errors"
	"testing"
)

func TestCompute(t *testing.T) {
	t.Parallel()
	tests := []struct {
		name        string
		lines       []Line
		discountBps int64
		want        int64
		wantErr     error
	}{
		{
			name:        "no-discount",
			lines:       []Line{{UnitCents: 1500, Qty: 2}, {UnitCents: 500, Qty: 3}},
			discountBps: 0,
			want:        4500,
		},
		{
			name:        "ten-percent",
			lines:       []Line{{UnitCents: 1500, Qty: 2}, {UnitCents: 500, Qty: 3}},
			discountBps: 1000,
			want:        4050,
		},
		{
			name:        "empty",
			lines:       nil,
			discountBps: 0,
			wantErr:     ErrNoLines,
		},
	}
	for _, tt := range tests {
		t.Run(tt.name, func(t *testing.T) {
			t.Parallel()
			got, err := Compute(tt.lines, tt.discountBps)
			if !errors.Is(err, tt.wantErr) {
				t.Fatalf("Compute err = %v, want %v", err, tt.wantErr)
			}
			if tt.wantErr == nil && got != tt.want {
				t.Errorf("Compute = %d, want %d", got, tt.want)
			}
		})
	}
}

func TestComputeDiscountRange(t *testing.T) {
	t.Parallel()
	if _, err := Compute([]Line{{UnitCents: 100, Qty: 1}}, 20000); err == nil {
		t.Fatal("Compute accepted a 200% discount; want an out-of-range error")
	}
}
```

### Comparing the naive and scoped numbers

The naive measurement counts all three packages, so the generated and wiring code
drag the total down:

```bash
go test -cover ./...
```

Expected output (shape) — `invoice` covered, the other two not:

```
ok      example.com/billing/invoice   0.00s  coverage: 100.0% of statements
?       example.com/billing/pb        [no test files]
?       example.com/billing/cmd/app   [no test files]
```

Scope the measured set to the business package. Now the number answers "how much
of the logic is covered" without the generated/wiring denominator:

```bash
go test -coverpkg=./invoice -coverprofile=logic.out ./...
go tool cover -func=logic.out | tail -1
```

Expected output:

```
total:  (statements)   100.0%
```

For a larger module, build the target list programmatically and exclude generated
and wiring packages by convention:

```bash
pkgs=$(go list ./... | grep -v '/pb$' | grep -v '/cmd/')
go test -coverpkg="$(echo "$pkgs" | paste -sd, -)" ./...
```

This keeps the denominator honest: generated stubs (recognizable by their
`DO NOT EDIT` header) and `main` wiring stay out of the number, so a low logic
percentage is a real signal to write more logic tests, not an artifact of counting
code that should not be unit-tested.

## Review

`invoice.Compute` is correct when it sums line items, applies a basis-point
discount, rejects an empty invoice with `ErrNoLines`, and rejects an out-of-range
discount — pinned by the table and the range test. The scoping lesson is correct
when `go test -cover ./...` reports a headline dragged down by the untested `pb`
and `cmd/app` packages while `go test -coverpkg=./invoice` reports the logic
coverage cleanly.

The mistake to avoid is chasing `./...` to 100% by writing tests that call
generated getters and `main` wiring — low-value tests that measure effort, not
logic. Scope `-coverpkg` (or a filtered `go list`) to the packages that carry
business logic, recognize generated code by its `DO NOT EDIT` header, and let the
number reflect the code a unit test should actually cover. Run `go test -race
./...` to confirm the logic package is clean.
