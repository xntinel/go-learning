# Exercise 5: gomock Ordering, Matchers, and DoAndReturn — Transactional Outbox Flush

Some contracts are about *order*, not just outcome. A transactional outbox flusher
must, for each pending message, `Publish` to the broker and only then
`MarkPublished` in the store — publish-before-mark, per message — and it must stop
on the first publish failure so a message is never marked without being delivered.
That ordering is the correctness property, and gomock's `InOrder`, matchers, and
`DoAndReturn` are exactly the tools to pin it.

This module is fully self-contained: its own `go mod init`, all code inline, its
own demo and tests. It pulls one external dependency, `go.uber.org/mock`.

### The flusher and why ordering is the contract

The outbox pattern exists to make "commit a row and publish an event" atomic: the
event is written to an outbox table in the same transaction as the business change,
and a separate flusher later reads pending rows, publishes each to the broker, and
marks it published. The invariant that makes it safe is *publish-before-mark*: if
you marked first and the publish failed, the event would be lost forever; by
publishing first and marking only on success, a crash between the two simply
re-delivers the message later (at-least-once). So the order of the two calls, per
message, is the correctness property — not an incidental detail. This is one of the
few genuine cases for interaction-and-ordering verification.

`Flush` also stops on the first `Publish` error rather than plowing ahead. If the
broker is down, continuing would mark nothing (good) but also waste attempts and
possibly reorder delivery; stopping returns the count published so far and the
error, letting the caller retry the whole flush later.

Create `outbox/outbox.go`:

```go
package outbox

import "context"

//go:generate mockgen -destination=mocks/mock_outbox.go -package=mocks example.com/gomockoutbox/outbox Broker,Store

// Message is one pending outbox row.
type Message struct {
	ID   string
	Body string
}

// Broker publishes messages to the outside world.
type Broker interface {
	Publish(ctx context.Context, m Message) error
}

// Store holds pending outbox rows and records which were published.
type Store interface {
	Pending(ctx context.Context) ([]Message, error)
	MarkPublished(ctx context.Context, id string) error
}

// Flusher drains the outbox: publish each pending message, then mark it.
type Flusher struct {
	store  Store
	broker Broker
}

// NewFlusher injects both ports through the constructor.
func NewFlusher(store Store, broker Broker) *Flusher {
	return &Flusher{store: store, broker: broker}
}

// Flush publishes each pending message and marks it published, in that order,
// stopping on the first Publish failure. It returns the number marked published.
func (f *Flusher) Flush(ctx context.Context) (int, error) {
	msgs, err := f.store.Pending(ctx)
	if err != nil {
		return 0, err
	}
	published := 0
	for _, m := range msgs {
		if err := f.broker.Publish(ctx, m); err != nil {
			return published, err // stop: never mark an unpublished message
		}
		if err := f.store.MarkPublished(ctx, m.ID); err != nil {
			return published, err
		}
		published++
	}
	return published, nil
}
```

### The runnable demo

Create `cmd/demo/main.go`:

```go
package main

import (
	"context"
	"fmt"

	"example.com/gomockoutbox/outbox"
)

// memStore and stdoutBroker are real (minimal) implementations for the demo.
type memStore struct {
	pending   []outbox.Message
	published []string
}

func (s *memStore) Pending(context.Context) ([]outbox.Message, error) {
	return s.pending, nil
}
func (s *memStore) MarkPublished(_ context.Context, id string) error {
	s.published = append(s.published, id)
	return nil
}

type stdoutBroker struct{}

func (stdoutBroker) Publish(_ context.Context, m outbox.Message) error {
	fmt.Printf("published %s: %s\n", m.ID, m.Body)
	return nil
}

func main() {
	store := &memStore{pending: []outbox.Message{
		{ID: "m1", Body: "order.created"},
		{ID: "m2", Body: "order.paid"},
	}}
	n, err := outbox.NewFlusher(store, stdoutBroker{}).Flush(context.Background())
	if err != nil {
		fmt.Println("error:", err)
		return
	}
	fmt.Printf("flushed %d, marked %v\n", n, store.published)
}
```

go run ./cmd/demo
```

published m2: order.paid
flushed 2, marked [m1 m2]
```

### The tests

`TestFlushPublishesThenMarksInOrder` is the ordering contract. `gomock.InOrder`
takes the four expected calls and requires them in that exact sequence:
`Publish(m1)`, `MarkPublished(m1)`, `Publish(m2)`, `MarkPublished(m2)`. The `Publish`
expectations match the message by a field with `gomock.Cond` (a typed predicate),
the `MarkPublished` ids with `gomock.Eq`. If `Flush` ever marked before it
published, the sequence would break and gomock would fatal on the out-of-order call.

`TestFlushStopsOnPublishError` uses `DoAndReturn` to make the *second* `Publish`
return an error dynamically, and then sets *no* `MarkPublished` expectation for the
second message. Because gomock is strict, if `Flush` wrongly marked m2 after its
publish failed, that unexpected `MarkPublished` call would fatal — so the test
proves "stop on failure, never mark an unpublished message" without a manual
negative assertion. It checks the return: one published, and a non-nil error.

Create `outbox/outbox_test.go`:

```go
package outbox_test

import (
	"context"
	"fmt"
	"testing"

	"example.com/gomockoutbox/outbox"
	"example.com/gomockoutbox/outbox/mocks"
	"go.uber.org/mock/gomock"
)

// byID matches a Message by its ID field via a typed predicate.
func byID(id string) gomock.Matcher {
	return gomock.Cond(func(m outbox.Message) bool { return m.ID == id })
}

func TestFlushPublishesThenMarksInOrder(t *testing.T) {
	t.Parallel()
	ctrl := gomock.NewController(t)
	store := mocks.NewMockStore(ctrl)
	broker := mocks.NewMockBroker(ctrl)
	ctx := context.Background()

	m1 := outbox.Message{ID: "m1", Body: "first"}
	m2 := outbox.Message{ID: "m2", Body: "second"}
	store.EXPECT().Pending(ctx).Return([]outbox.Message{m1, m2}, nil)

	gomock.InOrder(
		broker.EXPECT().Publish(ctx, byID("m1")).Return(nil),
		store.EXPECT().MarkPublished(ctx, gomock.Eq("m1")).Return(nil),
		broker.EXPECT().Publish(ctx, byID("m2")).Return(nil),
		store.EXPECT().MarkPublished(ctx, gomock.Eq("m2")).Return(nil),
	)

	n, err := outbox.NewFlusher(store, broker).Flush(ctx)
	if err != nil {
		t.Fatalf("Flush: %v", err)
	}
	if n != 2 {
		t.Fatalf("published %d, want 2", n)
	}
}

func TestFlushStopsOnPublishError(t *testing.T) {
	t.Parallel()
	ctrl := gomock.NewController(t)
	store := mocks.NewMockStore(ctrl)
	broker := mocks.NewMockBroker(ctrl)
	ctx := context.Background()

	m1 := outbox.Message{ID: "m1", Body: "first"}
	m2 := outbox.Message{ID: "m2", Body: "second"}
	store.EXPECT().Pending(ctx).Return([]outbox.Message{m1, m2}, nil)

	gomock.InOrder(
		broker.EXPECT().Publish(ctx, gomock.Any()).Return(nil), // m1 ok
		store.EXPECT().MarkPublished(ctx, "m1").Return(nil),
		broker.EXPECT().Publish(ctx, gomock.Any()).DoAndReturn(
			func(_ context.Context, m outbox.Message) error {
				return fmt.Errorf("broker down for %s", m.ID)
			}),
	)
	// Deliberately NO MarkPublished("m2"): reaching it would fatal (strict mock).

	n, err := outbox.NewFlusher(store, broker).Flush(ctx)
	if err == nil {
		t.Fatal("expected a publish error")
	}
	if n != 1 {
		t.Fatalf("published %d, want 1 before the failure", n)
	}
}
```

## Review

The flusher is correct when it publishes before it marks, per message, and stops on
the first publish failure so no message is ever marked without being delivered.
gomock makes that ordering testable directly: `InOrder` fails the test if
`MarkPublished` ever precedes its `Publish`, and the strict mock turns "never mark
m2 after its publish failed" into an automatic failure by simply not expecting the
call. The matchers show the range of precision available — `Cond` for a typed
predicate on a struct field, `Eq` for an exact id, `Any` where the value does not
matter — and `DoAndReturn` lets a single expectation compute its result from the
actual argument, which is how the second publish is made to fail. The trap is
over-pinning: use `InOrder` only where order is genuinely the contract; pinning the
order of calls that could legitimately run in any sequence produces a brittle test.
Run `go test -race`.
