# Exercise 4: gomock Generated Mocks for an OrderRepository

Hand-rolled doubles stop scaling when the interface widens or when you want strict,
self-verifying expectations. `gomock` (`go.uber.org/mock`, the maintained successor
to the archived `golang/mock`) generates compile-time-safe mocks from an interface
and checks expectations strictly: an unexpected call fatals immediately, and an
unmet expectation fails at cleanup. This module wires a real generated mock into an
order service and leans on that strictness.

This module is fully self-contained: its own `go mod init`, all code inline, its
own demo and tests. It pulls one external dependency, `go.uber.org/mock`.

## What you'll build

```text
gomockorder/                 independent module: example.com/gomockorder
  go.mod                     go 1.26; requires go.uber.org/mock
  order/
    order.go                 Order; OrderRepository port; Service; Fulfill; ErrNotFound
    mocks/
      mock_repository.go     mockgen output: MockOrderRepository (strict, self-verifying)
    service_test.go          package order_test: strict-expectation and not-found tests
  cmd/
    demo/
      main.go                runnable demo over a real in-memory repo
```

- Files: `order/order.go`, `order/mocks/mock_repository.go`, `order/service_test.go`, `cmd/demo/main.go`.
- Implement: `Service.Fulfill(ctx, id)` that `FindByID`, mutates status to `fulfilled`, then `Save`; the mock is the generated `MockOrderRepository`.
- Test: program `FindByID`/`Save` expectations with `.Times(1)`; let auto-`Finish` verify; a not-found path that returns `ErrNotFound` and sets *no* `Save` expectation, so any `Save` call fatals.
- Verify: `go test -count=1 -race ./...`

### The port and the service

`OrderRepository` is a two-method port: `FindByID` and `Save`. The service's
`Fulfill` reads an order, flips its status to `fulfilled`, and writes it back — the
canonical "load, mutate, persist" shape. A `go:generate` directive above the
interface records how the mock is produced; running `go generate ./...` invokes
`mockgen` and writes `mocks/mock_repository.go`. (This lesson ships that generated
file inline so it builds without running the tool, but in a real repo you commit
the directive and the generated file, and regenerate when the interface changes.)

Create `order/order.go`:

```go
package order

import (
	"context"
	"errors"
)

//go:generate mockgen -destination=mocks/mock_repository.go -package=mocks example.com/gomockorder/order OrderRepository

// ErrNotFound is returned when no order has the given id.
var ErrNotFound = errors.New("order not found")

// Order is the aggregate the service fulfills.
type Order struct {
	ID     string
	Status string
}

// OrderRepository is the persistence port. Generated mocks target this interface.
type OrderRepository interface {
	FindByID(ctx context.Context, id string) (*Order, error)
	Save(ctx context.Context, o *Order) error
}

// Service fulfills orders.
type Service struct {
	repo OrderRepository
}

// NewService injects the repository through the constructor.
func NewService(repo OrderRepository) *Service {
	return &Service{repo: repo}
}

// Fulfill loads an order, marks it fulfilled, and persists it. A load failure is
// returned without ever calling Save.
func (s *Service) Fulfill(ctx context.Context, id string) (*Order, error) {
	o, err := s.repo.FindByID(ctx, id)
	if err != nil {
		return nil, err
	}
	o.Status = "fulfilled"
	if err := s.repo.Save(ctx, o); err != nil {
		return nil, err
	}
	return o, nil
}
```

### The generated mock

This is exactly what `mockgen` emits for the interface above: a `MockOrderRepository`
that routes each method call through `m.ctrl.Call` (which checks it against the
programmed expectations) and a recorder type whose methods (`EXPECT().FindByID(...)`)
register expectations. You do not edit it by hand; it is shown here so the module
builds standalone.

Create `order/mocks/mock_repository.go`:

```go
// Code generated by MockGen. DO NOT EDIT.
// Source: example.com/gomockorder/order (interfaces: OrderRepository)
package mocks

import (
	context "context"
	reflect "reflect"

	order "example.com/gomockorder/order"
	gomock "go.uber.org/mock/gomock"
)

// MockOrderRepository is a mock of OrderRepository interface.
type MockOrderRepository struct {
	ctrl     *gomock.Controller
	recorder *MockOrderRepositoryMockRecorder
	isgomock struct{}
}

// MockOrderRepositoryMockRecorder is the mock recorder for MockOrderRepository.
type MockOrderRepositoryMockRecorder struct {
	mock *MockOrderRepository
}

// NewMockOrderRepository creates a new mock instance.
func NewMockOrderRepository(ctrl *gomock.Controller) *MockOrderRepository {
	mock := &MockOrderRepository{ctrl: ctrl}
	mock.recorder = &MockOrderRepositoryMockRecorder{mock}
	return mock
}

// EXPECT returns an object that allows the caller to indicate expected use.
func (m *MockOrderRepository) EXPECT() *MockOrderRepositoryMockRecorder {
	return m.recorder
}

// FindByID mocks base method.
func (m *MockOrderRepository) FindByID(ctx context.Context, id string) (*order.Order, error) {
	m.ctrl.T.Helper()
	ret := m.ctrl.Call(m, "FindByID", ctx, id)
	ret0, _ := ret[0].(*order.Order)
	ret1, _ := ret[1].(error)
	return ret0, ret1
}

// FindByID indicates an expected call of FindByID.
func (mr *MockOrderRepositoryMockRecorder) FindByID(ctx, id any) *gomock.Call {
	mr.mock.ctrl.T.Helper()
	return mr.mock.ctrl.RecordCallWithMethodType(mr.mock, "FindByID", reflect.TypeOf((*MockOrderRepository)(nil).FindByID), ctx, id)
}

// Save mocks base method.
func (m *MockOrderRepository) Save(ctx context.Context, o *order.Order) error {
	m.ctrl.T.Helper()
	ret := m.ctrl.Call(m, "Save", ctx, o)
	ret0, _ := ret[0].(error)
	return ret0
}

// Save indicates an expected call of Save.
func (mr *MockOrderRepositoryMockRecorder) Save(ctx, o any) *gomock.Call {
	mr.mock.ctrl.T.Helper()
	return mr.mock.ctrl.RecordCallWithMethodType(mr.mock, "Save", reflect.TypeOf((*MockOrderRepository)(nil).Save), ctx, o)
}
```

### The runnable demo

The demo uses a real in-memory repository — production code never imports the mock,
which is the whole reason the mock lives in its own `mocks` package rather than in
`package order`.

Create `cmd/demo/main.go`:

```go
package main

import (
	"context"
	"fmt"

	"example.com/gomockorder/order"
)

// memRepo is a real in-memory OrderRepository for the demo.
type memRepo struct {
	orders map[string]*order.Order
}

func (r *memRepo) FindByID(_ context.Context, id string) (*order.Order, error) {
	o, ok := r.orders[id]
	if !ok {
		return nil, order.ErrNotFound
	}
	return o, nil
}

func (r *memRepo) Save(_ context.Context, o *order.Order) error {
	r.orders[o.ID] = o
	return nil
}

func main() {
	repo := &memRepo{orders: map[string]*order.Order{
		"o-1": {ID: "o-1", Status: "pending"},
	}}
	svc := order.NewService(repo)

	o, err := svc.Fulfill(context.Background(), "o-1")
	if err != nil {
		fmt.Println("error:", err)
		return
	}
	fmt.Printf("order %s status: %s\n", o.ID, o.Status)
}
```

Run it:

```bash
go run ./cmd/demo
```

Expected output:

```
order o-1 status: fulfilled
```

### The tests

The tests are an external test package (`package order_test`) importing both the
`order` package and its `mocks` — that layering is what avoids an import cycle while
keeping gomock out of production. `gomock.NewController(t)` auto-registers `Finish`
via `t.Cleanup`, so there is no `defer ctrl.Finish()`; unmet expectations fail when
the test completes. `TestFulfillHappyPath` programs both `FindByID` and `Save` with
`.Times(1)` and lets auto-`Finish` prove both were called. `TestFulfillNotFound`
programs only `FindByID` to return `ErrNotFound` and deliberately sets *no* `Save`
expectation: because gomock is strict, if `Fulfill` erroneously called `Save` the
mock would fatal the test on the unexpected call. That is how you assert "Save was
not called" in gomock — by not expecting it.

Create `order/service_test.go`:

```go
package order_test

import (
	"context"
	"errors"
	"testing"

	"example.com/gomockorder/order"
	"example.com/gomockorder/order/mocks"
	"go.uber.org/mock/gomock"
)

func TestFulfillHappyPath(t *testing.T) {
	t.Parallel()
	ctrl := gomock.NewController(t)
	repo := mocks.NewMockOrderRepository(ctrl)

	ctx := context.Background()
	existing := &order.Order{ID: "o-1", Status: "pending"}

	repo.EXPECT().FindByID(ctx, "o-1").Return(existing, nil).Times(1)
	repo.EXPECT().Save(ctx, gomock.Any()).Return(nil).Times(1)

	svc := order.NewService(repo)
	got, err := svc.Fulfill(ctx, "o-1")
	if err != nil {
		t.Fatalf("Fulfill: %v", err)
	}
	if got.Status != "fulfilled" {
		t.Fatalf("status = %q, want fulfilled", got.Status)
	}
	// auto-Finish (t.Cleanup) verifies both expectations were met.
}

func TestFulfillNotFound(t *testing.T) {
	t.Parallel()
	ctrl := gomock.NewController(t)
	repo := mocks.NewMockOrderRepository(ctrl)

	ctx := context.Background()
	// Program only FindByID. No Save expectation: any Save call would fatal.
	repo.EXPECT().FindByID(ctx, "missing").Return(nil, order.ErrNotFound).Times(1)

	svc := order.NewService(repo)
	_, err := svc.Fulfill(ctx, "missing")
	if !errors.Is(err, order.ErrNotFound) {
		t.Fatalf("err = %v, want ErrNotFound", err)
	}
}

// Compile-time proof the generated mock satisfies the port.
var _ order.OrderRepository = (*mocks.MockOrderRepository)(nil)
```

## Review

The service is correct when `Fulfill` is "load, mutate to fulfilled, persist," and
propagates a load failure without saving. gomock's contribution is strictness: the
happy-path test does not manually assert the two calls happened — auto-`Finish`
does, because the `.Times(1)` expectations must be met. The not-found test asserts a
negative ("Save is never called") not with a call-count check but by *declining to
expect it*, so an accidental save fatals on the unexpected call. Two traps: do not
set expectations "just in case," because with `NewController(t)` an unmet one now
*fails* the test; and remember that the mock must live outside `package order` (its
own `mocks` package, imported by an external `order_test`) or you get an import
cycle and, worse, ship gomock in production. Run `go test -race`.
