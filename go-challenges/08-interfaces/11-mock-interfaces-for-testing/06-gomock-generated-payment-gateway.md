# Exercise 6: Generated Mocks With gomock — Expectations, Matchers, and Call Ordering

When an interface is wide or churns, hand-writing a spy for every method is
tedious. `go.uber.org/mock` generates a mock from the interface and gives you
expectation-based verification: you declare up front which calls to expect, with
which argument matchers, how many times, and in what order — and the controller
fails the test itself if the interaction does not match. This module tests a
`PaymentService` over a two-method `PaymentGateway` that way.

Fully self-contained: its own module, package, generated mock, demo, and test.
It fetches `go.uber.org/mock`, so gate it with `GOFLAGS=-mod=mod`.

## What you'll build

```text
gomockpayment/               independent module: example.com/payment
  go.mod                     go 1.26; requires go.uber.org/mock
  payment.go                 PaymentGateway; PaymentService.Charge; //go:generate
  mock_payment.go            MockPaymentGateway (generated by mockgen)
  cmd/
    demo/
      main.go                runnable demo with a real gateway impl
  payment_test.go            gomock: InOrder, Eq/Any, Times, DoAndReturn
```

- Files: `payment.go`, `mock_payment.go`, `cmd/demo/main.go`, `payment_test.go`.
- Implement: `PaymentService.Charge(ctx, amount)` that calls `Authorize` then `Capture`, stopping if `Authorize` fails.
- Test: a generated `MockPaymentGateway`; `gomock.InOrder` to pin Authorize-before-Capture, `gomock.Eq`/`gomock.Any` matchers, `Times`, `Return`, and `DoAndReturn` to inspect arguments; the controller auto-fails on missing/extra/out-of-order calls.
- Verify: `go test -count=1 -race ./...`

Set up the module. The mock file is normally produced by `mockgen`; it is included
here so the module is self-contained:

```bash

cd go-solutions/08-interfaces/11-mock-interfaces-for-testing/06-gomock-generated-payment-gateway
go get go.uber.org/mock/gomock@v0.6.0
# regenerate mock_payment.go from the interface:
go run go.uber.org/mock/mockgen@v0.6.0 -source=payment.go -destination=mock_payment.go -package=payment
```

### Expectation-based verification, and why it differs from a spy

The hand-rolled spy in Exercise 1 records calls and lets the test assert on the
record afterward — state-based verification. gomock inverts that: you program the
mock with *expectations* before running the SUT, and the mock verifies itself. A
`gomock.Controller` created with `NewController(t)` registers a cleanup that fails
the test if any expected call was not made, and each mock method fails immediately
if it is called in a way no expectation matches (wrong arguments, too many times,
out of order). You do not write "assert Authorize was called"; you *declare*
`EXPECT().Authorize(...)` and the controller enforces it.

The `//go:generate` directive at the top of `payment.go` records how the mock was
produced; `mockgen` reads the interface and emits `mock_payment.go`. In real code
that file is regenerated whenever the interface changes — the payoff over
hand-rolling grows with the interface's size and churn. Note `go.uber.org/mock` is
the maintained fork; the original `github.com/golang/mock` is archived and should
not be used in new code.

### Matchers, counts, and ordering are precision — and coupling

`EXPECT().Authorize(gomock.Any(), gomock.Eq(int64(500)))` says "expect Authorize
with any context and exactly the amount 500". `gomock.Any()` matches anything;
`gomock.Eq(x)` matches by deep equality. `.Times(1)` pins the count; `.Return(...)`
supplies the results. `gomock.InOrder(a, b)` declares that call `a` must happen
before call `b` — here Authorize strictly before Capture, which is a genuine
contract for a payment flow. `DoAndReturn` runs a function with the real arguments,
letting you inspect them and compute the return dynamically.

Every one of these is precision you are choosing to encode, and every constraint is
coupling. Pin the order when order is the contract (Authorize before Capture);
match the amount when the amount matters; but do not, for example, also assert a
specific context value that the contract does not require. The skill is asserting
the minimal true contract, not the maximal expressible one.

Create `payment.go`:

```go
package payment

import (
	"context"
	"fmt"
)

//go:generate go run go.uber.org/mock/mockgen@v0.6.0 -source=payment.go -destination=mock_payment.go -package=payment

// PaymentGateway is the outbound port to a payment provider.
type PaymentGateway interface {
	Authorize(ctx context.Context, amount int64) (string, error)
	Capture(ctx context.Context, authID string) error
}

// PaymentService charges a customer by authorizing then capturing.
type PaymentService struct {
	gw PaymentGateway
}

func NewPaymentService(gw PaymentGateway) *PaymentService {
	return &PaymentService{gw: gw}
}

// Charge authorizes the amount and, only on success, captures it.
func (s *PaymentService) Charge(ctx context.Context, amount int64) (string, error) {
	authID, err := s.gw.Authorize(ctx, amount)
	if err != nil {
		return "", fmt.Errorf("authorize: %w", err)
	}
	if err := s.gw.Capture(ctx, authID); err != nil {
		return "", fmt.Errorf("capture %s: %w", authID, err)
	}
	return authID, nil
}
```

The generated mock. Do not edit it by hand; `mockgen` produces exactly this from
the interface above.

Create `mock_payment.go`:

```go
// Code generated by MockGen. DO NOT EDIT.
// Source: payment.go
//
// Generated by this command:
//
//	mockgen -source=payment.go -destination=mock_payment.go -package=payment
//

// Package payment is a generated GoMock package.
package payment

import (
	context "context"
	reflect "reflect"

	gomock "go.uber.org/mock/gomock"
)

// MockPaymentGateway is a mock of PaymentGateway interface.
type MockPaymentGateway struct {
	ctrl     *gomock.Controller
	recorder *MockPaymentGatewayMockRecorder
	isgomock struct{}
}

// MockPaymentGatewayMockRecorder is the mock recorder for MockPaymentGateway.
type MockPaymentGatewayMockRecorder struct {
	mock *MockPaymentGateway
}

// NewMockPaymentGateway creates a new mock instance.
func NewMockPaymentGateway(ctrl *gomock.Controller) *MockPaymentGateway {
	mock := &MockPaymentGateway{ctrl: ctrl}
	mock.recorder = &MockPaymentGatewayMockRecorder{mock}
	return mock
}

// EXPECT returns an object that allows the caller to indicate expected use.
func (m *MockPaymentGateway) EXPECT() *MockPaymentGatewayMockRecorder {
	return m.recorder
}

// Authorize mocks base method.
func (m *MockPaymentGateway) Authorize(ctx context.Context, amount int64) (string, error) {
	m.ctrl.T.Helper()
	ret := m.ctrl.Call(m, "Authorize", ctx, amount)
	ret0, _ := ret[0].(string)
	ret1, _ := ret[1].(error)
	return ret0, ret1
}

// Authorize indicates an expected call of Authorize.
func (mr *MockPaymentGatewayMockRecorder) Authorize(ctx, amount any) *gomock.Call {
	mr.mock.ctrl.T.Helper()
	return mr.mock.ctrl.RecordCallWithMethodType(mr.mock, "Authorize", reflect.TypeOf((*MockPaymentGateway)(nil).Authorize), ctx, amount)
}

// Capture mocks base method.
func (m *MockPaymentGateway) Capture(ctx context.Context, authID string) error {
	m.ctrl.T.Helper()
	ret := m.ctrl.Call(m, "Capture", ctx, authID)
	ret0, _ := ret[0].(error)
	return ret0
}

// Capture indicates an expected call of Capture.
func (mr *MockPaymentGatewayMockRecorder) Capture(ctx, authID any) *gomock.Call {
	mr.mock.ctrl.T.Helper()
	return mr.mock.ctrl.RecordCallWithMethodType(mr.mock, "Capture", reflect.TypeOf((*MockPaymentGateway)(nil).Capture), ctx, authID)
}
```

### The runnable demo

The demo cannot use the mock — a mock needs a `*testing.T`-style reporter. It wires
a trivial real `PaymentGateway` so `go run ./cmd/demo` shows the service against a
working implementation.

Create `cmd/demo/main.go`:

```go
package main

import (
	"context"
	"fmt"

	"example.com/payment"
)

// realGateway is a trivial working PaymentGateway for the demo.
type realGateway struct{}

func (realGateway) Authorize(_ context.Context, amount int64) (string, error) {
	return fmt.Sprintf("auth_%d", amount), nil
}

func (realGateway) Capture(context.Context, string) error { return nil }

func main() {
	svc := payment.NewPaymentService(realGateway{})
	authID, err := svc.Charge(context.Background(), 500)
	if err != nil {
		fmt.Println("error:", err)
		return
	}
	fmt.Printf("charged, auth id: %s\n", authID)
}
```

Run it:

```bash
go run ./cmd/demo
```

Expected output:

```
charged, auth id: auth_500
```

### Tests

`TestChargeAuthorizesThenCaptures` uses `gomock.InOrder` with `Eq`/`Any` matchers
and `Times(1)` to pin the happy path; the controller's auto-cleanup fails the test
if either call is missing. `TestChargeStopsWhenAuthorizeFails` expects *only*
`Authorize` (returning an error) and sets no `Capture` expectation — so if the
service wrongly called `Capture`, the controller would flag an unexpected call.
`TestChargeInspectsAmountWithDoAndReturn` uses `DoAndReturn` to read the real
amount argument and derive the auth id.

Create `payment_test.go`:

```go
package payment

import (
	"context"
	"errors"
	"testing"

	"go.uber.org/mock/gomock"
)

func TestChargeAuthorizesThenCaptures(t *testing.T) {
	t.Parallel()

	ctrl := gomock.NewController(t) // auto-Finish via t.Cleanup
	gw := NewMockPaymentGateway(ctrl)

	gomock.InOrder(
		gw.EXPECT().Authorize(gomock.Any(), gomock.Eq(int64(500))).Return("auth_123", nil).Times(1),
		gw.EXPECT().Capture(gomock.Any(), gomock.Eq("auth_123")).Return(nil).Times(1),
	)

	svc := NewPaymentService(gw)
	got, err := svc.Charge(context.Background(), 500)
	if err != nil {
		t.Fatalf("Charge: %v", err)
	}
	if got != "auth_123" {
		t.Fatalf("auth id = %q, want auth_123", got)
	}
}

func TestChargeStopsWhenAuthorizeFails(t *testing.T) {
	t.Parallel()

	ctrl := gomock.NewController(t)
	gw := NewMockPaymentGateway(ctrl)

	declined := errors.New("card declined")
	// Only Authorize is expected. No Capture expectation: if Charge calls
	// Capture, the controller fails the test with an unexpected call.
	gw.EXPECT().Authorize(gomock.Any(), gomock.Any()).Return("", declined).Times(1)

	svc := NewPaymentService(gw)
	_, err := svc.Charge(context.Background(), 500)
	if !errors.Is(err, declined) {
		t.Fatalf("err = %v, want card declined", err)
	}
}

func TestChargeInspectsAmountWithDoAndReturn(t *testing.T) {
	t.Parallel()

	ctrl := gomock.NewController(t)
	gw := NewMockPaymentGateway(ctrl)

	gomock.InOrder(
		gw.EXPECT().Authorize(gomock.Any(), gomock.Any()).DoAndReturn(
			func(_ context.Context, amount int64) (string, error) {
				if amount <= 0 {
					return "", errors.New("non-positive amount")
				}
				return "auth_ok", nil
			}),
		gw.EXPECT().Capture(gomock.Any(), gomock.Eq("auth_ok")).Return(nil),
	)

	svc := NewPaymentService(gw)
	got, err := svc.Charge(context.Background(), 1200)
	if err != nil {
		t.Fatalf("Charge: %v", err)
	}
	if got != "auth_ok" {
		t.Fatalf("auth id = %q, want auth_ok", got)
	}
}
```

## Review

gomock trades the hand-rolled spy's readable, low-magic code for a codegen step and
a declarative expectation model. The win is precision with little boilerplate:
`InOrder` pins Authorize-before-Capture, `Times` pins counts, matchers pin
arguments, and the controller — auto-finished by `NewController(t)` — fails the test
on any missing, extra, or out-of-order call without a single explicit assertion.
The `TestChargeStopsWhenAuthorizeFails` case is the clearest expression of this:
the *absence* of a `Capture` expectation is what proves the service short-circuits.

The cost is coupling and reflection. Each matcher and ordering constraint ties the
test to the implementation, so encode only the contract that matters. And remember
gomock, like any mock, encodes your assumptions about the gateway, not the
provider's real behavior — keep at least one integration test against a sandbox
gateway so a renamed field or changed error semantics cannot pass unnoticed behind
a wall of green expectations.
